End-to-End System Design

Multi-Operator Bus
Booking Platform

A complete blueprint to build a production-grade bus ticketing system from scratch — covering every layer from data models and microservices to payment flows, real-time seat maps, and operator dashboards.

Node.js / TypeScript React + Next.js PostgreSQL + Redis Kafka / RabbitMQ Docker + Kubernetes Stripe / Razorpay
01

System Overview — What we're building

The platform connects multiple bus operators with passengers. Each operator can have different bus fleets, seat configurations, amenities, pricing rules, and cancellation policies. Passengers can search routes, compare buses, book seats, make payments, and get real-time updates — all in one app.

Core Stakeholders

👤

Passenger

Search routes, compare operators, book & cancel tickets, track bus location, view history, receive notifications.

🚌

Bus Operator

Register buses, define routes & schedules, manage seat inventory, view bookings, handle refunds, view revenue reports.

🛡

Platform Admin

Approve operators, manage commissions, monitor platform health, handle disputes, configure platform-wide policies.

Key Differentiators

🪑

Dynamic Seat Layouts

Each bus has a custom seat map — sleepers, semi-sleepers, seater, AC/Non-AC, upper/lower berths. Stored as JSON config per bus type.

💰

Multi-Operator Pricing

Operators set base prices; platform applies dynamic pricing algorithms based on demand, time-to-departure, and remaining inventory.

🔒

Atomic Seat Locking

Redis-based distributed locks hold seats for 10 minutes during payment. Prevents double-booking across concurrent requests.

📡

Real-Time Updates

WebSocket connections push live seat availability, bus tracking, and boarding notifications to all connected clients.

Requirements — Functional & Non-Functional

Functional Requirements

  • F1

    Operator Onboarding & Management

    Operators can register, upload documents for KYC, define their bus fleet (make, model, amenities, seat layout), and create routes with multiple stops and schedules.

  • F2

    Route & Schedule Search

    Passengers search by origin, destination, and date. Results show all operators, departure times, durations, amenities, price, and available seat count. Filters by price, rating, amenity, bus type.

  • F3

    Interactive Seat Selection

    Passengers view a real-time seat map. Booked seats are marked. Selected seats are held with a distributed lock for 10 minutes while the passenger completes payment.

  • F4

    Payment Processing

    Support UPI, cards, wallets, net banking. Integrate Razorpay (India) or Stripe (global). Handle payment success, failure, and partial refunds for cancellations.

  • F5

    Booking Management

    Passengers can view booking history, download e-tickets as PDF, cancel bookings (within operator's cancellation policy), and request refunds.

  • F6

    Real-Time Bus Tracking

    Bus driver app sends GPS coordinates. Passengers can see live bus location on map. Estimated arrival at each stop is computed dynamically.

  • F7

    Notifications

    Booking confirmations, departure reminders, delay alerts, and refund status sent via email, SMS, and push notifications.

  • F8

    Operator Revenue & Analytics

    Operators see daily/monthly revenue, occupancy rates per trip, cancellation rates, ratings per route, and payout history.

Non-Functional Requirements

⚡

Performance

Search API < 500ms P99. Seat lock under 100ms. 10,000 concurrent users at launch.

🔄

Availability

99.9% uptime (8.7 hrs downtime/year). Zero downtime deployments. Circuit breakers on all service-to-service calls.

📦

Scalability

Horizontal scaling of stateless services. Read replicas for DB. Sharded Redis for seat locks.

🔐

Security

JWT + refresh tokens. PCI-DSS compliance for payments. HTTPS everywhere. Rate limiting per IP & user.

Design Decision: Shared DB vs Per-Service DB

Each microservice owns its own database schema/tables within PostgreSQL (not separate DB servers initially). As the platform grows, high-traffic services (Booking, Search) get dedicated PostgreSQL instances. This balances operational simplicity at launch with future isolation.

04

Microservices — Responsibilities & Contracts

👤

User Service

Port 3001 · Database: users schema · Cache: Redis session store
  • Register, login, OTP-based phone auth, Google OAuth
  • Issue JWT access tokens (15 min TTL) + refresh tokens (30 days)
  • Store user profile, preferences, saved passenger details
  • Role management: passenger, operator_admin, platform_admin
  • Emit user.registered, user.verified events to Kafka
🔍

Search Service

Port 3002 · Elasticsearch + Redis cache · Stateless, horizontally scalable
  • Index all active trips with their metadata into Elasticsearch on every schedule change
  • Handle queries: origin → destination → date with fuzzy matching on city names
  • Apply filters: price range, departure time, operator, amenities, bus type
  • Cache popular route searches in Redis (5 min TTL) for repeated queries
  • Return aggregated results: trip details + operator info + real-time available seat count from Booking Service
🚌

Operator Service

Port 3003 · Database: operators schema · Object store for documents
  • Operator registration, KYC document upload (Aadhar, PAN, RC, permits)
  • Bus fleet management: register buses with make, model, year, capacity
  • Seat layout definition: JSON schema describing each seat (id, type, row, column, deck, price_modifier)
  • Route management: define origin–destination–stops with distances and durations
  • Schedule management: recurring trips with departure times, valid date ranges, seasonal overrides
  • Pricing rules: base fare + dynamic multipliers by seat type, advance booking, load factor
🪑

Booking Service CORE

Port 3004 · Database: bookings schema · Redis distributed locks · WebSocket server
  • Maintain trip_seats table: one row per seat per trip instance (date-specific)
  • Seat lock: SET seat:{tripId}:{seatId} {userId} EX 600 NX in Redis — atomic, expires in 10 min
  • On payment success: mark seats as BOOKED, create booking record, generate PNR
  • On lock expiry: release seats, publish seat.released event to WebSocket clients
  • Cancellation engine: apply operator's cancellation slab policy, calculate refund amount
  • Generate e-ticket PDF with QR code (unique per booking, verified at boarding)
  • Push real-time seat map updates to all connected clients via WebSocket rooms (one room per trip)
💳

Payment Service

Port 3005 · Database: payments schema · Idempotency keys · Webhook receiver
  • Create payment intent / order with Razorpay or Stripe
  • Store idempotency key per booking to prevent duplicate charges
  • Receive payment webhooks; verify signature; update payment status
  • Trigger payment.succeeded or payment.failed events to Kafka
  • Handle refunds: partial (seat cancellation) and full (trip cancelled by operator)
  • Commission split: hold platform commission, transfer remainder to operator payout account
  • Payout scheduler: batch operator payouts via NEFT/IMPS on D+2 settlement cycle
📡

Tracking Service

Port 3006 · Redis time-series + PostGIS · WebSocket broadcast
  • Driver app sends GPS coordinates every 10 seconds via WebSocket
  • Store last known position in Redis with GEO commands; write to PostGIS for history
  • Compute distance to each upcoming stop; publish ETA updates to trip WebSocket room
  • Detect geofence arrival at stops: trigger bus.arrived_at_stop event
  • Passengers subscribed to that trip receive live ETA on their booking detail screen

⚠️

Seat Inventory Concurrency

Never do a naive SELECT + UPDATE pattern for seats — it causes race conditions under concurrency. The correct pattern: Redis atomic SET NX EX for temporary lock during checkout, then PostgreSQL UPDATE trip_seats SET status='BOOKED' WHERE seat_id=? AND status='AVAILABLE' with a row-level check. Use PostgreSQL FOR UPDATE SKIP LOCKED for batch operations.

JWT Token Structure

// Access Token Payload (expires in 15 minutes) { "sub": "user-uuid", "role": "passenger", // passenger | operator_admin | platform_admin "operator_id": "operator-uuid", // present if role = operator_admin "iat": 1712345678, "exp": 1712346578 // 15 minutes from iat } // Signed with RS256 (private key on Auth Service, public keys exposed at /auth/.well-known/jwks) // API Gateway validates using public key — no auth DB hit per request

Role-Based Access Control (RBAC)

👤

Passenger

Search, view, book, cancel own bookings, view own profile. Cannot access any operator or admin routes.

🚌

Operator Admin

Manage own buses, routes, schedules. View own bookings and revenue. Scoped to their operator_id — cannot access other operators' data.

🛡

Platform Admin

Approve/reject operators. Access all data. Manage commissions. Trigger manual refunds. Configure platform settings.

Security Checklist

  • 🔒

    HTTPS Everywhere + HSTS

    All traffic over TLS 1.2+. HSTS preload enforced. API Gateway terminates TLS.

  • 🚦

    Rate Limiting

    Per-IP: 100 req/min on public endpoints. Per-user: 30 req/min on booking endpoints. OTP: 3 per phone per hour. Implemented at API Gateway level with Redis counters.

  • 🛡

    Payment Security (PCI DSS)

    Card data never touches our servers — all card collection via Razorpay/Stripe hosted fields (iFrame). We only store gateway order IDs and payment intent IDs, never raw card numbers.

  • 🔑

    Secrets Management

    All API keys (payment gateways, SMS provider, Google Maps) stored in HashiCorp Vault. Services fetch secrets at startup. Never in environment variables or source code.

  • 🧹

    Input Validation

    All API inputs validated with Zod/Joi schemas before processing. SQL injection prevented by parameterized queries via Prisma/TypeORM. XSS prevention via output encoding.

Payment Flow — Booking & Refund Lifecycle

Complete Booking + Payment Sequence

  • 1

    Passenger selects seats

    Frontend sends POST /bookings/hold-seats. Booking Service acquires Redis locks on each seat: SET seat:{tripId}:{seatId} {userId} NX EX 600. If any lock fails (seat already locked), returns 409 Conflict.

  • 2

    Payment order created

    POST /bookings/initiate-payment calls Razorpay API to create an order for the total amount. Stores order_id in DB with status PENDING. Returns order_id to frontend.

  • 3

    Passenger completes payment

    Frontend opens Razorpay checkout iframe. Passenger enters payment details. Razorpay processes payment client-side. On success, frontend receives payment_id + signature.

  • 4

    Frontend calls confirm endpoint

    POST /bookings/confirm sends { holdId, paymentId, signature }. Booking Service verifies Razorpay signature (HMAC-SHA256), updates payment as PAID, marks seats as BOOKED, generates PNR.

  • 5

    Async confirmation flow

    Booking Service publishes booking.confirmed to Kafka. Notification Service consumes it → sends confirmation SMS + email. Analytics Service updates occupancy stats. E-ticket PDF generated async.

  • 6

    Webhook as backup confirmation

    Razorpay also sends payment.captured webhook to our endpoint. Payment Service verifies and processes idempotently (if booking not yet confirmed, triggers confirm flow). Handles cases where frontend crashes after payment.

Cancellation & Refund Logic

// Cancellation policy slabs (configured per operator per route) const cancelPolicy = [ { hoursBeforeDeparture: 48, refundPercent: 100 }, { hoursBeforeDeparture: 24, refundPercent: 75 }, { hoursBeforeDeparture: 12, refundPercent: 50 }, { hoursBeforeDeparture: 6, refundPercent: 25 }, { hoursBeforeDeparture: 0, refundPercent: 0 }, ]; // Refund amount computation function computeRefund(booking, policy) { const hoursLeft = diffInHours(booking.departureTime, new Date()); const slab = policy.findLast(s => hoursLeft >= s.hoursBeforeDeparture); return (booking.totalAmount * slab.refundPercent) / 100; }
✅

Idempotency is Critical in Payments

Every payment operation (create order, confirm booking, initiate refund) must use an idempotency key. Store it in DB before calling the payment gateway. On retry, return the same stored result rather than re-calling the gateway. This prevents double-charges when clients retry on network timeouts.

User Flows — Passenger Journey

End-to-End Booking Flow

  • 1

    Search

    User enters origin city (autocomplete from Elasticsearch), destination, travel date, and passenger count. Hits GET /trips/search. Results load in <500ms from Elasticsearch + Redis cache.

  • 2

    Compare & Select Bus

    User sees sorted list of buses: price, departure time, duration, amenity icons, rating, available seats. Can filter and sort. Clicking a bus shows operator profile + cancellation policy before proceeding.

  • 3

    Seat Selection

    Real-time seat map loads. WebSocket connection established to trip room. User clicks available seats (turns them to "selecting" state). For multi-passenger, fills passenger details (name, age, gender) per seat.

  • 4

    Boarding & Dropping Points

    User selects from the operator's defined stops along the route. Price adjusts based on boarding–dropping pair distance.

  • 5

    Review & Pay

    Summary screen: seats, passengers, boarding point, fare breakdown (base fare + taxes + convenience fee). User selects payment method → Razorpay checkout. On success, booking confirmed screen with PNR.

  • 6

    Post-Booking

    Confirmation SMS/email within 30s. E-ticket downloadable. On day of travel: push notification 2hrs before departure. Live tracking available after departure. QR code scanned at boarding gate.

Scaling Strategy — Bottlenecks & Solutions

🔍

Search Bottleneck

Elasticsearch cluster with 3 shards + 1 replica each. Separate read nodes for search queries. Trip index is small (few million documents) — fast by default. Cache top-100 route searches in Redis.

🪑

Seat Lock Bottleneck

Redis cluster sharded by tripId. Seat locks are short-lived (10 min). At 10k concurrent users, ~1000 active lock operations/sec — well within Redis's 100k ops/sec capacity.

🗄

Database Bottleneck

Read replicas for all read-heavy queries (search results, booking history). Booking confirmation writes go to primary. Connection pooling with PgBouncer (max 100 connections per service).

📡

WebSocket Bottleneck

Socket.io with Redis adapter so WebSocket messages can be broadcast across multiple server instances. Each trip room distributes updates to all subscribers regardless of which server they connected to.

Notification Service Design

The Notification Service is a pure consumer — it listens to Kafka events and dispatches messages via the appropriate channel. It is completely decoupled from all other services. A Notification Template Engine stores templates per event type with variable substitution.

📋

Notification Deduplication

Kafka messages can be delivered more than once (at-least-once semantics). Store a notification_log table with eventId + channel as a unique key. Before sending, check if this eventId was already dispatched on that channel. This prevents duplicate SMS/emails on Kafka consumer restarts.

Data Flow Diagrams — How data moves through the system

Understanding exactly how data moves through the platform is critical before writing a single line of code. These flows cover the three most complex scenarios: booking, cancellation, and operator onboarding.

Flow 1 — Complete Booking Data Journey

STEP-BY-STEP DATA MOVEMENT: PASSENGER → CONFIRMED BOOKING
1
Browser → API Gateway → Search Service
GET /trips/search?from=BLR&to=MYS&date=2025-08-15 — Gateway validates JWT, forwards to Search. Search queries Elasticsearch index, enriches with live available_seats count from Redis, returns sorted list.
2
Browser → Booking Service + Redis
GET /trips/:id/seat-map — Booking Service reads trip_seats from PostgreSQL (status per seat). Merges with Redis keys seat:{tripId}:* to show which seats are currently locked by other users. Returns complete map to browser. WebSocket connection opened.
3
Browser → Booking Service → Redis SETNX
POST /bookings/hold-seats — For each selected seat, Booking Service executes SET seat:{tripId}:{seatId} {userId} NX EX 600. If all succeed → return holdId. If any fail (race condition) → release all acquired locks, return 409. Broadcast seat:locked event to WebSocket room.
4
Booking Service → Payment Service → Razorpay API
POST /bookings/initiate-payment — Booking Service calls Payment Service via gRPC. Payment Service creates a Razorpay order, stores { orderId, holdId, amount, idempotencyKey } in DB with status PENDING. Returns Razorpay order_id to browser.
5
Razorpay Webhook → Payment Service → Kafka
Razorpay sends payment.captured webhook. Payment Service verifies HMAC signature, marks payment as PAID, publishes payment.succeeded to Kafka topic.
6
Kafka consumer → Booking Service → PostgreSQL
Booking Service consumes payment.succeeded. In a single DB transaction: UPDATE trip_seats set status=BOOKED, INSERT into bookings with PNR, DELETE Redis seat lock keys. Publishes booking.confirmed to Kafka.
7
booking.confirmed → Notify: SMS + Email + E-ticket PDF + Analytics update
Multiple Kafka consumers act in parallel: Notification Service sends confirmation SMS + email with ticket. A PDF worker generates e-ticket and uploads to S3. Analytics Service updates occupancy counters in ClickHouse. Search Service decrements available_seats in Elasticsearch index.

Flow 2 — Cancellation & Refund Journey

  • C1

    Passenger requests cancellation

    POST /bookings/:id/cancel with optional seatIds[] (partial cancel). Booking Service verifies the booking belongs to this user. Checks if cancellation is allowed (trip not departed, not already cancelled).

  • C2

    Refund amount computed

    Booking Service fetches operator's cancellation policy slabs. Computes hours remaining until departure. Applies the matching slab percentage to the seat price. If partial cancel, computes per-seat refund only.

  • C3

    DB updated atomically

    In a single PostgreSQL transaction: update trip_seats status → AVAILABLE, update bookings.status → CANCELLED (or PARTIALLY_CANCELLED), insert into refund_requests table with computed amount.

  • C4

    Refund initiated asynchronously

    Booking Service publishes booking.cancelled to Kafka. Payment Service consumes it, calls Razorpay Refund API with idempotency key = refund_{bookingId}_{timestamp}. Updates refund status in DB as Razorpay responds.

  • C5

    Downstream updates fan out

    Notification Service: sends cancellation confirmation + refund ETA SMS. Search Service: increments available seat count in Elasticsearch. Analytics: records cancellation event. WebSocket: broadcasts seat released to anyone viewing that trip's seat map.

Flow 3 — Operator Onboarding Journey

  • 2

  • 3

Caching Strategy — What, Where & How Long

Poor caching kills performance. Over-aggressive caching causes stale data bugs. This section defines exactly what gets cached, at which layer, with what TTL, and when to invalidate.

Cache Layers

🌐

CDN Cache (CloudFront)

Static assets: JS bundles, CSS, images — 1 year TTL with content-hash in filename. Operator logos and bus photos — 7 days. E-ticket PDFs — no cache (private, auth required).

⚡

Redis App Cache

Search results per query hash — 5 min TTL. Trip seat map — no TTL (event-driven invalidation). City list — 24 hrs. Operator profile — 10 min.

🖥

Browser Cache

React Query caches API responses in memory. Search results — 2 min stale-while-revalidate. Booking history — invalidated on cancel/new booking. Seat map — never cached, always fresh from WebSocket.

Redis Key Design

// ─── SEAT LOCKS (Booking Service) ─────────────────────────────────── seat:{tripId}:{seatId} → userId TTL: 600s (10 min) seat_hold:{holdId} → JSON{tripId,seats,userId,expiresAt} TTL: 600s // ─── SEARCH CACHE (Search Service) ────────────────────────────────── search:{hash(from+to+date)} → JSON[] TTL: 300s (5 min) city:autocomplete:{prefix} → JSON[] TTL: 86400s (24 hr) // ─── TRIP AVAILABILITY (Booking Service) ───────────────────────────── trip:avail:{tripId} → INT (seat count) TTL: none (event-invalidated) // ─── AUTH (User Service) ───────────────────────────────────────────── refresh:{userId}:{tokenId} → "1" TTL: 30 days blacklist:token:{jti} → "1" TTL: until access token expires otp:{phone} → hashedOtp TTL: 300s // ─── BUS TRACKING (Tracking Service) ───────────────────────────────── bus:loc:{busId} → JSON{lat,lng,ts} TTL: 60s bus:eta:{tripId}:{stopId} → INT (minutes) TTL: 120s // ─── RATE LIMITING (API Gateway) ───────────────────────────────────── ratelimit:ip:{ip}:{window} → INT count TTL: window duration ratelimit:user:{userId}:{window}→ INT count TTL: window duration

Cache Invalidation Rules

Cache KeyInvalidated WhenMethod
search:{hash}Operator changes price, adds/removes trip, trip cancelledDelete key on Kafka event trip.updated
trip:avail:{tripId}Seat booked, seat lock acquired/released, trip cancelledDECR/INCR atomically or DEL + recompute
seat:{tripId}:{seatId}Payment confirmed (seat booked), hold timer expiresDEL on confirm; Redis TTL auto-expires lock

⚠️

Cache Stampede Prevention

When a popular search cache key expires, hundreds of concurrent requests hit Elasticsearch simultaneously — a "thundering herd." Prevent this with probabilistic early expiration: recompute cache when TTL drops below a random threshold, not when it hits zero. Alternatively, use a Redis lock on the key so only one request recomputes while others wait for the result.

Error Handling — Resilience & Graceful Degradation

Production systems fail. The goal isn't to prevent all failures — it's to fail predictably, recover automatically, and give users the best possible experience even when things go wrong.

Standard API Error Response Format

// Every error response across ALL services follows this envelope { "success": false, "error": { "code": "SEAT_ALREADY_LOCKED", // machine-readable, stable across versions "message": "Seat A3 is currently held by another user", "details": { "seatId": "A3", "retryAfter": 45 }, // optional contextual data "traceId": "req_3kj9fxm2p" // correlate with logs in Grafana } } // HTTP status codes used consistently: // 400 Bad Request — validation errors, malformed input // 401 Unauthorized — missing or invalid JWT // 403 Forbidden — valid JWT but insufficient role/scope // 404 Not Found — resource doesn't exist // 409 Conflict — seat already locked, duplicate booking // 422 Unprocessable — valid input but business rule violation // 429 Too Many Requests— rate limit exceeded // 503 Service Unavailable — downstream dependency down

Circuit Breaker Pattern

Every service-to-service call is wrapped in a circuit breaker (using opossum library in Node.js). This prevents cascade failures when a downstream service slows down or crashes.

import CircuitBreaker from 'opossum'; const paymentServiceBreaker = new CircuitBreaker(callPaymentService, { timeout: 3000, // if function takes > 3s, trip the breaker errorThresholdPercentage: 50, // trip when 50% of last 10 calls fail resetTimeout: 30000, // try again after 30s in OPEN state volumeThreshold: 10, // need at least 10 calls before tripping }); paymentServiceBreaker.fallback(() => ({ error: 'PAYMENT_SERVICE_UNAVAILABLE', message: 'Payment service is temporarily unavailable. Please retry.' })); // States: CLOSED (healthy) → OPEN (tripped) → HALF_OPEN (testing recovery)

Dead Letter Queue (DLQ) for Kafka

// Every Kafka consumer has a DLQ for failed messages async function consumeWithDLQ(message) { for (let attempt = 1; attempt <= 3; attempt++) { try { await processMessage(message); return; // success } catch (err) { if (attempt === 3) { // All retries exhausted → send to Dead Letter Queue await kafka.publish(`dlq.${originalTopic}`, { originalMessage: message, error: err.message, failedAt: new Date().toISOString(), attempts: 3 }); // Alert on-call engineer via PagerDuty if DLQ depth > 10 } await sleep(1000 * attempt); // exponential backoff } } } // DLQ messages are replayed manually after investigation

Payment Failure Handling

❌

Payment Declined

Seat locks remain active for the full 10-min window. User can retry with a different payment method. If they abandon, locks expire automatically via Redis TTL. No DB booking record created.

🔌

Network Timeout During Payment

User may not know if payment went through. We poll Razorpay order status every 2s for 60s. If captured but our webhook missed it, we confirm the booking. The Razorpay webhook is the final backup for all edge cases.

Multi-Tenancy Design — Isolating Operator Data

This platform is a shared infrastructure, logically isolated multi-tenant system. All operators run on the same database, same services, and same Kubernetes cluster — but their data is strictly isolated by design.

Tenant Isolation Strategy

🗄

Row-Level Isolation (Database)

Every table that belongs to an operator has an operator_id column. All queries from the Operator Service automatically filter by the operator_id from the JWT token. A middleware layer injects this filter — operators literally cannot query data belonging to other operators even if they craft a custom API request.

🔑

Token-Scoped Access

Operator admin JWTs contain operator_id claim. Every protected endpoint checks that the resource being accessed belongs to the operator_id in the token. This check happens in the service layer, not just the gateway.

📦

S3 Prefix Isolation

All operator files (KYC docs, bus images) are stored under s3://bucket/operators/{operatorId}/. Pre-signed URLs are generated per-operator and expire in 15 minutes. Operators cannot access files from other operator prefixes.

💰

Financial Isolation

Each operator has a separate virtual ledger account in the payments schema. Commission is deducted at booking time. Payouts only go to that operator's verified bank account via Razorpay Route — never cross-contaminated.

Operator Data Isolation in Code

Redis Sliding Window Algorithm

// Sliding window rate limiter using Redis sorted sets async function isRateLimited(key, limit, windowMs) { const now = Date.now(); const windowStart = now - windowMs; const pipeline = redis.pipeline(); // Remove entries older than the window pipeline.zremrangebyscore(key, '-inf', windowStart); // Count entries in current window pipeline.zcard(key); // Add current request timestamp pipeline.zadd(key, now, `${now}-${Math.random()}`); // Set key expiry to window duration pipeline.expire(key, Math.ceil(windowMs / 1000)); const results = await pipeline.exec(); const currentCount = results[1][1]; return { limited: currentCount >= limit, remaining: Math.max(0, limit - currentCount - 1), resetAt: now + windowMs }; }
💡

Rate Limit Response Headers

Always return X-RateLimit-Limit, X-RateLimit-Remaining, and X-RateLimit-Reset headers on every response, not just when throttled. This lets frontend apps show real-time quota feedback and implement smart retry logic with the exact reset time.

Testing Strategy — Unit, Integration, Load

Testing for a booking platform requires special attention to concurrency (seat locking races), idempotency (payment retries), and money (refund calculations). Tests are the only way to confidently deploy 10 times a day.

Test Pyramid

🧱

Unit Tests (70%)

Pure business logic: refund amount calculation, cancellation policy slabs, seat availability count, JWT validation, price computation with dynamic modifiers. Use Jest. Run in <30s. No DB, no Redis.

🔗

Integration Tests (20%)

API endpoints with real PostgreSQL (Docker) and Redis (Docker). Test the full booking flow end-to-end through HTTP. Test seat lock contention with concurrent requests. Use Supertest + testcontainers.

🎭

E2E Tests (10%)

Playwright tests against staging environment. Cover: search → select → book → download ticket. Run on every PR merge to main. Also test payment failure and cancellation flows.

Critical Test Cases

  • T1

    Concurrent Seat Lock — Race Condition Test

    Send 50 simultaneous POST /bookings/hold-seats requests for the same seat from 50 different users. Assert: exactly 1 succeeds (200), 49 return 409. Assert: Redis has exactly 1 lock key for that seat. This is the most important test in the entire codebase.

  • T2

    Payment Idempotency Test

    Call POST /bookings/confirm with the same paymentId twice (simulating client retry). Assert: only 1 booking record created, second call returns the existing booking without creating a duplicate.

  • T3

    Seat Lock Expiry Test

    Set Redis TTL to 2s for testing, hold seats, wait 3s without completing payment. Assert: Redis keys deleted, trip:avail:{tripId} counter incremented, WebSocket broadcasts seat release to connected clients.

  • T4

    Refund Calculation Accuracy

    Parameterized tests: booking with departure 50 hrs away → expect 100% refund. 20 hrs → 75%. 8 hrs → 50%. 4 hrs → 25%. 1 hr → 0%. Test each cancellation slab boundary exactly. Test partial cancellation (cancel 1 of 3 seats).

  • T5

    Operator Data Isolation Test

    Create two operators, create buses for each. Log in as Operator A, attempt to access Operator B's bus via GET /operator/buses/:busId. Assert: 403 Forbidden. Attempt to book a trip from Operator B's bus via Operator A's token. Assert: correct behavior (passengers can book any operator's trips, but operator APIs are scoped).
