Data Layer

Protocol

Communication Layer

Dual protocol design — HTTP/2 for REST APIs, WebSocket (WS) for real-time messaging. XMPP considered but WebSocket preferred for web clients.

  • REST — auth, profile, history
  • WebSocket — live messages, typing, presence
  • HTTP/2 Push — notifications fallback
  • QUIC/HTTP3 — mobile networks
Delivery

Message Guarantees

At-least-once delivery with client-side deduplication using message IDs. Exactly-once semantics enforced via idempotency keys.

  • Client generates UUID per message
  • Server deduplicates on UUID
  • ACK-based receipt system
  • Retry with exponential backoff
Scale Numbers

Target Scale

Designed for WhatsApp-scale — 2B users, 100B messages/day, 65B messages active daily.

  • 100M concurrent connections
  • ~1M messages/second peak
  • 500ms p99 delivery latency
  • 99.99% availability SLA

Real-time bi-directional communication at scale

Connection Strategy

Each WebSocket server manages ~50K concurrent connections. Users connect to the nearest WS server (via geo-DNS). Redis Pub/Sub broadcasts events across WS servers. Sticky sessions via consistent hashing ensure the same user reconnects to the same server.

Client sends message:send via WebSocket with media_id. Recipients receive the signed CDN URL for delivery.

03d
Push Notifications
FCM / APNs for offline users
Flow

Notification Pipeline

When a recipient is offline, the message is queued in Kafka. A Notification Worker consumes the queue and sends via FCM (Android) or APNs (iOS). Web uses Web Push (VAPID).

  • Message → Kafka topic: notifications
  • Worker checks: user online?
  • If offline → push via FCM/APNs
  • Notification includes: sender, preview
Privacy

E2E Encrypted Notifications

Push payload is minimal for privacy — only shows "You have a new message". Actual content is fetched from server only when app is opened. This respects E2E encryption.

  • No message content in push payload
  • Badge count only
  • Silent push for background sync
  • Notification preferences: mute/custom

05c
Security
Defense-in-depth across all layers
🔑
Authentication — JWT + Refresh Tokens
Short-lived JWT (15 min) + long-lived refresh token (30 days) in httpOnly cookie. Refresh tokens stored hashed in Redis. Device fingerprinting to detect token theft. Revoke all sessions on password change.
🔐
End-to-End Encryption (Signal Protocol)
X3DH key exchange for initial handshake. Double Ratchet algorithm for forward secrecy. Each message encrypted with a unique symmetric key. Server never has access to plaintext. Public keys stored on server; private keys stay on device.
🛡️
Transport Security
TLS 1.3 enforced for all connections. Certificate pinning in mobile apps. HSTS with preloading. WSS (WebSocket Secure) only. Perfect Forward Secrecy via ECDHE cipher suites.
🚫
Media Security
ClamAV virus scanning on all uploaded files. Time-limited signed S3 URLs (1 hour). Content-type validation. File size limits enforced server-side. EXIF data stripped from images. No public media URLs — all require auth.
🌐
Network Security
AWS WAF rules block known attack patterns. DDoS protection via CloudFront Shield. API rate limiting per user and IP. CORS configured per service. VPC isolation — databases not internet-facing. Secrets in AWS Secrets Manager.
📋
Compliance & Privacy
GDPR-compliant data deletion. Data minimization — only collect what's needed. Audit logs for admin actions. PBKDF2-hashed phone numbers in analytics. Right to export data. Regional data residency options.

06
Message Flow (1:1 Chat)
Complete lifecycle — send to read receipt
1

User A types & sends

Client encrypts message with recipient's public key. Generates client_msg_id (UUID). Shows message as "pending" (single grey tick). Sends via WebSocket: message:send event.

2

WS Server receives → Chat Service

WS server validates JWT, calls Chat Service via gRPC. Chat Service deduplicates on client_msg_id, writes to Cassandra, publishes message:new event to Kafka topic chat.{chat_id}.

3

Server ACKs to Sender

Chat Service responds with server_msg_id + timestamp. WS server sends message:ack to User A's socket. Client updates local message ID and shows "sent" (double grey tick).

4

Fan-out to Recipient

Kafka consumer checks if User B is online via Redis online:{userId}. If online → push via WS. If offline → enqueue in notification queue → send FCM/APNs push.

5

Recipient Receives Message

User B's client receives message:new via WebSocket. Client decrypts with private key, displays message, sends message:delivered event back to server. Server updates status in Cassandra.

6

Delivery Receipt to Sender

Server relays delivered status to User A's WS connection. User A's client updates message to "delivered" (double grey tick). Redis updates msg_status.

7

Read Receipt

When User B opens the chat, client sends message:read with message ID. Server updates status. Relays to User A → client shows "read" (double blue tick). Updates unread:{userId}:{chatId} counter in Redis to 0.

06b
Group Chat Architecture
Fan-out strategies for small and large groups

The Core Challenge: Fan-out at Scale

A group with 500 members means 1 message → 499 deliveries. At 1M messages/second, naive fan-out breaks. WhatsApp uses client-side encryption (each recipient gets separately encrypted copy) + server-side fan-out optimization.

Small Groups (< 100)

Direct WS Fan-out

For small groups, the WS server fetches the member list from Redis cache and pushes the message directly to each member's WebSocket connection. Simple, low latency.

  • Cache group members in Redis
  • Fan-out in parallel goroutines
  • Single Kafka event consumed by WS cluster
  • Each member gets individual delivery
Large Groups (100–1024)

Kafka Fan-out Pipeline

Message published to Kafka. Fan-out Service consumes event, fetches member list from DB, creates individual delivery tasks in Kafka partitioned by userId — one consumer per WS server cluster.

  • Message → Kafka: chat.events
  • Fan-out Service → Kafka: delivery.{userId}
  • WS Consumers deliver per-user
  • Async — eventually consistent

06c
End-to-End Encryption
Signal Protocol — X3DH + Double Ratchet
1

Key Generation (on registration)

Device generates: Identity Key (IK), Signed PreKey (SPK), and 100 One-Time PreKeys (OPK). Public keys uploaded to server. Private keys never leave device.

2

Session Initiation (X3DH)

Sender fetches recipient's public key bundle. Runs X3DH (Extended Triple Diffie-Hellman) to derive shared secret. Establishes Double Ratchet session. First message includes sender's ephemeral key.

3

Ongoing Encryption (Double Ratchet)

Each message uses a new message key derived from ratchet state. Forward secrecy: past sessions can't be decrypted even if current key is compromised. Break-in recovery built in.

4

Group E2EE (Sender Keys)

Each member generates a Sender Key. On joining, sender key is distributed to all group members encrypted individually. Messages encrypted once with sender key — efficient fan-out with E2EE.

Key Storage

What Server Stores vs. Client

Server stores:

  • Public identity key
  • Public signed prekeys
  • Public one-time prekeys
  • Encrypted ciphertext blobs
  • Message metadata (timestamp, size)

Server NEVER stores:

  • Private keys (any)
  • Plaintext message content
  • Decryption keys
  • Session state

Key decisions and their consequences
DecisionChosen ApproachTrade-off / Why
DB for messagesCassandra over MySQLSacrifices complex joins for massive write throughput + horizontal scale. Perfect for append-heavy message data.
WebSocket vs HTTP pollingWebSocketHigher complexity (connection state, reconnection) but dramatically lower latency and server load vs long-polling.

Stories / Status
24-hour ephemeral content — WhatsApp Status feature
1

User Posts a Status

Client uploads media to S3 (same presigned URL flow). Status metadata posted to POST /status. Status Service writes to status table with expires_at = now + 24h.

2

Distribution to Contacts

Status is NOT pushed proactively. When contacts open the Status tab, client polls GET /status/feed — returns statuses from contacts respecting privacy settings (who can see my status).

3

View Tracking

When a contact views the status, client sends POST /status/:id/view. Server writes to status_views table. Owner can see viewer list. View count cached in Redis.

4

Auto-Expiry (TTL)

A cron job / Kafka Streams job runs every 5 min. Deletes statuses where expires_at < now. Triggers S3 object deletion via lifecycle rule. Removes from feed cache.

Broadcast Lists
One-to-many messaging — recipients get individual messages

How Broadcast Differs from Groups

In a broadcast, the sender sends ONE message but each recipient receives it as a SEPARATE 1:1 conversation — replies go only to the sender, not to other recipients. Recipients don't know they're part of a broadcast. Max 256 recipients per broadcast list.

Architecture

Broadcast Fan-out

When a broadcast message is sent, the Broadcast Service creates N individual messages (one per recipient) and enqueues them to Kafka topic broadcast.delivery. Workers process deliveries in parallel — each as a normal 1:1 message.

  • 1 broadcast → N delivery tasks
  • Each task = individual chat message
  • Kafka partitioned by recipient_id
  • No shared conversation thread
  • Replies → private to sender only
Constraints

Rules & Limitations

  • Max 256 recipients per list
  • Recipient must have sender in contacts (anti-spam)
  • No read receipt aggregate for sender
  • Cannot forward broadcast to group
  • Broadcast list stored client-side (not server)
  • E2EE: each recipient gets separate ciphertext

broadcast_lists table

Disappearing Messages
TTL-based message auto-deletion
How It Works

Server-side TTL

When disappearing messages are enabled for a chat, all new messages get a disappears_at timestamp. A Cassandra TTL is set on the message row. The client-side timer is derived from the server timestamp — no client can cheat by system clock manipulation.

  • Timer options: 24h, 7 days, 90 days
  • Set per-chat by either participant (1:1) or admin (group)
  • Cassandra TTL handles server-side deletion
  • Client deletes on timer expiry
  • Media files: S3 lifecycle rule matches TTL
  • Forwarded messages keep their own timer
Edge Cases

Tricky Scenarios

  • Offline recipient: Message expires before they come online → delivered as "deleted message" placeholder
  • Screenshots: Cannot prevent — show screenshot notification (like Snapchat) optionally
  • Quoted/replied messages: Parent deletes, but quoted preview in reply remains visible
  • Backup: Disappearing messages are excluded from chat backup by default
  • Starred messages: Starred DMs are excluded from auto-delete

disappears_atTIMESTAMP NULL — null means never disappears. Set on insert if chat has disappearing mode on.

Reactions & Polls
Emoji reactions on messages + in-chat polls

Reactions Design

Reactions are stored as a MAP<user_id, emoji> directly on the message row in Cassandra (one reaction per user per message). This avoids a separate reactions table and keeps fan-out simple — just update the message row and broadcast message:reaction_updated event.

Reaction Flow

Add / Change / Remove

  • Client sends message:react via WS with emoji + message_id
  • Server: UPDATE messages SET reactions[user_id] = emoji WHERE …
  • Server broadcasts updated reactions map to chat members
  • Remove: send null emoji → DELETE reactions[user_id]
  • One reaction per user per message (overwrite allowed)
  • Emoji set: WhatsApp limits to 6 preset emojis
Polls

Poll Architecture

Polls are a special message type (type = poll). Poll options and votes stored in a separate polls table. Real-time vote updates broadcast to group via WS event poll:updated.

  • polls table: id, message_id, question, options JSON, is_multi_select, closes_at
  • poll_votes: poll_id, user_id, option_index, voted_at
  • Vote = INSERT or UPDATE poll_votes
  • Aggregate counts cached in Redis
  • Real-time: WS broadcasts poll:vote_update
  • Anonymous polls: votes stored but names hidden
Other Message Interactions

Reply, Forward, Star

  • Reply: reply_to_id FK on messages table. Preview text + sender shown in bubble
  • Forward: New message row, forwarded_from_id meta. "Forwarded" label shown
  • Star: starred_messages table (user_id, message_id, starred_at). Accessible from profile menu
  • Pin: pinned_messages table per chat. Max 3 pins. Shown in banner

Chat Backup
Encrypted cloud backup to Google Drive / iCloud
Backup Flow

How Backup Works

Backup is CLIENT-SIDE generated. The app exports all local messages to an encrypted archive, then uploads to the user's personal cloud storage (Google Drive / iCloud). WhatsApp servers never store backup data.

  • Backup key = derived from user's password / pin
  • AES-256 encryption of backup archive
  • Upload via Google Drive API / iCloud API
  • Schedule: daily/weekly/monthly or manual
  • Media optionally included (size warning)
  • End-to-end encrypted backup (since 2021)
Restore Flow

Restoring on New Device

  • New device login → prompt for backup restore
  • User authenticates with Google/Apple
  • App downloads encrypted backup file
  • User provides backup key / password to decrypt
  • Messages imported to local IndexedDB / SQLite
  • Media fetched on-demand from backup or S3
Security Note

E2EE Backup Trade-off

Traditional backup (pre-2021) broke E2EE because the key was held by WhatsApp. End-to-end encrypted backup uses a 64-digit key that only the user holds — not WhatsApp, not Google/Apple. If lost → backup permanently inaccessible.

  • Key stored in HSM on user's device
  • Optional: password as recovery factor
  • WhatsApp HSM service stores encrypted key (not plain)

Multi-Device Support
Up to 4 linked devices — each independently capable

Key Challenge: E2EE + Multi-Device

Each device has its own identity key pair. When you send a message, it must be encrypted separately for EACH linked device of the recipient. For a user with 4 devices receiving from a user with 4 devices → 1 message becomes 4 × 4 = 16 ciphertexts. This is the "sender key" efficiency optimization used in groups.

How Linking Works

Device Registration Flow

  • Primary device shows QR code containing a one-time token
  • New device scans QR → sends token to auth server
  • Server validates token, registers device_id + public key
  • Primary device syncs: contacts, chat list, group keys
  • New device downloads last 90 days message history
  • Each device gets its own WS connection + push token
  • Max 4 linked devices (WhatsApp limit)
Message Sync

Keeping Devices in Sync

  • All sent/received messages delivered to ALL linked devices
  • Read receipt on one device → synced to all (read:sync event)
  • Deleted message → broadcast delete:sync to all devices
  • Device offline → messages queued, delivered on reconnect
  • Queue stored in Cassandra (offline_messages table) with device_id
  • Primary phone can be offline — web/desktop still works

Local Database & Storage
On-device encrypted storage — messages, media, keys

Why On-Device DB?

Chat apps need instant message display with zero latency — hitting the server for every scroll event is unacceptable. A local SQLite database (encrypted with SQLCipher) stores recent messages, contacts, and settings. The server is the source of truth; the device is a synchronized cache.

SQLCipher (Encrypted SQLite)

Local Message Store

All on-device message data encrypted with AES-256. Database key derived from user's PIN + device key stored in Keychain/Keystore. Schema mirrors server but is trimmed (last 1000 msgs per chat by default).

  • messages — local copy, same schema as server
  • chats — conversation list + last message
  • contacts — phone book + registered users
  • media_cache — local file paths for downloaded media
  • pending_messages — outbox for offline sends
  • signal_sessions — E2EE session states (critical)
  • prekeys — one-time prekeys awaiting use
File System Storage

Media File Organization

  • /WhatsApp/Media/Images/ — received images
  • /WhatsApp/Media/Video/ — received videos
  • /WhatsApp/Media/Voice/ — voice messages
  • /WhatsApp/Media/Documents/ — PDFs, docs
  • /WhatsApp/Media/.Statuses/ — cached statuses
  • Sent media kept in original camera roll
  • Media not downloaded until user taps
  • Auto-download respects WiFi/cellular setting
Secure Key Storage

iOS Keychain / Android Keystore

Cryptographic keys are NEVER stored in the SQLite database or UserDefaults/SharedPreferences. Platform-provided secure enclaves are used.

  • Identity key pair → Keychain/Keystore
  • JWT refresh token → Keychain/EncryptedSharedPrefs
  • DB encryption key → Keychain/Keystore
  • Backup key → Secure Enclave (iOS) / StrongBox (Android)
  • Biometric-gated access for sensitive operations
Storage Limits & Cleanup

Disk Management

  • Storage usage screen shows breakdown by media type
  • Per-chat storage view: tap → "Free up space"
  • Auto-delete old media when disk < 500MB free
  • Messages: no auto-delete (only disappearing msgs)
  • Thumbnail cache LRU eviction
  • Temp files cleaned on app foreground

Mobile Network Layer
Offline-first design, smart reconnection, bandwidth optimization
Offline-First Strategy

How the App Handles No Connectivity

  • All reads served from local SQLite — zero network dependency
  • Sends go to pending_messages outbox table
  • Message shown as "pending" (clock icon) in UI
  • Network monitor (NWPathMonitor on iOS / ConnectivityManager on Android) watches for connectivity
  • On reconnect: flush outbox in FIFO order
  • WS reconnect with exponential backoff: 1s → 2s → 4s → 8s → max 60s
  • Missed messages fetched via REST on reconnect
Network Optimization

Bandwidth Efficiency

  • Protobuf binary serialization (vs JSON — 3-10x smaller)
  • HTTP/2 multiplexing — no head-of-line blocking
  • Delta sync — only fetch messages newer than last_message_id
  • Image thumbnails first, full image on tap
  • Adaptive media quality: 4G → high, 3G → medium, 2G → low
  • Compressed WebSocket frames (permessage-deflate)
  • Request coalescing: batch multiple API calls

Mobile Media Handling
Camera, compression, upload, gallery — end to end
1

Capture / Select

User opens camera (CameraX / AVFoundation) or picks from gallery (MediaPicker / PHPickerViewController). App requests CAMERA + READ_MEDIA_IMAGES permissions at runtime.

2

Client-Side Compression

Before upload: images resized to max 1600px, converted to WebP (quality 85%). Videos transcoded to H.264/AAC in a background thread. EXIF GPS data stripped for privacy. Runs on background thread (DispatchQueue / Coroutine).

3

Chunked Upload to S3

Large files use S3 multipart upload (5MB chunks). Progress tracked in UI. If connection drops, upload resumes from last successful chunk using upload_id. Foreground service (Android) keeps upload alive.

4

Optimistic UI + Thumbnail

App shows local thumbnail immediately (before upload completes). Message appears in chat with upload progress bar. Only after server ACKs the media_id does status change to "sent".

5

Receiving & Caching

Received image shows thumbnail (100px, pre-generated by server Lambda). Full image downloaded lazily on scroll-into-view or tap. Cached in LRU disk cache (max 200MB). Evicted using LRU when limit exceeded.

Developer pushes code │ ▼ GitHub Actions / Bitrise / Fastlane ├── Lint (SwiftLint / Detekt / ESLint) ├── Unit Tests (XCTest / JUnit5) ├── Build (xcodebuild / Gradle) ├── UI Tests on Simulator/Emulator ├── Snapshot Tests (diff check) ├── Code Coverage Gate (>80%) └── Security Scan (Snyk / MobSF) │ ▼ (on merge to main) Build Artifacts ├── iOS: .ipa signed with Distribution cert (Fastlane match) └── Android: .aab signed with Keystore (Gradle signing config) │ ▼ Distribution Stages ├── Internal (same day) → TestFlight Internal / Firebase App Distribution ├── Beta (weekly) → TestFlight External / Play Internal Testing ├── Staged Rollout → 1% → 5% → 25% → 100% users └── Production → App Store / Play Store public release
Fastlane Automation

Key Fastlane Lanes

  • lane :beta — build + sign + upload to TestFlight
  • lane :release — bump version, tag, submit to App Store
  • match — sync certificates + provisioning profiles via Git
  • scan — run all tests and generate JUnit report
  • snapshot — auto-generate App Store screenshots in all locales
  • deliver — upload metadata + screenshots to App Store Connect
Feature Flags

Safe Feature Rollouts

New features wrapped in feature flags (Firebase Remote Config / LaunchDarkly). Ship code dark, enable for 1% → 10% → 100%. Instant kill switch if issues found — no need for emergency app release.

  • Flag: enable_new_reactions_ui
  • Flag: enable_polls_feature
  • A/B test new UI before full rollout
  • Server-side flag eval — no app update needed
OTA Updates

Hot Patch (React Native Only)

React Native apps can use CodePush (Microsoft App Center) to ship JS bundle updates without going through App Store review. Critical bug fixes can reach users in minutes.

  • JS/assets updated OTA instantly
  • Native code changes still need store review
  • Rollback if crash rate spikes
  • Not available for native Swift/Kotlin apps
Monitoring Post-Release

Crash & Performance Monitoring

  • Firebase Crashlytics — real-time crash alerts
  • Sentry — errors + performance traces
  • ANR / App Not Responding alerts (Android vitals)
  • Core Web Vitals equivalent: App startup time
  • Rollback trigger: crash-free sessions < 99.5%
  • Staged rollout pause on crash spike detection
