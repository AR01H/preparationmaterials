01 — INTRODUCTION

FreshBowl Subscription Platform

A full-stack, subscription-based food delivery web & mobile application delivering daily fruit bowls, healthy snack bowls, and nutrition items to customers on Daily, Weekly, and Monthly plans.

What We Are Building

FreshBowl is not just a food ordering app — it's a recurring delivery subscription engine. The key insight is that once a customer subscribes, every delivery should happen automatically without the customer doing anything. The system generates daily orders, assigns delivery agents, routes deliveries, processes payments, and sends notifications — all without manual intervention.

Core Value Proposition "Subscribe once, eat healthy every day." Customers pick a plan, customize their bowl, set a delivery time — and fresh nutrition arrives at their door daily without re-ordering.

03 — ARCHITECTURE

System Topology

End-to-end view of all system components, how they interact, and where they live.

Architecture Decision: Modular Monolith First

We choose a Modular Monolith for Phase 1. This means all backend code lives in one deployable unit, but is structured as strict independent modules with no cross-module imports except through well-defined interfaces. This gives startup speed now, and a clear microservices migration path later.

Why Not Microservices from Day 1? Microservices add massive operational complexity (service discovery, inter-service auth, distributed tracing, network latency). For a startup, this slows development by 2–3x. A well-structured modular monolith is easy to split when you actually need to scale a specific module.

Inter-Module Communication

Modules never import each other's repositories. They communicate via:

📢 EventEmitter2 (In-process Events)
When SubscriptionService creates a subscription, it emits subscription.created. The NotificationModule listens and sends a welcome message. Zero coupling between modules.
🔌 Service Injection
When OrderModule needs to know if a subscription is active, it injects SubscriptionService and calls its public method. Never touches SubscriptionRepository directly.

Fetch all ACTIVE subscriptions for today const activeSubs = await this.subRepo.findActive({ deliveryDate: tomorrow, statuses: ['ACTIVE'], }); let created = 0, skipped = 0; for (const sub of activeSubs) { // 2. Authentication & Authorization Deep Dive

Token Strategy

⚡ Access Token (JWT)
Lifetime: 15 minutes
Storage: In-memory (React state) — never localStorage
Contents: userId, role, email
Verified: On every API request at NestJS guard layer
Rotation: New one fetched silently when expired via refresh token
🔄 Refresh Token
Lifetime: 30 days
Storage: HTTP-only cookie (XSS-proof)
Strategy: Rotation — each use issues a new refresh token, old one is immediately revoked
DB: Stored hashed in refresh_tokens table (one per session)

// Proration calculation for upgrade: Daily → Monthly function calculateProration(sub: Subscription, newPlan: Plan) { const daysRemaining = differenceInDays(sub.endDate, today()); const dailyRate = sub.plan.price / sub.plan.durationDays; const creditForUnused = dailyRate * daysRemaining; const newPlanTotal = newPlan.price; const chargeNow = Math.max(0, newPlanTotal - creditForUnused); return { creditForUnused, // ₹ credit from old plan remaining days chargeNow, // ₹ to charge immediately for new plan newEndDate: addDays(today(), newPlan.durationDays), }; }

Security

21 — SECURITY

Security Architecture

Encryption, PCI compliance, input validation, rate limiting, and secret management.

Security Layers

🔒 Encryption
In transit: TLS 1.3 enforced via CloudFront + ALB. HTTP redirected to HTTPS.
At rest: RDS storage encrypted (AES-256). S3 server-side encryption. Redis ElastiCache encrypted.
Passwords: bcrypt with cost factor 12. Never stored plain.
🛑 Rate Limiting
Nginx + Redis-backed rate limiting:
• Auth endpoints: 5 req/min per IP
• OTP send: 3 per phone per 15 min
• General API: 100 req/min per user
• Admin API: 200 req/min
429 response with Retry-After header.
💳 PCI Compliance
Zero card data passes through our servers. Razorpay handles all card processing. We store only:
• Razorpay payment_id (reference)
• Razorpay subscription_id (reference)
• Masked card last 4 digits (from webhook, display only)
🧪 Input Validation
All API inputs validated with Zod schemas at controller level before reaching service layer. Prisma parameterized queries prevent SQL injection. Content-Security-Policy headers prevent XSS. CORS whitelist: only freshbowl.in domains.
🔑 Secret Management
All credentials in AWS Secrets Manager. Never in .env files in production. Accessed via AWS SDK at runtime. Rotated quarterly. Local dev uses .env.local (gitignored).
📋 Audit Log
Every payment event, subscription change, admin action, and login written to audit_log table. Append-only (no UPDATE/DELETE on this table). Retained 2 years for compliance. Queryable by admin.

Scaling Strategy

How the system scales from 100 to 100,000 subscribers without architectural rewrites.

Current Architecture Limits

ComponentCurrent CapacityScales ToHow
API (ECS Fargate)500 concurrent10,000+Auto-scale ECS tasks. Add more task instances.

Redis (ElastiCache)100K cache keysMillionsCluster mode with sharding.
Order Generation Job10,000 orders in 5 min100,000+Parallelize with multiple BullMQ workers. Batch DB inserts.
Notifications10K/day1M+/dayIncrease BullMQ concurrency. Rate-limit per channel to avoid provider limits.

Database Scaling Path

Phase 1 (0–5K subs): Single RDS db.t3.medium Phase 2 (5K–20K subs): Add 1 Read Replica → analytics queries go to replica Phase 3 (20K–100K): Upgrade to db.r6g.large + 2 read replicas Add connection pooler (PgBouncer via RDS Proxy) Phase 4 (100K+): Partition orders table by month (range partitioning) Consider TimescaleDB for time-series analytics

Microservices Extraction Order

When specific modules hit limits, extract them in this priority order:

  1. Notification Service — highest message volume, needs dedicated rate limiting
  2. Order Generation Service — can parallelize across multiple instances
  3. Analytics Service — separate read DB, OLAP workload
  4. Payment Service — isolation for PCI compliance simplification
Key Design Decision: Because we built the monolith with strict module boundaries and event-based communication, extracting a module to a microservice means: move the module code to a new repo, replace EventEmitter with Kafka/RabbitMQ events, replace in-process service calls with HTTP/gRPC. The business logic doesn't change — only the transport layer.
