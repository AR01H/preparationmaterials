01 — System Design

High-Level Architecture

RideFlow uses a microservices architecture deployed on Kubernetes, with separate services for user management, ride/route matching, real-time tracking, payments, notifications, and analytics.

04 — State

State Management Strategy

Zustand for local UI state, React Query (TanStack Query) for server state, Socket.io for real-time map/tracking updates. No Redux overhead.
🐻 Zustand Stores
authStore, bookingStore, mapStore, driverStore. Lightweight slices with persist middleware for tokens.
🔄 React Query
All API calls, with cache invalidation on mutations. 30s stale time for ride listings, 0s for seat availability.
⚡ Socket.io Hooks
useTrackingSocket(), useDriverSocket(). Auto-reconnect, heartbeat, optimistic map updates.
📦 Local Storage
Encrypted auth tokens via Expo SecureStore (mobile), httpOnly cookies (web).

Each service is independently deployable, has its own database, and communicates via REST/gRPC or Kafka events. Domain-driven design boundaries.

07 — Events

Real-time & Event-Driven Design

Kafka topics handle async communication between services. WebSockets (Socket.io) deliver live updates to clients.

Kafka TopicProducers → Consumers
booking.createdbooking → payment, notify
booking.confirmedpayment → notify, tracking
booking.cancelledbooking → payment, notify
driver.locationdriver-app → tracking, analytics
trip.starteddriver → notify, analytics
trip.completeddriver → payment, review, analytics
payment.successpayment → booking, notify
payment.failedpayment → booking, notify
sos.triggeredsafety → notify, analytics

WebSocket rooms are scoped per booking. The tracking-service is the only WebSocket server; other services publish to it via Kafka.

WebSocket Events (Client ← Server)
  • vehicle:location — lat/lng/heading update
  • trip:eta — updated ETA in seconds
  • booking:status — lifecycle changes
  • seat:reserved — other user took seat
  • sos:received — admin alert
Driver location updates are rate-limited to 1 update/5s per vehicle. Redis pub/sub distributes to Socket.io clusters.

12 — Security

Security Architecture

🔐 Authentication JWT + OTP
  • JWT access tokens (15m expiry)
  • Refresh tokens in httpOnly cookies
  • OTP via SMS/email (6-digit, 5min TTL)
  • OAuth2: Google, Apple Sign-In
  • TOTP 2FA for admin accounts
  • Device fingerprinting for anomaly
🛡️ API Security WAF
  • Rate limiting: 100 req/min per user
  • IP allowlisting for admin endpoints
  • Helmet.js security headers
  • CORS policy (allowlisted origins)
  • Input sanitization + Zod validation
  • SQL injection prevention (Prisma)
🔒 Data Protection GDPR
  • AES-256 encryption at rest (RDS)
  • TLS 1.3 in transit everywhere
  • PII tokenization (phone, email)
  • Payment data never stored (Stripe vault)
  • GDPR: right to erasure endpoint
  • Audit logs for all data access
🚨 Fraud & Safety ML
  • ML fraud scoring on bookings
  • Duplicate booking detection
  • Unusual GPS pattern alerting
  • SOS panic button with location share
  • Driver background check integration
  • Vehicle insurance expiry alerts
📜 Compliance
  • PCI-DSS via Stripe (no card storage)
  • GDPR data residency (EU region)
  • SOC 2 Type II roadmap
  • 90-day pen testing schedule
  • Vulnerability disclosure policy
  • AWS Config + Security Hub
🗝️ Secrets Management
  • AWS Secrets Manager for all credentials
  • K8s External Secrets Operator
  • No secrets in environment vars or code
  • Rotation: DB passwords every 30 days
  • GitLeaks pre-commit hook
  • Vault for dynamic DB credentials

14 — User Journey

End-to-End Booking Flow

1
Search for a Route
Rider enters origin, destination, date, and passenger count. route-service runs geo-search with PostGIS, returns matching trips sorted by departs_at and price.
GET /api/v1/routes/search?from=Mumbai&to=Pune&date=2026-05-01&seats=2&type=bus
2
Select Seats
Rider sees a real-time seat map. Available seats fetched from Redis (TTL 5s). Seat map shows booked/available/locked states.
GET /api/v1/routes/{tripId}/seats ← Redis cache (5s TTL)
3
Create Booking (Optimistic Lock)
booking-service acquires a Redis lock on selected seats for 5 minutes. Booking created in status "pending". Kafka event: booking.created.
POST /api/v1/bookings {tripId, seatNumbers:[12,13], promoCode:"FIRST20"}
4
Payment
payment-service creates a Stripe PaymentIntent or Razorpay order. Client completes 3DS if required. Webhook confirms payment → booking status → "confirmed". Kafka event: payment.success.
POST /api/v1/payments/initiate {bookingId} POST /api/v1/payments/webhook ← Stripe/Razorpay webhook
5
Confirmation & QR Code
Rider receives push notification + email with booking confirmation and QR boarding pass. QR token is a signed JWT (HS256) containing bookingId + seatNumbers.
6
Day of Travel — Live Tracking
Rider opens app → connects to tracking WebSocket room. Driver app pushes GPS every 5s. Tracking service computes ETA and broadcasts via Socket.io to all riders in that trip.
WS /track/{bookingId} ← {type:"vehicle:location", lat:18.52, lng:73.85, etaSec:420}
7
Boarding (QR Check-in)
Driver scans rider's QR with driver app. Server validates QR token signature and checks bookingId is for this trip. Booking status → "active".
8
Trip Completion & Review
Driver marks trip complete → booking status → "completed". Rider receives review prompt (24h window). Driver earnings credited to wallet.
15 — Payments

Payment Architecture

💳
Payment Methods
Multi-gateway
  • Credit/Debit cards via Stripe (global)
  • UPI via Razorpay (India)
  • In-app wallet (pre-funded balance)
  • Net banking integration
  • EMI / BNPL via Simpl (future)
💰
Fare Calculation
Dynamic pricing engine
  • Base fare × surge multiplier
  • Surge: demand/supply ratio (Redis)
  • Per-stop fare for partial journeys
  • Promo code discount applied server-side
  • Loyalty cashback (5% for frequent riders)
  • GST/VAT calculated by region
🔄
Cancellation & Refund Policy
Automated engine
  • >24h before departure: 100% refund
  • 12–24h before: 75% refund
  • 2–12h before: 50% refund
  • <2h before: no refund
  • Driver cancellation: 100% refund + ₹50 credit
  • Refund processed in 5–7 business days
  • Wallet refund: instant
All payment webhooks are verified via HMAC signature. Idempotency keys prevent double-charging on network retries. Payment records are immutable — refunds create new records.
