Docker — Concepts

The mental model you need before running commands

Core concepts
1

Image vs Container

An Image is a blueprint (like a class). A Container is a running instance of that image (like an object). Images are immutable; containers are ephemeral.

2

Dockerfile

A text file with instructions to build an image. Each instruction creates a layer. Layers are cached — only changed layers re-run.

3

Multi-stage builds

Build your app in one stage (with all dev deps), copy only the output to a slim final stage. Our frontend uses this: Stage 1 = Node.js (builds React), Stage 2 = Nginx (serves static files).

4

Docker Compose

Define and run multiple containers together. Handles networking (services find each other by name), volumes, and environment variables. Perfect for local dev.

5

Volumes

Named volumes (like pgdata) persist data beyond a container's life. Bind mounts link host directories into containers for live reloading.

Essential Docker commands
# Build an image
docker build -t myapp:latest .

# Run a container
docker run -p 3000:80 -d myapp:latest

# List running containers
docker ps

# View logs
docker logs <container-id>

# Shell into container
docker exec -it <container-id> sh

# Remove all stopped containers
docker container prune

# Push to Docker Hub
docker tag myapp:latest username/myapp:latest
docker push username/myapp:latest

Docker — Practice Steps

Hands-on steps from zero to running app

Step 1 — Install Docker
# macOS
# Download Docker Desktop from https://docker.com/products/docker-desktop

# Ubuntu/Linux
sudo apt-get update
sudo apt-get install docker.io docker-compose-v2
sudo usermod -aG docker $USER

# Verify
docker --version
docker compose version
Step 2 — Build images individually
# Build backend
cd backend
docker build -t todo-backend:v1 .

# Build frontend
cd ../frontend
docker build -t todo-frontend:v1 .

# List your images
docker images

# Inspect image layers
docker history todo-backend:v1
Step 3 — Run with Docker Compose
# From the todo-app/ root directory
cd todo-app

# Start all 3 services (db, backend, frontend)
docker compose up --build

# Run in background
docker compose up -d --build

# View logs for specific service
docker compose logs -f backend

# Stop everything
docker compose down

# Destroy including volumes (wipes db!)
docker compose down -v
💡
App runs at http://localhost:3000. API at http://localhost:5000/api/todos.
Step 4 — Practice exercises
  • →Change the backend port from 5000 to 8080. Update docker-compose.yml. Rebuild.
  • →Add a .dockerignore file. Put node_modules in it. Notice smaller build context.
  • →Shell into the running postgres container and run psql -U postgres tododb to view data.
  • →Kill the backend container. Watch the frontend get an error. Restart it. This is why health checks matter.
  • →Add an environment variable APP_VERSION=1.0.0 to the backend and print it in the /health endpoint.

Kubernetes — Concepts

The building blocks of container orchestration

Core objects — what each file does
📦

Pod

Smallest deployable unit. One or more containers sharing network and storage. You almost never create Pods directly — Deployments manage them.

🔄

Deployment

Declares desired state: "I want 2 replicas of this container image." K8s continuously reconciles reality to match. Auto-restarts crashed pods.

🌐

Service

Stable network endpoint for a set of pods. Types: ClusterIP (internal), NodePort (external via node), LoadBalancer (cloud LB).

🚪

Ingress

HTTP/HTTPS routing rules. Routes /api → backend service, / → frontend service. One external IP for multiple services.

🔒

Secret

Stores sensitive data (passwords, tokens) as base64. Injected as env vars or files. Never hard-code passwords in Deployments.

💾

PersistentVolumeClaim

Requests storage. The cluster provisions it. PostgreSQL uses this so data survives pod restarts.

📈

HorizontalPodAutoscaler

Watches CPU/memory metrics. Scales replica count up/down automatically. Our backend goes from 2 → 10 replicas under load.

Kubernetes — Practice Steps

From local minikube to production cluster

Step 1 — Setup local cluster with minikube
# Install minikube
brew install minikube      # macOS
# Linux: curl -Lo minikube https://storage.googleapis.com/minikube/releases/latest/minikube-linux-amd64

# Start cluster
minikube start --driver=docker --memory=4g

# Enable ingress addon
minikube addons enable ingress
minikube addons enable metrics-server

# Check nodes
kubectl get nodes
Step 2 — Push images and deploy
# Load local images into minikube
minikube image load todo-backend:v1
minikube image load todo-frontend:v1

# Create secret first!
kubectl apply -f k8s/secret.yaml

# Deploy everything
kubectl apply -f k8s/manifests.yaml
kubectl apply -f k8s/hpa.yaml

# Watch pods come up
kubectl get pods -w
Step 3 — Key kubectl commands to practice
# Get everything in default namespace
kubectl get all

# Describe a pod (events, resource usage)
kubectl describe pod <pod-name>

# View logs
kubectl logs -f <pod-name>

# Shell into a running pod
kubectl exec -it <pod-name> -- sh

# Scale manually
kubectl scale deployment backend --replicas=4

# Rolling update (zero-downtime deploy)
kubectl set image deployment/backend backend=todo-backend:v2

# Rollback
kubectl rollout undo deployment/backend

# View rollout history
kubectl rollout history deployment/backend

# Port-forward to test locally
kubectl port-forward svc/backend 5000:5000
Practice exercises
  • →Delete a backend pod manually. Watch K8s restart it automatically.
  • →Update the backend image to v2. Observe rolling update. Rollback to v1.
  • →Use kubectl top pods to see CPU/memory usage.
  • →Add resource limits to the backend deployment: cpu: 100m, memory: 128Mi.
  • →Create a ConfigMap for non-secret env vars (like DB_NAME) and use it in the deployment.
  • →Run a load test with hey or k6 and watch HPA scale the backend.

AWS Services — Full Coverage

Every major service, what it does, and how to practice it

🖥️ EC2
Virtual machines in the cloud. You pick OS, CPU, RAM, storage. Full control. SSH in and run anything.
Practice: Deploy your backend directly on EC2. SSH in, install Node.js, run it as a systemd service.
☸️ EKS
Managed Kubernetes control plane. AWS runs the masters, you manage worker nodes. Perfect for our K8s deployment.
Practice: Create an EKS cluster with eksctl and migrate your K8s manifests to it.
🐳 ECS
Managed container orchestration (alternative to K8s). Simpler than EKS. Fargate mode = serverless containers.
Practice: Run your Docker Compose app on ECS Fargate without managing servers.
⚡ Lambda
Serverless functions. Run code without servers. Pay per invocation. Great for APIs, cron jobs, event processing.
Practice: Move the GET /todos endpoint to Lambda behind API Gateway.
🔧 Elastic Beanstalk
PaaS for web apps. Upload code, AWS handles deployment, scaling, load balancing. Good for quick deploys.
Practice: Deploy your Express backend with one command: eb deploy.
🌐 CloudFront
CDN with 400+ edge locations. Cache and serve your React build globally with low latency.
Practice: Put CloudFront in front of your S3-hosted React app.
🪣 S3
Object storage. Store any file with near-infinite scale. Static website hosting, backups, data lakes, CI artifacts.
Practice: Build your React app, upload dist/ to S3, enable static hosting.
💾 EBS
Block storage for EC2. Like a hard drive attached to your VM. Used for OS disks and databases on EC2.
Practice: Attach an EBS volume to EC2, format it, mount it, write data, detach, re-attach.
📁 EFS
Elastic File System. NFS-compatible shared storage. Multiple EC2 instances can mount the same filesystem.
Practice: Share uploaded files across multiple backend EC2 instances with EFS.
🗄️ Glacier
Ultra-cheap archival storage. Retrieval takes minutes to hours. For compliance data, old backups.
Practice: Set up an S3 lifecycle policy to archive data to Glacier after 90 days.
🔒 VPC
Virtual Private Cloud. Your isolated network in AWS. Control subnets, routing, internet access, security groups.
Practice: Create a VPC with public subnet (frontend) and private subnet (DB). No direct internet to DB.
⚖️ ALB/NLB
Application/Network Load Balancers. Distribute traffic across instances. ALB = HTTP/path routing. NLB = TCP/UDP.
Practice: Put an ALB in front of your EKS cluster. Route /api to backend, / to frontend.
🌍 Route 53
DNS service. Map your domain to AWS resources. Health checks, failover routing, latency routing.
Practice: Register a domain, create a hosted zone, point it to your ALB.
🛡️ WAF
Web Application Firewall. Block SQL injection, XSS, bot traffic. Attach to ALB or CloudFront.
Practice: Enable AWS-managed WAF rules on your ALB to block common attacks.
🔗 API Gateway
Managed API frontend. Rate limiting, authentication, request transformation. Pairs with Lambda for serverless APIs.
Practice: Create a REST API in API Gateway, connect it to your Lambda function.
🐘 RDS
Managed relational databases: PostgreSQL, MySQL, Aurora. Handles backups, patching, failover. Replace your K8s postgres with this.
Practice: Create an RDS PostgreSQL instance. Update your backend to connect to it.
⚡ ElastiCache
Managed Redis or Memcached. Add caching to reduce DB load. Cache your GET /todos response.
Practice: Cache the todos list in Redis. Invalidate cache on POST/PUT/DELETE.
🔍 DynamoDB
Fully managed NoSQL. Single-digit ms latency at any scale. Serverless billing. Great for sessions, user data.
Practice: Migrate from PostgreSQL to DynamoDB. Handle missing JOINs differently.
🔎 OpenSearch
Managed Elasticsearch. Full-text search across todos. Analyze logs with Kibana.
Practice: Index your todos into OpenSearch. Add a search endpoint to your backend.
📦 ECR
Elastic Container Registry. Private Docker registry in AWS. Push images here instead of Docker Hub for EKS.
Practice: Push backend and frontend images to ECR. Update K8s manifests to pull from ECR.
🔄 CodePipeline
CI/CD pipeline: source → build → test → deploy. Chain AWS services together.
Practice: Git push → CodePipeline triggers → CodeBuild builds image → deploys to EKS.
🛠️ CodeBuild
Managed build service. Runs your buildspec.yml: install deps, run tests, build Docker image, push to ECR.
Practice: Write a buildspec.yml that builds and pushes your Docker image.
🚀 CodeDeploy
Automate deployments to EC2, ECS, Lambda. Blue/green and rolling update strategies.
Practice: Blue/green deploy your backend — zero downtime by switching traffic between two environments.
🏗️ CloudFormation
Infrastructure as Code. Define all AWS resources in YAML/JSON. Like docker-compose but for cloud infrastructure.
Practice: Write a CloudFormation stack that creates VPC + RDS + EKS cluster.
🔑 IAM
Identity and Access Management. Users, groups, roles, policies. Every AWS action requires IAM permission.
Practice: Create an IAM role for your EKS pods with only the permissions they need (least privilege).
🗝️ Secrets Manager
Store and rotate secrets (DB passwords, API keys). Access from code without hard-coding credentials.
Practice: Move your DB password from K8s secret to AWS Secrets Manager. Fetch it at runtime.
🏢 KMS
Key Management Service. Create and manage encryption keys. Encrypt S3, EBS, RDS, Secrets Manager.
Practice: Enable RDS encryption with a KMS customer-managed key.
📊 CloudWatch
Monitoring, logs, alarms. View container logs, set CPU alarms, create dashboards. Essential for production.
Practice: Set up a CloudWatch alarm that fires when backend CPU > 70%.
📋 CloudTrail
Audit log of every AWS API call. Who did what, when, from where. Compliance and security investigation.
Practice: Find when and who last modified your RDS instance using CloudTrail.

Deploy to AWS — End to End

Complete production deployment walkthrough

Phase 1 — Push images to ECR
# Install AWS CLI and eksctl
brew install awscli eksctl
aws configure   # Enter Access Key, Secret, region

# Create ECR repositories
aws ecr create-repository --repository-name todo-backend
aws ecr create-repository --repository-name todo-frontend

# Authenticate Docker to ECR
aws ecr get-login-password --region ap-south-1 | \
  docker login --username AWS --password-stdin \
  123456789.dkr.ecr.ap-south-1.amazonaws.com

# Tag and push images
docker tag todo-backend:v1 123456789.dkr.ecr.ap-south-1.amazonaws.com/todo-backend:v1
docker push 123456789.dkr.ecr.ap-south-1.amazonaws.com/todo-backend:v1
Phase 2 — Create EKS cluster
# Create cluster (takes ~15 mins)
eksctl create cluster \
  --name todo-cluster \
  --region ap-south-1 \
  --nodegroup-name workers \
  --node-type t3.small \
  --nodes 2 \
  --nodes-min 1 \
  --nodes-max 4 \
  --managed

# Update kubeconfig
aws eks update-kubeconfig --name todo-cluster --region ap-south-1

# Verify
kubectl get nodes

# Install AWS Load Balancer Controller
helm repo add eks https://aws.github.io/eks-charts
helm install aws-load-balancer-controller eks/aws-load-balancer-controller \
  -n kube-system \
  --set clusterName=todo-cluster
Phase 3 — Create RDS (production DB)
# Create PostgreSQL RDS (instead of K8s postgres pod)
aws rds create-db-instance \
  --db-instance-identifier tododb \
  --db-instance-class db.t3.micro \
  --engine postgres \
  --master-username postgres \
  --master-user-password YourPassword123 \
  --allocated-storage 20 \
  --no-publicly-accessible

# Get the endpoint after creation (~5 mins)
aws rds describe-db-instances \
  --query 'DBInstances[0].Endpoint.Address'

# Store in Secrets Manager
aws secretsmanager create-secret \
  --name prod/tododb \
  --secret-string '{"username":"postgres","password":"YourPassword123"}'
Phase 4 — Deploy to EKS
# Update manifests.yaml to use ECR images, then:
kubectl apply -f k8s/secret.yaml
kubectl apply -f k8s/manifests.yaml
kubectl apply -f k8s/hpa.yaml

# Wait for LoadBalancer IP
kubectl get ingress -w

# Set up CloudWatch Container Insights
aws eks create-addon \
  --cluster-name todo-cluster \
  --addon-name amazon-cloudwatch-observability

# View logs in CloudWatch
aws logs tail /aws/containerinsights/todo-cluster/application
⚠️
Cost alert: EKS cluster costs ~$0.10/hr + EC2 nodes. Delete when done: eksctl delete cluster --name todo-cluster

30-Day Learning Roadmap

A structured path from zero to production

Week 1 — Docker fundamentals
  • ✓Day 1–2: Docker concepts. Build backend image. Shell in, inspect layers.
  • ✓Day 3–4: Docker Compose. Run all 3 services. Practice env vars, volumes, networking.
  • ✓Day 5–7: Multi-stage builds. Optimize image sizes. Push to Docker Hub.
Week 2 — Kubernetes locally
  • ✓Day 8–9: Minikube setup. Deploy all K8s manifests. Practice kubectl commands.
  • ✓Day 10–11: Services, Ingress, rolling updates, rollbacks.
  • ✓Day 12–14: HPA load testing. ConfigMaps. RBAC basics.
Week 3 — AWS core services
  • ✓Day 15–16: IAM, VPC, Security Groups. Network design.
  • ✓Day 17–18: EC2 + RDS. Deploy app the "old school" way first.
  • ✓Day 19–21: S3 + CloudFront for frontend. ECR for images.
Week 4 — Production deployment
  • ✓Day 22–23: EKS cluster. Migrate from minikube to AWS.
  • ✓Day 24–25: ALB Ingress. Route53 domain. HTTPS with ACM.
  • ✓Day 26–27: CloudWatch monitoring. Alarms. Container Insights.
  • ✓Day 28–30: CI/CD with CodePipeline or GitHub Actions → ECR → EKS.
💡
The key insight: Docker Compose and K8s manifests solve the same problem at different scales. docker-compose.yml ≈ a simplified version of Deployments + Services + Networks. Once you see the parallel, Kubernetes becomes much less intimidating.
