Real-World Use Cases

When to Use Which Platform

Industry-specific use cases with recommended platform choices and rationale.

🛍️ E-Commerce Platform (e.g., Amazon-like store)
Recommended: AWS

High traffic variability, need for CDN, search, recommendation engines, payment processing, and inventory management.

AWS Stack
EC2 Auto Scaling → ALB → RDS Aurora → ElastiCache → CloudFront → S3 → SQS → Lambda
Azure Alternative
App Service → Azure SQL → Redis Cache → Azure CDN → Service Bus → Functions
On-Prem (Hybrid)
Use On-Prem for payment data/PCI, burst to cloud for seasonal traffic spikes
🏦 Banking & Financial Services
Preferred: On-Premises Alt: Azure

Strict data sovereignty, regulatory compliance (RBI, SEBI, Basel), real-time transaction processing, fraud detection, and auditability.

  • On-Prem: Full data sovereignty, predictable latency, compliance
  • Azure: Best for hybrid — on-prem SWIFT connected to Azure analytics
  • AWS GovCloud: For US-regulated financial institutions
  • Avoid: Single-cloud for all sensitive data (vendor lock-in risk)
Architecture Pattern
# Core Banking → On-Prem
Core Transactions: IBM Z / Oracle Exadata (On-Prem)
Analytics/Reports: Azure Synapse (Cloud)
Customer Portal: Azure App Service
Fraud Detection: GCP Vertex AI (ML)
DR Site: Azure / AWS Secondary Region
🏥 Healthcare & Hospital Systems
Recommended: Azure Alt: AWS

HIPAA compliance, EHR systems, medical imaging AI, patient portals, telemedicine, and clinical data analytics.

  • Azure: HIPAA BAA available, Azure Health Data Services (FHIR)
  • AWS: HealthLake, Comprehend Medical for NLP on clinical notes
  • GCP: Healthcare API for FHIR/DICOM, AutoML for diagnostics
  • On-Prem: Required for patient-identifiable data in many countries
Healthcare Stack

DICOM Imaging:GCP Healthcare API
AI Diagnosis: AWS Rekognition + SageMaker
Telemedicine: Azure Communication Services
PHI Storage: On-Prem (encrypted)
🤖 AI/ML Startup
Recommended: GCP

Model training at scale, experiment tracking, serving predictions at low latency, data pipelines, and cost-efficient GPU compute.

  • GCP: TPU access, Vertex AI, BigQuery ML, best Kubernetes (GKE)
  • AWS: SageMaker best for MLOps, Trainium/Inferentia chips
  • Azure: OpenAI partnership, Azure ML, GPU VMs
  • On-Prem: Only if you have large GPU farm already
ML Pipeline

🏛️ Government & Defence
Preferred: On-Premises Alt: AWS GovCloud

National data sovereignty, classified workloads, citizen data, e-governance portals, and national ID systems.

  • On-Prem: Mandatory for classified/secret data (NIC in India, DSCC)
  • AWS GovCloud: FedRAMP, DoD IL4/IL5/IL6 compliance (US)
  • Azure Government: DoD Impact Level 5, CJIS, ITAR compliance
  • Public cloud: Never for Top Secret / eyes-only classified data

Real-World End-to-End Scenarios

Scenario: A fashion e-commerce platform handling 10,000 concurrent users, with product search, cart, orders, payments, and recommendations.
1
DNS & CDN Layer

Route 53 routes traffic → CloudFront serves static assets (images, CSS, JS) from S3 edge locations globally with <20ms latency. SSL terminated at CloudFront.

# CloudFront distribution config (CDK)
distribution = CloudFrontWebDistribution(self, "EcomCDN",
  origin_configs=[SourceConfiguration(s3_origin_source=S3OriginConfig(s3_bucket_source=bucket),
    behaviors=[Behavior(is_default_behavior=True, compress=True)])]
)
2
Load Balancer & Auto Scaling

Application Load Balancer (ALB) distributes traffic to EC2 Auto Scaling Group (min:2, max:20 instances). Target Tracking policy based on CPU > 60%.

3
Application Layer (Microservices)

Containerized services on ECS Fargate: Product Service, User Service, Cart Service, Order Service. Each has its own task definition. API Gateway for external APIs.

# ECS Task Definition (simplified)
task_definition = FargateTaskDefinition(stack, "ProductService",
  cpu=512, memory_limit_mib=1024
)
task_definition.add_container("ProductAPI",
  image=ContainerImage.from_ecr_repository(repo),
  port_mappings=[PortMapping(container_port=8080)]
)
4
Caching Layer

ElastiCache Redis caches: product listings (TTL: 5min), user sessions (TTL: 24hr), cart data (TTL: 1hr). Cache hit rate target: >85%

5
Database Layer

Aurora MySQL (Multi-AZ) for orders/users/products. Read replicas for reporting queries. DynamoDB for sessions and cart (single-digit ms latency).

6
Event-Driven Order Processing

Order placed → SQS queue → Lambda processes order → SNS sends notifications (SMS/email via SES) → EventBridge triggers fulfilment workflow

# Lambda function for order processing
import boto3, json
def lambda_handler(event, context):
  for record in event['Records']:
    order = json.loads(record['body'])
    process_order(order['order_id'])
    send_confirmation_email(order['email'])
    update_inventory(order['items'])
7
ML Recommendations

Amazon Personalize provides real-time product recommendations. User behavior logged to S3 → Glue ETL → Redshift for analytics → QuickSight dashboards.

8
Monitoring & Alerting

CloudWatch metrics, dashboards, alarms. X-Ray for distributed tracing. PagerDuty integration for on-call alerts. Cost anomaly detection enabled.

Scenario: A fraud detection ML system that processes 50,000 transactions/sec in real-time, trains models weekly, and serves predictions with <10ms latency.
1
Data Ingestion

Transaction data streams into Cloud Pub/Sub (Kafka equivalent). Topics: raw-transactions, enriched-transactions. 50k msg/sec throughput, 7-day retention.

2
Stream Processing

Dataflow (Apache Beam) reads from Pub/Sub, enriches with user profile data from Bigtable, computes features (rolling averages, velocity checks), writes to BigQuery and Bigtable.

# Apache Beam pipeline (Python)
with beam.Pipeline(options=options) as p:
  transactions = (
    p | 'ReadPubSub' >> beam.io.ReadFromPubSub(topic=TOPIC)
      | 'ParseJSON' >> beam.Map(json.loads)
      | 'EnrichData' >> beam.Map(enrich_with_profile)
      | 'ComputeFeats' >> beam.Map(compute_features)
      | 'WriteBigQuery' >> WriteToBigQuery(TABLE)
  )
3
Feature Store

Vertex AI Feature Store stores precomputed features (last 30 transactions, spending patterns, device fingerprints). Online serving via gRPC with <5ms P99.

4
Model Training

Weekly retraining on Vertex AI Training with NVIDIA T4 GPUs. XGBoost + Neural Network ensemble. MLflow for experiment tracking. Training data in BigQuery.

5
Model Registry & Deployment

Vertex AI Model Registry → canary deploy (5% traffic) → A/B test 1 week → full rollout. Model served via Vertex AI Prediction endpoints (auto-scaling).

6
Real-Time Scoring

Transaction API calls Vertex AI endpoint → prediction returned in <10ms → if fraud probability > 0.85, block transaction and alert via Pub/Sub → case management system.

# Real-time fraud prediction
from google.cloud import aiplatform
endpoint = aiplatform.Endpoint("projects/myproject/locations/us-central1/endpoints/123")

  block_transaction(transaction_id)

7
Model Monitoring & Analytics

Vertex AI Model Monitoring detects data drift. All predictions + outcomes logged to BigQuery. Looker Studio dashboards for fraud analysts. Feedback loop for retraining.

FAQ

Frequently Asked Questions

Common doubts when evaluating cloud platforms, answered directly.

AWS is the most mature and has the largest ecosystem, but it's not always the best choice. Azure wins for Microsoft-heavy enterprises. GCP wins for AI/ML and BigQuery-heavy workloads. Evaluate based on your team's expertise, existing investments, and specific use case — not just popularity.

Single cloud is simpler and cheaper to operate initially. Multi-cloud reduces vendor lock-in and lets you use best-of-breed services (e.g., GCP for BigQuery + AWS for everything else). The complexity overhead of multi-cloud is real — only pursue it if you have clear business reasons.

GCP is generally slightly cheaper for compute and offers sustained use discounts automatically. AWS and Azure are comparable. On-Prem has high upfront CapEx but low OpEx at scale (breaks even vs cloud at ~3-5 years for stable workloads). Committed Use Discounts (CUDs/Reserved Instances) on any cloud can save 40-70%.

Start with: AWS Cloud Practitioner (foundational) → AWS Solutions Architect Associate (most in-demand). For Azure: AZ-900 → AZ-104 → AZ-305. For GCP: Associate Cloud Engineer → Professional Cloud Architect. Multi-cloud: Terraform Associate is platform-agnostic and very valuable.

All major clouds offer compliance certifications. For GDPR: use EU regions and configure Data Processing Agreements. For HIPAA: sign BAA with AWS/Azure/GCP. For RBI (India): use Mumbai regions; some data requires on-prem under RBI guidelines. Always consult legal counsel for specific compliance requirements — cloud certifications don't automatically make your app compliant.

Use the 6Rs strategy: Rehost (lift-and-shift), Replatform (lift-tinker-shift), Repurchase (move to SaaS), Refactor (redesign), Retire (decommission), Retain (keep on-prem). Most migrations start with Rehost for speed. Use AWS Migration Hub, Azure Migrate, or GCP Migrate tools for assessment and migration.

FinOps is cloud financial management — the practice of optimizing cloud spend. Cloud costs can spiral without visibility. Use: AWS Cost Explorer, Azure Cost Management, or GCP Billing. Key practices: right-size instances, use spot/preemptible VMs for batch jobs, set budget alerts, tag all resources, use savings plans. Typical optimization: 30-40% savings.

Deploy across multiple Availability Zones (AZs) within a region — this handles 99.99% of outages. For 99.999% (five nines), deploy Multi-Region with active-active or active-passive setup. Use: Auto Scaling, Load Balancers, Multi-AZ databases, S3/GCS (11 nines durability). Design for failure: implement circuit breakers, retries with exponential backoff, and chaos engineering.
