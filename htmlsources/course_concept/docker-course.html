MODULE 01

Introduction to Docker Beginner

Understand what Docker is, why it exists, and how containerization solves real-world problems.

What is Docker?

Docker is an open-source platform that automates the deployment, scaling, and management of applications using containers. A container is a lightweight, standalone, executable software package that includes everything needed to run a piece of software: code, runtime, system tools, libraries, and settings.

Released in 2013 by Solomon Hykes at dotCloud (now Docker Inc.), Docker democratized containerization — a technology that had existed in Linux for years — by wrapping it in a developer-friendly toolset.

📦
Portable

Runs identically on any machine: laptop, CI server, cloud VM.

⚡
Fast

Starts in milliseconds, not minutes like virtual machines.

🔒
Isolated

Containers share the host kernel but have isolated processes, networks, and filesystems.

📐
Reproducible

Same image = same environment, every single time.

Containers vs. Virtual Machines

The classic question. Both provide isolation, but they work very differently at the OS level.

┌─────────────────────────────────────────────────────────────────┐ │ VIRTUAL MACHINES │ ├────────────────┬────────────────┬────────────────────────────── │ │ App A │ App B │ App C │ │ Bins/Libs │ Bins/Libs │ Bins/Libs │ │ Guest OS │ Guest OS │ Guest OS │ ├────────────────┴────────────────┴────────────────────────────── │ │ Hypervisor │ ├─────────────────────────────────────────────────────────────────│ │ Host OS │ ├─────────────────────────────────────────────────────────────────│ │ Hardware │ └─────────────────────────────────────────────────────────────────┘ ┌─────────────────────────────────────────────────────────────────┐ │ DOCKER CONTAINERS │ ├────────────────┬────────────────┬───────────────────────────────│ │ App A │ App B │ App C │ │ Bins/Libs │ Bins/Libs │ Bins/Libs │ ├────────────────┴────────────────┴───────────────────────────────│ │ Docker Engine │ ├─────────────────────────────────────────────────────────────────│ │ Host OS + Kernel │ ├─────────────────────────────────────────────────────────────────│ │ Hardware │ └─────────────────────────────────────────────────────────────────┘

Core Docker Architecture

Docker uses a client-server architecture. The Docker client talks to the Docker daemon, which does the heavy lifting of building, running, and managing containers.

┌──────────────┐ REST API ┌─────────────────────────────────────┐ │ Docker CLI │ ──────────────────▶ │ Docker Daemon (dockerd) │ │ docker build │ │ │ │ docker run │ ◀────────────────── │ ┌─────────┐ ┌────────┐ ┌──────┐ │ │ docker push │ │ │ Builder │ │ Netw. │ │ Vol. │ │ └──────────────┘ │ └─────────┘ └────────┘ └──────┘ │ └─────────────────────────────────────┘ │ ┌──────────▼──────────┐ │ Container Runtime │ │ (containerd/runc) │ └─────────────────────┘

# Key Components

  • Docker Daemon (dockerd) — Background service managing containers, images, networks, and volumes
  • Docker CLI — Command-line tool you interact with via docker commands
  • Docker Hub / Registry — Repository where images are stored and shared
  • containerd — Industry-standard container runtime that dockerd delegates to
  • runc — OCI-compliant low-level runtime that creates containers

Installing Docker

Shell — Linux (Ubuntu)
# Remove old versions
sudo apt-get remove docker docker-engine docker.io containerd runc

# Install using apt
sudo apt-get update
sudo apt-get install ca-certificates curl gnupg

# Add Docker's official GPG key
sudo install -m 0755 -d /etc/apt/keyrings
curl -fsSL https://download.docker.com/linux/ubuntu/gpg | \
  sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg

# Set up the repository
echo "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.gpg] \
  https://download.docker.com/linux/ubuntu $(. /etc/os-release && echo "$VERSION_CODENAME") stable" | \
  sudo tee /etc/apt/sources.list.d/docker.list > /dev/null

# Install Docker Engine
sudo apt-get update
sudo apt-get install docker-ce docker-ce-cli containerd.io docker-compose-plugin

# Run without sudo (add user to docker group)
sudo usermod -aG docker $USER
newgrp docker

# Verify installation
docker --version
docker run hello-world

macOS & Windows

Install Docker Desktop from https://www.docker.com/products/docker-desktop — it bundles the daemon, CLI, and a GUI dashboard. It includes a lightweight Linux VM to run Linux containers.

MODULE 02

Images & Containers Beginner

Understand the relationship between images and containers, and master the essential CLI commands.

Images vs. Containers

If you come from OOP, think of an image as a class and a container as an instance. An image is a read-only blueprint; a container is a running instance of that blueprint with a writable layer on top.

Docker Image (read-only layers) ┌────────────────────────────┐ │ Layer 4: App code │ ← COPY . /app │ Layer 3: npm packages │ ← RUN npm install │ Layer 2: Node.js 20 │ ← FROM node:20 │ Layer 1: Ubuntu base │ └────────────────────────────┘ │ docker run ▼ Container (adds writable layer) ┌────────────────────────────┐ │ Writable Layer ✏️ │ ← Files written at runtime ├────────────────────────────┤ │ Image layers (read-only) │ └────────────────────────────┘

Multiple containers can run from the same image simultaneously — each gets its own writable layer but shares the same read-only image layers. This is extremely efficient on disk.

⚠️
Containers are ephemeral!

Any data written to the container's writable layer is lost when the container is removed. Use volumes (Module 5) to persist data.

Understanding Image Tags & Registries

Image names follow a structured format:

registry/username/image-name:tag Examples: nginx → docker.io/library/nginx:latest ubuntu:22.04 → docker.io/library/ubuntu:22.04 python:3.11-slim-bullseye → docker.io/library/python:3.11-slim-bullseye ghcr.io/myorg/myapp:v1.2.3 → GitHub Container Registry 123456.dkr.ecr.us-east-1.amazonaws.com/myapp:prod → AWS ECR
ℹ️
Use specific tags in production

Never use :latest in production — it's mutable and can break your deployment. Pin to a specific version like node:20.10.0-alpine3.18.

MODULE 03

Dockerfile Mastery Intermediate

Learn to write production-grade Dockerfiles with multi-stage builds, caching strategies, and best practices.

Dockerfile Fundamentals

A Dockerfile is a text file containing a sequence of instructions that Docker executes to build an image. Each instruction creates a new layer in the image.

Dockerfile — Annotated Reference
# syntax=docker/dockerfile:1

# FROM: base image (required, must be first instruction)
FROM node:20-alpine

# LABEL: metadata (good practice)
LABEL maintainer="you@example.com"
LABEL version="1.0"

# ARG: build-time variables (not in final image env)
ARG NODE_ENV=production

# ENV: runtime environment variables (persisted in image)
ENV NODE_ENV=$NODE_ENV
ENV PORT=3000

# WORKDIR: set working directory (creates if missing)
WORKDIR /app

# COPY: copy files from build context to image
# Copy package files first (for layer caching!)
COPY package*.json ./

# RUN: execute command during build, creates a layer
RUN npm ci --only=production

# COPY remaining source files
COPY . .

# EXPOSE: document which port the app listens on (informational)
EXPOSE 3000

# USER: switch to non-root user (security!)
USER node

# HEALTHCHECK: tell Docker how to check if the container is healthy
HEALTHCHECK --interval=30s --timeout=3s \
  CMD wget -qO- http://localhost:3000/health || exit 1

# CMD: default command when container starts (can be overridden)
CMD ["node", "server.js"]

# ENTRYPOINT: defines the executable (CMD becomes its arguments)
# ENTRYPOINT ["node"]
# CMD ["server.js"]

Layer Caching — The Most Important Concept

Docker caches each layer. If a layer's instruction and its inputs haven't changed, Docker reuses the cached layer instead of rebuilding it. Order matters enormously.

# Bad Order (no caching benefit)

Dockerfile — ❌ Slow Build
FROM node:20
WORKDIR /app
COPY . .           # ← copies ALL files including source
RUN npm install   # ← invalidated on every source file change!
CMD ["node", "app.js"]

# Optimal Order (leverage caching)

Dockerfile — ✅ Fast Build
FROM node:20-alpine
WORKDIR /app

# Step 1: Copy ONLY package files → cached as long as deps don't change
COPY package.json package-lock.json ./

# Step 2: Install deps → cached until package files change
RUN npm ci --only=production

# Step 3: Copy source (changes often, but deps layer stays cached)
COPY src/ ./src/

CMD ["node", "src/app.js"]

Multi-Stage Builds

Multi-stage builds are one of Docker's most powerful features. They let you use one image to build your application and a much smaller image to run it — keeping your final image lean and free of build tools.

Dockerfile — Go Application Multi-Stage Build
# ── Stage 1: Builder ──────────────────────────────────────────
FROM golang:1.21-alpine AS builder

WORKDIR /src

# Download dependencies first (caching)
COPY go.mod go.sum ./
RUN go mod download

# Build the binary
COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build \
    -ldflags="-w -s" \
    -o /app/server ./cmd/server

# ── Stage 2: Final image (tiny!) ──────────────────────────────
FROM scratch

# Copy only the compiled binary from the builder
COPY --from=builder /app/server /server

# Copy SSL certificates (needed for HTTPS calls)
COPY --from=builder /etc/ssl/certs/ca-certificates.crt /etc/ssl/certs/

EXPOSE 8080
ENTRYPOINT ["/server"]

# Final image: ~5MB vs builder image at ~350MB!
Dockerfile — React + Nginx Multi-Stage
# Stage 1: Build React app
FROM node:20-alpine AS build

WORKDIR /app
COPY package*.json ./
RUN npm ci

COPY . .
RUN npm run build   # outputs to /app/dist

# Stage 2: Serve with Nginx
FROM nginx:1.25-alpine

# Copy built assets from 'build' stage
COPY --from=build /app/dist /usr/share/nginx/html

# Copy custom nginx config
COPY nginx.conf /etc/nginx/conf.d/default.conf

EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]

CMD vs ENTRYPOINT

Both define what runs when the container starts, but they work differently in combination:

ScenarioBehavior
CMD ["node", "app.js"]Default command, fully overridable with: docker run myapp python script.py
ENTRYPOINT ["node"]Always runs node; docker run myapp other.js → node other.js
Both: ENTRYPOINT ["node"] + CMD ["app.js"]Default: node app.js. Override CMD only: docker run myapp other.js → node other.js
💡
Best Practice

Use ENTRYPOINT for the executable and CMD for default arguments. Always use the exec form (JSON array) for both: ["executable", "arg1"] — not the shell form executable arg1. The exec form doesn't spawn a shell, so signals (SIGTERM) reach your process directly, enabling graceful shutdown.

Building Images

Shell
# Basic build
docker build -t myapp:latest .

# With a tag and build args
docker build \
  --tag myapp:v2.0 \
  --build-arg NODE_ENV=production \
  --build-arg VERSION=2.0 \
  .

# Build from a different Dockerfile
docker build -f Dockerfile.prod -t myapp:prod .

# Build without cache (force fresh build)
docker build --no-cache -t myapp:latest .

# Build a specific target stage

# Multi-platform build (requires buildx)

  --platform linux/amd64,linux/arm64 \
  -t myapp:latest \
  --push .

# View build cache
docker buildx du

Shell — Working with Networks
# Create a user-defined network
docker network create app-network
docker network create --driver bridge --subnet 192.168.100.0/24 my-net

# List networks
docker network ls

# Inspect network (see connected containers)
docker network inspect app-network

# Run containers on a specific network
docker run -d --name api --network app-network myapi
docker run -d --name db --network app-network postgres:16

# Now 'api' can reach 'db' by name:
# postgres://db:5432/mydb ← 'db' resolves via Docker DNS

# Connect a running container to an additional network
docker network connect monitoring-net api

# Disconnect from a network
docker network disconnect app-network api

# Remove a network
docker network rm app-network

# Remove all unused networks
docker network prune

Port Mapping

By default, containers are isolated — no traffic reaches them. You use -p or --publish to map host ports to container ports:

Shell
# Map host port 8080 → container port 80
docker run -p 8080:80 nginx

# Bind to a specific host interface (localhost only)
docker run -p 127.0.0.1:8080:80 nginx

# Map a random available host port
docker run -p 80 nginx
docker port my-container   # see which port was assigned

# Map multiple ports
docker run -p 80:80 -p 443:443 nginx

# UDP port mapping
docker run -p 5353:5353/udp dns-server

Host & None Networks

Shell
# Host network — bypasses Docker NAT, maximum performance
# Container listens directly on host interface
docker run --network host nginx
# Access nginx at localhost:80 directly (no -p needed or possible)
# Linux only — on macOS/Windows, host network goes to the VM

# None network — total network isolation
# Useful for batch jobs, security-sensitive processing
docker run --network none myapp
🚨
Host Network Security

The --network host mode removes network isolation between the container and the host. Use only in trusted environments where raw network performance is critical (e.g., high-frequency trading, network monitoring).

Volumes & Storage Intermediate

Persist data beyond container lifecycles and share data between containers.

TypeManaged byLocationBest for

tmpfsDocker (RAM)Host memorySensitive data, temp files

Named Volumes

Docker-managed volumes are the recommended approach for production. Docker handles location, backup APIs, and you can use volume drivers for cloud storage.

Shell
# Create a named volume
docker volume create postgres-data

# Use a named volume with -v flag
docker run -d \
  --name postgres \
  -v postgres-data:/var/lib/postgresql/data \
  -e POSTGRES_PASSWORD=secret \
  postgres:16

# Equivalent with --mount (more explicit, preferred)
docker run -d \
  --name postgres \
  --mount type=volume,source=postgres-data,target=/var/lib/postgresql/data \
  -e POSTGRES_PASSWORD=secret \
  postgres:16

# List volumes
docker volume ls

# Inspect a volume
docker volume inspect postgres-data

# Remove a volume (DATA WILL BE LOST!)
docker volume rm postgres-data

# Remove all unused volumes
docker volume prune

# Backup a volume
docker run --rm \
  -v postgres-data:/data \
  -v $(pwd):/backup \
  alpine tar czf /backup/postgres-backup.tar.gz /data

# Restore a volume from backup
docker run --rm \
  -v postgres-data:/data \
  -v $(pwd):/backup \
  alpine tar xzf /backup/postgres-backup.tar.gz -C /

Bind Mounts

Bind mounts map a specific host directory into the container. Changes on either side are reflected immediately — perfect for development workflows.

Shell
# Mount current directory into container (dev workflow)
docker run -d \
  --name dev-server \
  -v $(pwd):/app \       # host:container
  -w /app \
  -p 3000:3000 \
  node:20-alpine \
  npm run dev

# Changes to files on host instantly visible in container!

# Read-only bind mount (prevent container from writing)
docker run -v /etc/nginx/conf.d:/etc/nginx/conf.d:ro nginx

# Using --mount syntax (recommended)
docker run \
  --mount type=bind,source="$(pwd)/config",target=/app/config,readonly \
  myapp
⚠️
Bind Mount Caveat

If you bind-mount a host directory that's empty over a container path that has files, you'll hide the container's files. Be careful not to accidentally mask your node_modules or installed packages this way in development.

tmpfs Mounts

Shell
# Store sensitive data in RAM (never written to disk)
docker run -d \
  --tmpfs /run \
  --tmpfs /tmp:rw,noexec,nosuid,size=100m \
  myapp

# Useful for: session data, secrets, scratch space
# Data is lost when container stops
MODULE 06

Docker Compose Intermediate

Orchestrate multi-container applications with a single YAML file and a single command.

What is Docker Compose?

Docker Compose is a tool for defining and running multi-container applications. You describe your entire stack in a compose.yaml file, then bring it all up with docker compose up.

Think of it as running docker run for multiple containers simultaneously, with automatic networking, dependency management, and environment configuration.

Full Stack Example — Web App + API + DB + Cache

compose.yaml
name: myapp

services:

  # ── Frontend (React) ───────────────────────────────────────────
  frontend:
    build:
      context: ./frontend
      target: production
    ports:
      - "3000:80"
    depends_on:
      api:
        condition: service_healthy
    networks:
      - frontend-net

  # ── API (Node.js) ──────────────────────────────────────────────
  api:
    build:
      context: ./api
      args:
        NODE_ENV: production
    ports:
      - "4000:4000"
    environment:
      DATABASE_URL: postgresql://postgres:${DB_PASS}@db:5432/mydb
      REDIS_URL: redis://cache:6379
      JWT_SECRET: ${JWT_SECRET}
    env_file:
      - .env
    depends_on:
      db:
        condition: service_healthy
      cache:
        condition: service_started
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:4000/health"]
      interval: 30s
      timeout: 5s
      retries: 3
      start_period: 10s
    restart: unless-stopped
    networks:
      - frontend-net
      - backend-net

  # ── PostgreSQL ─────────────────────────────────────────────────
  db:
    image: postgres:16-alpine
    volumes:
      - db-data:/var/lib/postgresql/data
      - ./db/init.sql:/docker-entrypoint-initdb.d/init.sql:ro
    environment:
      POSTGRES_DB: mydb
      POSTGRES_USER: postgres
      POSTGRES_PASSWORD: ${DB_PASS}
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 10s
      timeout: 5s
      retries: 5
    networks:
      - backend-net

  # ── Redis cache ────────────────────────────────────────────────
  cache:
    image: redis:7-alpine
    volumes:
      - redis-data:/data
    command: redis-server --maxmemory 256mb --maxmemory-policy allkeys-lru
    networks:
      - backend-net

volumes:
  db-data:
  redis-data:

networks:
  frontend-net:
  backend-net:
    internal: true   # no external access to backend

Compose CLI Commands

Shell
# Start all services (detached)
docker compose up -d

# Start and rebuild images
docker compose up -d --build

# Start only specific services
docker compose up -d api db

# View running services
docker compose ps

# Follow logs from all services
docker compose logs -f

# Logs from a specific service
docker compose logs -f api

# Execute command in running service
docker compose exec api bash
docker compose exec db psql -U postgres mydb

# Run one-off command (new container)
docker compose run --rm api node scripts/seed.js

# Scale a service to multiple instances
docker compose up -d --scale api=3

# Stop services (preserves volumes)
docker compose stop

# Stop and remove containers + networks
docker compose down

# Stop, remove containers + networks + volumes (DELETES DATA!)
docker compose down -v

# Pull latest images
docker compose pull

# Rebuild specific service
docker compose build api

Override Files & Environments

Docker Compose supports layered configuration — perfect for dev/staging/prod differences:

compose.yaml (base)
services:
  api:
    image: myapp/api:latest
    environment:
      NODE_ENV: production
compose.override.yaml (auto-loaded in dev)
services:
  api:
    build: .                          # build locally in dev
    volumes:
      - .:/app                        # hot reload
    environment:
      NODE_ENV: development
      DEBUG: "*"
    ports:
      - "9229:9229"                   # Node.js debugger port
Shell — Using explicit override files
# Development (auto-merges compose.override.yaml)
docker compose up -d

# Production (explicit file, no override)
docker compose -f compose.yaml -f compose.prod.yaml up -d

# CI testing
docker compose -f compose.yaml -f compose.test.yaml run --rm tests

Security Advanced

Harden your containers and images against real-world threats.

The Top Security Risks

👤
Running as Root

Container processes run as root by default — a breakout means full host access.

📦
Bloated Images

More packages = more CVEs. Slim images have dramatically smaller attack surfaces.

🔑
Secrets in Images

API keys in ENV vars or baked into layers are readable via docker inspect.

🔓
Privileged Mode

--privileged gives full access to the host. Almost never needed.

Running as Non-Root

Dockerfile — Non-root User
FROM node:20-alpine

WORKDIR /app

COPY --chown=appuser:appgroup package*.json ./
RUN npm ci --only=production

COPY --chown=appuser:appgroup . .

USER appuser

EXPOSE 3000
CMD ["node", "server.js"]

Security Hardening Checklist

Shell — Runtime Hardening Flags
# Drop ALL linux capabilities, add only what's needed
docker run \
  --cap-drop ALL \
  --cap-add NET_BIND_SERVICE \
  --read-only \                        # read-only root filesystem
  --tmpfs /tmp:rw,noexec,nosuid \      # writable /tmp in RAM
  --no-new-privileges \                # prevent privilege escalation
  --security-opt="no-new-privileges:true" \
  --security-opt seccomp=seccomp.json \   # custom seccomp profile
  --memory="256m" \                    # prevent memory exhaustion
  --cpus="1.0" \                       # prevent CPU starvation
  --pids-limit 100 \                   # prevent fork bombs
  --user 1001:1001 \                   # explicit non-root UID
  myapp:latest

Image Size Optimization

Dockerfile — Size Optimization Techniques
# ✅ Use slim or alpine base images
FROM python:3.11-slim           # 130MB vs python:3.11 at 900MB
FROM node:20-alpine            # 180MB vs node:20 at 1.1GB

# ✅ Combine RUN commands (fewer layers)
RUN apt-get update \
    && apt-get install -y --no-install-recommends \
        curl \
        ca-certificates \
    && rm -rf /var/lib/apt/lists/*

# ❌ Don't do this — creates 3 layers, apt cache persists
RUN apt-get update
RUN apt-get install -y curl
RUN rm -rf /var/lib/apt/lists/*

# ✅ Python: install deps only (no dev packages)
RUN pip install --no-cache-dir -r requirements.txt
#   --no-cache-dir: saves ~100MB for large projects

# ✅ npm: production only + clean cache
RUN npm ci --only=production && npm cache clean --force
Shell — Analyze image layers
# Show layer sizes
docker history --no-trunc --format "{{.Size}}\t{{.CreatedBy}}" myapp

# Dive — interactive TUI image explorer (highly recommended)
dive myapp:latest

# Install dive
brew install dive  # macOS

BuildKit & Advanced Caching

Dockerfile — BuildKit Cache Mounts
# syntax=docker/dockerfile:1
FROM python:3.11-slim

WORKDIR /app
COPY requirements.txt .

# Cache pip downloads between builds (BuildKit feature)
# --mount=type=cache persists the pip cache directory across builds
RUN --mount=type=cache,target=/root/.cache/pip \
    pip install -r requirements.txt

COPY . .
CMD ["python", "app.py"]

    NPM_TOKEN=$(cat /run/secrets/npm_token) npm install

Container Resource Management

Shell
# Memory limits
docker run --memory="512m" myapp        # hard limit (OOM killed if exceeded)
docker run --memory-reservation="256m" myapp  # soft limit (hint to scheduler)
docker run --memory="512m" --memory-swap="512m" myapp  # disable swap

# CPU limits
docker run --cpus="1.5" myapp           # use at most 1.5 CPUs
docker run --cpu-shares=512 myapp       # relative weight (default 1024)
docker run --cpuset-cpus="0,2" myapp    # pin to CPU cores 0 and 2

# Real-time monitoring
docker stats                            # live CPU/memory/network/IO
docker stats --format "table {{.Name}}\t{{.CPUPerc}}\t{{.MemUsage}}"

Docker Swarm Overview

Docker Swarm is Docker's built-in clustering and orchestration solution. It turns a pool of Docker hosts into a single, virtual Docker host. While Kubernetes dominates at scale, Swarm is simpler and often sufficient for most teams.

┌──────────────────────────────────────────────────────────────────┐ │ DOCKER SWARM CLUSTER │ │ │ │ ┌─────────────────────┐ ┌──────────────────────────────────┐ │ │ │ MANAGER NODES │ │ WORKER NODES │ │ │ │ │ │ │ │ │ │ ┌───────────────┐ │ │ ┌──────────┐ ┌──────────┐ │ │ │ │ │ Raft Consensus│ │────▶ │ Worker 1 │ │ Worker 2 │ │ │ │ │ │ (Cluster state│ │ │ │ │ │ │ │ │ │ │ │ management) │ │ │ │ Tasks │ │ Tasks │ │ │ │ │ └───────────────┘ │ │ └──────────┘ └──────────┘ │ │ │ │ │ │ │ │ │ │ Scheduling, Health │ │ 3+ managers for HA (odd number) │ │ │ └─────────────────────┘ └──────────────────────────────────┘ │ └──────────────────────────────────────────────────────────────────┘

Setting Up a Swarm

Shell
# Initialize swarm on the manager node
docker swarm init --advertise-addr 192.168.1.10

# Output: join token for workers
# docker swarm join --token SWMTKN-1-xxx... 192.168.1.10:2377

# On each worker node:
docker swarm join --token SWMTKN-1-xxx... 192.168.1.10:2377

# List nodes in the swarm
docker node ls

# Promote a worker to manager
docker node promote worker-2

# Get join tokens again
docker swarm join-token worker
docker swarm join-token manager

Services in Docker Swarm

In Swarm, you deploy services (not individual containers). A service defines the desired state — what image to run, how many replicas, which ports to expose — and Swarm continuously ensures that state is maintained.

Shell
# Create a service with 3 replicas
docker service create \
  --name webapp \
  --replicas 3 \
  --publish published=80,target=3000 \
  --update-delay 10s \
  --update-parallelism 1 \
  myapp:v1.0

# List services
docker service ls

# List tasks (which node runs each container)
docker service ps webapp

# Scale up/down
docker service scale webapp=5
docker service scale webapp=2

# Rolling update (zero-downtime deployment)
docker service update \
  --image myapp:v2.0 \
  --update-delay 10s \
  --update-parallelism 1 \
  --update-failure-action rollback \
  webapp

# Rollback to previous version
docker service rollback webapp

# Remove a service
docker service rm webapp

Deploying Stacks

A Stack is Compose for Swarm — deploy your entire compose.yaml as a Swarm stack:

compose.yaml — Swarm-compatible Stack
services:
  api:
    image: myregistry/myapp:v2.0
    deploy:
      replicas: 3
      update_config:
        parallelism: 1
        delay: 10s
        failure_action: rollback
      restart_policy:
        condition: on-failure
        delay: 5s
        max_attempts: 3
      resources:
        limits:
          memory: 512M
          cpus: '0.5'
        reservations:
          memory: 128M
      placement:
        constraints:
          - "node.role == worker"
    secrets:
      - db_password

secrets:
  db_password:
    external: true

CI/CD Pipeline with Docker

GitHub Actions — Build, Test, Push, Deploy
name: CI/CD

on:
  push:
    branches: [main]

env:
  REGISTRY: ghcr.io
  IMAGE_NAME: ${{ github.repository }}

jobs:
  build-test-push:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      - name: Log in to registry
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Build and push
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: |
            ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:latest
            ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }}
          cache-from: type=gha
          cache-to: type=gha,mode=max
          platforms: linux/amd64,linux/arm64

      - name: Security scan
        uses: aquasecurity/trivy-action@master
        with:
          image-ref: '${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }}'
          severity: 'CRITICAL'
          exit-code: '1'

      - name: Deploy to Swarm
        run: |
          docker -H ssh://deploy@prod.server.com \
            service update \
            --image ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }} \
            mystack_api
