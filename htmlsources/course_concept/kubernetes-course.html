Module 01

What is Kubernetes?

Kubernetes (often abbreviated as K8s) is an open-source container orchestration platform originally developed by Google, inspired by their internal system called Borg. It was donated to the Cloud Native Computing Foundation (CNCF) in 2014 and has since become the de-facto standard for deploying, scaling, and managing containerized applications.

ℹ️
Why "K8s"?

The name "K8s" is a numeronym — the 8 replaces the eight letters between "K" and "s" in Kubernetes. Similarly, i18n = internationalization, a11y = accessibility.

The Problem Kubernetes Solves

Before container orchestration, deploying applications at scale required enormous manual effort. Consider what happens when you have hundreds of microservices that need to:

  • Be distributed intelligently across dozens of servers
  • Restart automatically if they crash
  • Scale up when traffic increases, scale down to save costs
  • Get new versions deployed with zero downtime
  • Communicate securely with each other
  • Have secrets and configuration injected at runtime

Kubernetes automates all of this. It acts as a "data center operating system" — you describe the desired state of your system in declarative configuration files, and Kubernetes continuously works to make the actual state match that desired state.

Kubernetes vs. Docker

A common point of confusion for newcomers: Docker and Kubernetes are complementary, not competing technologies.

DockerKubernetes
Builds and runs containers on a single machineOrchestrates containers across many machines
Great for development and simple deploymentsDesigned for production at scale
Manual scaling and restartAutomatic self-healing and scaling
Docker Compose for multi-container appsRich abstractions for complex workloads
💡
Key Mental Model

Think of Kubernetes like an airline operations system. You (the operator) declare "I need 10 planes flying route X." The system handles which specific planes, reassigns if one breaks down, and reroutes to meet demand — you don't manage individual planes manually.

Core Kubernetes Principles

📋
Declarative Configuration

You describe the desired end state in YAML/JSON. Kubernetes figures out how to get there and maintain it.

🔄
Control Loops

Controllers constantly watch the current state and take actions to reconcile it with the desired state.

🏥
Self-Healing

Failed containers are restarted. Pods are rescheduled if nodes fail. Unhealthy instances are replaced.

📈
Horizontal Scaling

Scale workloads in or out with a single command or automatically based on CPU/memory metrics.

Control Plane Components

1. kube-apiserver

The API server is the central hub of Kubernetes. Every interaction — whether from kubectl, the scheduler, controllers, or kubelet — goes through the API server. It validates requests, persists state to etcd, and serves as the single source of truth for the cluster state.

🔑
REST API at the Core

The API server exposes a RESTful HTTP API. When you run kubectl get pods, kubectl is simply making a GET request to /api/v1/namespaces/default/pods. You can make the same calls with curl.

2. etcd

etcd is a distributed, consistent key-value store that acts as Kubernetes' "database." It stores all cluster state: the spec of every resource, current status, secrets, and configuration. If etcd goes down, the cluster becomes read-only. This is why HA clusters run etcd in a 3 or 5 node quorum.

3. kube-scheduler

When a new Pod is created without an assigned node, the scheduler watches for these "unbound" pods and selects the best node for them based on resource requirements, affinity rules, taints/tolerations, and hardware constraints.

4. kube-controller-manager

This is a collection of individual controllers running in a single process. Each controller implements a control loop watching specific resource types. Key controllers include:

  • Node Controller — Responds when nodes go down
  • ReplicaSet Controller — Ensures the correct number of pod replicas
  • Deployment Controller — Manages rolling updates
  • Service Account Controller — Creates default service accounts

Worker Node Components

kubelet

The kubelet is the primary "node agent" running on every worker node. It takes a set of PodSpecs and ensures the described containers are running and healthy. It communicates with the container runtime (Docker, containerd, CRI-O) to start/stop containers.

kube-proxy

kube-proxy runs on every node and maintains network rules for Services. When traffic arrives for a Service's ClusterIP, kube-proxy handles forwarding it to the correct backend Pod. It uses either iptables or IPVS for this.

Container Runtime

The software responsible for running containers. Kubernetes uses the Container Runtime Interface (CRI), supporting containerd, CRI-O, and historically Docker (via dockershim, deprecated in K8s 1.24).

Module 03

Installation & Setup

There are several ways to get a Kubernetes cluster running. We'll cover the most common options, from local development to managed cloud clusters.

Option 1: Minikube (Local Development)

Minikube is the easiest way to run Kubernetes locally. It creates a single-node cluster inside a VM or Docker container.

bash — macOS
# Install minikube
brew install minikube

# Start a cluster with 4 CPUs and 4GB RAM
minikube start --cpus=4 --memory=4096

# Verify the cluster is running
kubectl cluster-info

# Enable the ingress addon
minikube addons enable ingress

# Open the Kubernetes dashboard
minikube dashboard

Option 2: kind (Kubernetes in Docker)

kind runs K8s cluster nodes as Docker containers. It's faster than Minikube and ideal for CI pipelines and multi-node testing.

bash
# Install kind
brew install kind

# Create a multi-node cluster with a config file
cat > kind-config.yaml <<EOF
kind: Cluster
apiVersion: kind.x-k8s.io/v1alpha4
nodes:
  - role: control-plane
  - role: worker
  - role: worker
EOF

kind create cluster --config kind-config.yaml --name my-cluster

# Check nodes
kubectl get nodes

Option 3: Managed Kubernetes (Cloud)

For production, managed Kubernetes services handle the control plane for you:

☁️
Amazon EKS

AWS managed Kubernetes. Integrates with IAM, VPC, ALB, and other AWS services natively.

🔵
Google GKE

The most mature managed K8s. Autopilot mode manages nodes entirely for you.

🔷
Azure AKS

Deep integration with Azure AD, Monitor, and Container Registry.

🌊
DigitalOcean DOKS

Simple, affordable managed Kubernetes ideal for smaller teams.

Mastering kubectl

kubectl is the command-line tool for interacting with Kubernetes clusters. Learning it deeply will save you hours of debugging.

bash — kubectl essentials
# --- Cluster Info ---
kubectl cluster-info
kubectl get nodes -o wide
kubectl config get-contexts    # list clusters you can connect to
kubectl config use-context my-cluster

# --- Viewing Resources ---
kubectl get pods
kubectl get pods -n kube-system   # in a specific namespace
kubectl get all                    # pods, services, deployments
kubectl describe pod my-pod        # detailed info + events
kubectl get pod my-pod -o yaml    # raw YAML output

# --- Creating Resources ---
kubectl apply -f manifest.yaml     # declarative (preferred)
kubectl create deployment nginx --image=nginx  # imperative

# --- Debugging ---
kubectl logs my-pod                # container logs
kubectl logs -f my-pod            # follow (stream) logs
kubectl exec -it my-pod -- /bin/sh # shell into a pod
kubectl port-forward my-pod 8080:80 # forward local port to pod

# --- Managing Resources ---
kubectl scale deployment nginx --replicas=5
kubectl delete pod my-pod
kubectl delete -f manifest.yaml
Module 04

Pods

A Pod is the smallest deployable unit in Kubernetes. It represents one or more containers that share the same network namespace (IP address and ports) and storage volumes, and are always co-located on the same node.

⚠️
Pods are Ephemeral

Pods are designed to be disposable. They are created, killed, and replaced regularly. Never depend on a Pod's IP address or assume it will persist. Use higher-level resources like Deployments.

Your First Pod

yaml — simple-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-nginx
  labels:
    app: nginx
    environment: dev
spec:
  containers:
  - name: nginx
    image: nginx:1.25
    ports:
    - containerPort: 80
    resources:
      requests:
        cpu: "100m"
        memory: "128Mi"
      limits:
        cpu: "500m"
        memory: "256Mi"
bash
kubectl apply -f simple-pod.yaml
kubectl get pod my-nginx
kubectl describe pod my-nginx
kubectl logs my-nginx

Multi-Container Pods

Multiple containers in one Pod share the same network and can communicate via localhost. They also share any mounted volumes. The most common patterns are:

🔌
Sidecar Pattern

A helper container augments the main app. Example: a log-shipper sidecar that reads logs written by the main app and forwards them to Elasticsearch.

🚪
Ambassador Pattern

A proxy container that handles communication to the outside world for the main container. Example: a local proxy that routes database connections.

yaml — sidecar-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: app-with-sidecar
spec:
  volumes:
  - name: shared-logs
    emptyDir: {}
  containers:
  - name: app                    # main application
    image: my-app:latest
    volumeMounts:
    - name: shared-logs
      mountPath: /var/log/app
  - name: log-shipper             # sidecar container
    image: fluent/fluent-bit:latest
    volumeMounts:
    - name: shared-logs
      mountPath: /var/log/app
      readOnly: true

Init Containers

Init containers run to completion before app containers start. They're perfect for setup tasks like waiting for dependencies, running database migrations, or downloading assets.

yaml — init-container.yaml
spec:
  initContainers:
  - name: wait-for-db
    image: busybox
    command: ['sh', '-c']
    args: ['until nc -z postgres-service 5432; do echo waiting; sleep 2; done']
  containers:
  - name: app
    image: my-app:latest
Module 05

ReplicaSets & Deployments

In practice, you almost never create Pods directly. Instead, you use higher-level abstractions that manage Pods for you.

ReplicaSet

A ReplicaSet ensures that a specified number of pod replicas are running at all times. If a Pod fails, the ReplicaSet creates a new one. If there are too many, it deletes the excess.

📝
ReplicaSet vs Deployment

You rarely use ReplicaSets directly. Deployments create and manage ReplicaSets for you, adding rolling updates and rollback capabilities on top.

Deployments

A Deployment is the standard way to run stateless applications in Kubernetes. It provides:

  • Desired number of replicas always running
  • Rolling updates with zero downtime
  • Easy rollback to previous versions
  • Pause and resume update capabilities
yaml — deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: webapp
  labels:
    app: webapp
spec:
  replicas: 3
  selector:
    matchLabels:
      app: webapp           # must match template labels
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1          # max extra pods during update
      maxUnavailable: 0   # zero downtime
  template:
    metadata:
      labels:
        app: webapp
    spec:
      containers:
      - name: webapp
        image: my-webapp:1.0.0
        ports:
        - containerPort: 8080
        resources:
          requests:
            cpu: "250m"
            memory: "256Mi"
          limits:
            cpu: "1"
            memory: "512Mi"

Rolling Updates & Rollbacks

bash — update workflow
# Update the image (triggers a rolling update)
kubectl set image deployment/webapp webapp=my-webapp:2.0.0

# Watch the rollout in real time
kubectl rollout status deployment/webapp

# View rollout history
kubectl rollout history deployment/webapp

# Rollback to previous version
kubectl rollout undo deployment/webapp

# Rollback to a specific revision
kubectl rollout undo deployment/webapp --to-revision=2

# Pause a rollout (to test canary)
kubectl rollout pause deployment/webapp

# Resume the rollout
kubectl rollout resume deployment/webapp
Module 06

Services & Networking

Since Pods are ephemeral and their IP addresses change, Services provide a stable endpoint to access a set of Pods. A Service uses a label selector to identify which Pods it targets and provides load balancing across them.

Service Types

TypeDescriptionUse Case
ClusterIPStable IP within the cluster. Default type.Internal communication between services
NodePortExposes service on each node's IP at a static port (30000–32767)Development access, on-prem clusters
LoadBalancerProvisions a cloud load balancerProduction HTTP/HTTPS traffic on cloud
ExternalNameMaps service to a DNS nameAccessing external services by DNS alias
yaml — service examples
# ClusterIP (default) — internal only
apiVersion: v1
kind: Service
metadata:
  name: webapp-svc
spec:
  selector:
    app: webapp    # targets pods with this label
  ports:
  - port: 80        # port the service exposes
    targetPort: 8080 # port on the pod
  type: ClusterIP
---
# LoadBalancer — gets a cloud LB with external IP
apiVersion: v1
kind: Service
metadata:
  name: webapp-lb
spec:
  selector:
    app: webapp
  ports:
  - port: 80
    targetPort: 8080
  type: LoadBalancer

DNS Within the Cluster

Kubernetes includes a built-in DNS server (CoreDNS). Every Service gets a DNS entry following the pattern:

<service-name>.<namespace>.svc.cluster.local

Within the same namespace, you can simply use webapp-svc as the hostname. Across namespaces, use webapp-svc.production.svc.cluster.local.

Network Policies

By default, all pods can talk to all other pods. Network Policies act as firewalls at the pod level.

yaml — deny all ingress, allow from specific namespace
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-from-monitoring
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: webapp        # applies to webapp pods
  policyTypes:
  - Ingress
  ingress:
  - from:
    - namespaceSelector:
        matchLabels:
          name: monitoring  # only from monitoring ns
    ports:
    - port: 8080
Module 07

ConfigMaps & Secrets

Hardcoding configuration and secrets in container images is an antipattern. Kubernetes provides two dedicated objects to decouple configuration from code.

ConfigMaps

ConfigMaps store non-sensitive configuration data as key-value pairs. They can be consumed as environment variables, command-line arguments, or mounted as files.

yaml — configmap.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  DATABASE_HOST: "postgres.production.svc.cluster.local"
  DATABASE_PORT: "5432"
  LOG_LEVEL: "info"
  app.properties: |          # multi-line file content
    max.connections=100
    timeout=30s
    feature.dark-mode=true
yaml — using a configmap in a pod
spec:
  containers:
  - name: app
    image: my-app:1.0
    envFrom:
    - configMapRef:
        name: app-config     # inject all keys as env vars
    volumeMounts:
    - name: config-volume
      mountPath: /etc/config  # mount as files
  volumes:
  - name: config-volume
    configMap:
      name: app-config

Secrets

Secrets store sensitive data like passwords, tokens, and TLS certificates. They are base64-encoded (not encrypted by default!) and stored in etcd. For production, enable etcd encryption at rest and consider external secret managers like Vault or AWS Secrets Manager.

🔒
Security Warning

Base64 is encoding, NOT encryption. Anyone with access to etcd or the namespace can decode Secrets trivially. Always enable encryption at rest in production and use RBAC to restrict secret access.

bash — creating secrets
# Create from literal values (values auto-encoded)
kubectl create secret generic db-credentials \
  --from-literal=username=admin \
  --from-literal=password=SuperSecret123

# Create a TLS secret from cert files
kubectl create secret tls my-tls-secret \
  --cert=tls.crt \
  --key=tls.key

# View a secret (base64 decoded)
kubectl get secret db-credentials -o jsonpath='{.data.password}' | base64 -d
yaml — secret manifest
apiVersion: v1
kind: Secret
metadata:
  name: db-credentials
type: Opaque
data:
  username: YWRtaW4=         # base64("admin")
  password: U3VwZXJTZWNyZXQxMjM= # base64("SuperSecret123")
---
# Using in a pod
spec:
  containers:
  - name: app
    env:
    - name: DB_USERNAME
      valueFrom:
        secretKeyRef:
          name: db-credentials
          key: username
    - name: DB_PASSWORD
      valueFrom:
        secretKeyRef:
          name: db-credentials
          key: password
Module 08

Volumes & Persistent Storage

Container filesystems are ephemeral — when a container restarts, any data written to it is lost. Kubernetes volumes solve this with a rich set of storage abstractions.

Volume Types

Volume TypePersistenceUse Case
emptyDirPod lifetimeScratch space, inter-container sharing
hostPathNode lifetimeAccess node filesystem (dev/logging)
PersistentVolumeClaimIndependentDatabases, stateful applications
configMapConfigMap objectConfiguration files
secretSecret objectSensitive files (certs, keys)
nfsExternal NFSShared storage across pods

PersistentVolumes & PersistentVolumeClaims

The PV/PVC abstraction decouples storage provisioning from storage consumption. Admins provision PersistentVolumes; developers claim storage via PersistentVolumeClaims without needing to know the underlying storage details.

yaml — PV and PVC
# Admin creates a PersistentVolume
apiVersion: v1
kind: PersistentVolume
metadata:
  name: postgres-pv
spec:
  capacity:
    storage: 10Gi
  accessModes:
  - ReadWriteOnce        # RWO: single node read/write
  persistentVolumeReclaimPolicy: Retain
  storageClassName: standard
  hostPath:              # for local dev; use cloud volumes in prod
    path: /data/postgres
---
# Developer creates a PersistentVolumeClaim
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: postgres-pvc
spec:
  accessModes:
  - ReadWriteOnce
  resources:
    requests:
      storage: 5Gi       # request 5Gi from available PVs
  storageClassName: standard
---
# Pod uses the PVC
volumes:
- name: postgres-data
  persistentVolumeClaim:
    claimName: postgres-pvc

StorageClasses & Dynamic Provisioning

With StorageClasses, PVs are created automatically when a PVC is submitted. This is how production cloud clusters work — you don't pre-provision PVs.

yaml — StorageClass (AWS EBS)
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: fast-ssd
provisioner: ebs.csi.aws.com
parameters:
  type: gp3
  iops: "3000"
  throughput: "125"
reclaimPolicy: Delete      # delete EBS when PVC deleted
allowVolumeExpansion: true  # allow resizing
volumeBindingMode: WaitForFirstConsumer
Module 09

Namespaces & RBAC

Namespaces

Namespaces provide a mechanism to divide cluster resources among multiple users or teams. They create isolated virtual clusters within one physical cluster. Resources within a namespace share a name scope (two deployments with the same name can exist in different namespaces).

bash — namespace management
# Default namespaces in a fresh cluster
kubectl get namespaces
# default        — your resources if no namespace specified
# kube-system    — K8s system components
# kube-public    — publicly readable
# kube-node-lease — node heartbeats

# Create namespaces for environments
kubectl create namespace development
kubectl create namespace staging
kubectl create namespace production

# Deploy to a specific namespace
kubectl apply -f deployment.yaml -n production

# Set a default namespace for your context
kubectl config set-context --current --namespace=production

Role-Based Access Control (RBAC)

RBAC regulates access to Kubernetes resources based on the roles of users. It follows four key objects:

👤
Role / ClusterRole

Defines a set of permissions. Role is namespace-scoped. ClusterRole applies cluster-wide.

🔗
RoleBinding / ClusterRoleBinding

Grants a Role to a user, group, or ServiceAccount within a namespace or cluster-wide.

yaml — RBAC: read-only developer access
# Create a Role that allows reading pods and deployments
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: developer-viewer
  namespace: production
rules:
- apiGroups: [""]          # "" = core API group
  resources: ["pods", "pods/log"]
  verbs: ["get", "list", "watch"]
- apiGroups: ["apps"]
  resources: ["deployments"]
  verbs: ["get", "list"]
---
# Bind the role to a user
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: jane-developer
  namespace: production
subjects:
- kind: User
  name: jane@company.com
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: Role
  name: developer-viewer
  apiGroup: rbac.authorization.k8s.io

Service Accounts

ServiceAccounts provide an identity for processes running in Pods. If your application needs to call the Kubernetes API (e.g., a CI system), it should use a dedicated ServiceAccount with minimal permissions.

yaml — ServiceAccount with permissions
apiVersion: v1
kind: ServiceAccount
metadata:
  name: app-service-account
  namespace: production
---
# Use in a pod spec
spec:
  serviceAccountName: app-service-account
  containers: [...]
Module 10

StatefulSets

Deployments are perfect for stateless applications. But databases, message queues, and distributed systems require stable, persistent identity. That's what StatefulSets provide.

StatefulSet vs Deployment

FeatureDeploymentStatefulSet
Pod identityRandom (pod-xyz-abc)Ordered, stable (pod-0, pod-1)
StorageShared or recreatedIndividual PVC per pod
Scaling orderParallelSequential (0→1→2)
DNS nameVia service selectorUnique per-pod DNS names
Use casesWeb servers, APIsDatabases, Kafka, ZooKeeper
yaml — PostgreSQL StatefulSet
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: postgres
spec:
  serviceName: "postgres-headless"  # headless service required
  replicas: 3
  selector:
    matchLabels:
      app: postgres
  template:
    metadata:
      labels:
        app: postgres
    spec:
      containers:
      - name: postgres
        image: postgres:16
        env:
        - name: POSTGRES_PASSWORD
          valueFrom:
            secretKeyRef:
              name: postgres-secret
              key: password
        volumeMounts:
        - name: data
          mountPath: /var/lib/postgresql/data
  volumeClaimTemplates:      # creates a unique PVC per pod!
  - metadata:
      name: data
    spec:
      accessModes: ["ReadWriteOnce"]
      resources:
        requests:
          storage: 20Gi
      storageClassName: fast-ssd
💡
Per-Pod DNS

Each pod in a StatefulSet gets a DNS entry: postgres-0.postgres-headless.default.svc.cluster.local. This lets pods address each other directly, which is essential for primary/replica database setups.

Module 11

DaemonSets & Jobs

DaemonSets

A DaemonSet ensures that a copy of a Pod runs on every (or some) node in the cluster. As new nodes are added, pods are automatically added to them. As nodes are removed, pods are garbage collected. Classic use cases:

  • Log collectors (Fluentd, Filebeat)
  • Metrics exporters (node_exporter)
  • Storage daemons (Ceph, GlusterFS)
  • Network plugins (Calico, Cilium, Weave)
yaml — log collector DaemonSet
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: fluentd
spec:
  selector:
    matchLabels:
      name: fluentd
  template:
    metadata:
      labels:
        name: fluentd
    spec:
      tolerations:             # run on control plane nodes too
      - key: node-role.kubernetes.io/control-plane
        effect: NoSchedule
      containers:
      - name: fluentd
        image: fluent/fluentd:v1.16
        volumeMounts:
        - name: varlog
          mountPath: /var/log
      volumes:
      - name: varlog
        hostPath:
          path: /var/log

Jobs

A Job creates one or more Pods and ensures a specified number of them successfully terminate. Perfect for batch processing, database migrations, or one-off tasks.

yaml — batch job
apiVersion: batch/v1
kind: Job
metadata:
  name: db-migration
spec:
  completions: 1         # number of successful completions needed
  parallelism: 1         # pods running in parallel
  backoffLimit: 3       # retry up to 3 times on failure
  template:
    spec:
      restartPolicy: OnFailure
      containers:
      - name: migration
        image: my-app:latest
        command: ["python", "manage.py", "migrate"]

CronJobs

yaml — scheduled CronJob
apiVersion: batch/v1
kind: CronJob
metadata:
  name: db-backup
spec:
  schedule: "0 2 * * *"      # every day at 2am UTC
  concurrencyPolicy: Forbid   # don't run if previous still running

Ingress Controllers

An Ingress exposes HTTP/HTTPS routes from outside the cluster to services within the cluster. Unlike NodePort or LoadBalancer services, a single Ingress can route traffic to multiple services based on the request's hostname or path — saving you from needing a cloud load balancer per service.

ℹ️
Ingress vs Ingress Controller

An Ingress resource is the K8s API object (YAML). An Ingress Controller is the actual software (nginx, Traefik, HAProxy, AWS ALB) that reads Ingress resources and configures the reverse proxy accordingly. You need both.

bash — install nginx ingress controller
helm upgrade --install ingress-nginx ingress-nginx \
  --repo https://kubernetes.github.io/ingress-nginx \
  --namespace ingress-nginx --create-namespace
yaml — ingress with path & host routing
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: app-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
    cert-manager.io/cluster-issuer: "letsencrypt-prod"
spec:
  ingressClassName: nginx
  tls:
  - hosts:
    - myapp.example.com
    secretName: myapp-tls
  rules:
  - host: myapp.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: frontend-svc
            port:
              number: 80
      - path: /api
        pathType: Prefix
        backend:
          service:
            name: api-svc
            port:
              number: 8080

Helm Package Manager

Helm is the package manager for Kubernetes — often called the "apt-get of Kubernetes." It bundles related Kubernetes manifests into Charts that can be versioned, shared, and deployed with a single command. It also provides powerful templating to parameterize deployments.

Core Concepts

📦
Chart

A package of pre-configured Kubernetes resources. Like a Dockerfile but for entire applications.

🚀
Release

An instance of a Chart deployed into a cluster. You can install the same chart multiple times creating multiple releases.

🗃️
Repository

A place where Charts are stored. ArtifactHub.io is the central repository of public Helm Charts.

⚙️
Values

YAML configuration overrides that customize a Chart during installation or upgrade.

bash — helm essentials

# Search for charts
helm search repo bitnami/postgresql

# Install a chart (creates a release named "my-db")
helm install my-db bitnami/postgresql \
  --set auth.postgresPassword=secret \
  --namespace production

# Install with a custom values file
helm install my-db bitnami/postgresql -f my-values.yaml

# List releases
helm list -n production

# Upgrade a release
helm upgrade my-db bitnami/postgresql --set auth.postgresPassword=newsecret

# Rollback to previous release version
helm rollback my-db 1

# Uninstall
helm uninstall my-db

# Creates this structure:

Module 14

Auto-scaling

Kubernetes provides three complementary auto-scaling mechanisms that work at different levels of the stack.

Horizontal Pod Autoscaler (HPA)

HPA automatically scales the number of Pod replicas based on observed CPU utilization, memory usage, or custom metrics. It polls the metrics API every 15 seconds by default.

yaml — HPA with CPU and memory targets
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: webapp-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: webapp
  minReplicas: 2
  maxReplicas: 20
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 60    # scale when avg CPU > 60%
  - type: Resource
    resource:
      name: memory
      target:
        type: AverageValue
        averageValue: 400Mi

Vertical Pod Autoscaler (VPA)

VPA automatically adjusts the CPU and memory requests for containers, right-sizing pods based on actual usage. This is particularly useful for workloads with unpredictable resource needs. VPA requires installing the VPA admission controller.

yaml — VPA in recommendation mode
apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: webapp-vpa
spec:
  targetRef:
    apiVersion: "apps/v1"
    kind: Deployment
    name: webapp
  updatePolicy:
    updateMode: "Off"   # "Off" = recommendations only (no auto-apply)
                          # "Auto" = auto-apply (restarts pods)

Cluster Autoscaler

The Cluster Autoscaler automatically adjusts the number of nodes in a cluster. When pods can't be scheduled due to insufficient resources, it provisions new nodes. When nodes are underutilized, it drains and removes them to save costs.

💡
Scaling Strategy

Use HPA + Cluster Autoscaler together: HPA scales pods, and when HPA exhausts available nodes, Cluster Autoscaler adds more nodes. Set proper resource requests so the scheduler can make informed decisions.

Module 15

Resource Management

Requests & Limits

Requests tell the scheduler how much CPU/memory to reserve for a pod. Limits cap the maximum a container can use. If a container exceeds its memory limit, it's OOMKilled. If it exceeds CPU limits, it's throttled (not killed).

⚠️
Always Set Requests and Limits

Pods without resource requests get scheduled to nodes that may be overloaded, causing evictions. Pods without limits can starve neighboring pods. Both lead to unpredictable production outages.

LimitRanges

LimitRanges enforce default requests/limits and constrain resource allocation per pod or container in a namespace — protecting you from teams forgetting to set them.

yaml — LimitRange
apiVersion: v1
kind: LimitRange
metadata:
  name: default-limits
  namespace: development
spec:
  limits:
  - type: Container
    default:                # applied when no limit specified
      cpu: 500m
      memory: 512Mi
    defaultRequest:       # applied when no request specified
      cpu: 100m
      memory: 128Mi
    max:                   # hard maximum per container
      cpu: 2
      memory: 2Gi

ResourceQuotas

ResourceQuotas limit the total resource consumption in a namespace. Useful for multi-tenant clusters where teams share a cluster.

yaml — ResourceQuota for a team namespace
apiVersion: v1
kind: ResourceQuota
metadata:
  name: team-quota
  namespace: team-alpha
spec:
  hard:
    requests.cpu: "20"
    requests.memory: 40Gi
    limits.cpu: "40"
    limits.memory: 80Gi
    pods: "50"
    services: "20"
    persistentvolumeclaims: "10"

Quality of Service (QoS) Classes

Kubernetes assigns a QoS class to each Pod, which determines eviction priority when a node is under memory pressure:

QoS ClassConditionEviction Priority
Guaranteedrequests == limits for all containersLast to be evicted
Burstablerequests < limitsMiddle priority
BestEffortNo requests or limits setFirst to be evicted
Module 16

Health Probes

Health probes tell Kubernetes when a container is ready to serve traffic and when it needs to be restarted. Properly configured probes are essential for zero-downtime deployments.

The Three Probe Types

💓
Liveness Probe

Detects if the container is in a broken state. If it fails, Kubernetes kills and restarts the container. Use for detecting deadlocks.

✅
Readiness Probe

Detects when a container is ready to receive traffic. If it fails, the pod is removed from service endpoints — no traffic is sent to it.

🚀
Startup Probe

Protects slow-starting containers. Disables liveness and readiness checks until the startup probe succeeds. Prevents premature killing.

yaml — all three probes
containers:
- name: app
  image: my-app:1.0

  startupProbe:               # waits for app to boot
    httpGet:
      path: /health/startup
      port: 8080
    failureThreshold: 30    # 30 × 10s = 5min to start
    periodSeconds: 10

  livenessProbe:
    httpGet:
      path: /health/live
      port: 8080
    initialDelaySeconds: 0   # 0 since startup probe handles it
    periodSeconds: 15
    failureThreshold: 3
    timeoutSeconds: 5

  readinessProbe:
    httpGet:
      path: /health/ready
      port: 8080
    periodSeconds: 5
    failureThreshold: 3

Probe Mechanisms

yaml — probe types: HTTP, TCP, exec, gRPC
# HTTP GET probe (most common)
livenessProbe:
  httpGet:
    path: /healthz
    port: 8080
    httpHeaders:
    - name: X-Custom-Header
      value: health-check

# TCP socket probe (for non-HTTP services like DB)
livenessProbe:
  tcpSocket:
    port: 5432

# Exec probe (runs a command)
livenessProbe:
  exec:
    command: ["redis-cli", "ping"]

# gRPC probe (Kubernetes 1.24+)
livenessProbe:
  grpc:
    port: 50051

  name: webapp-monitor
  namespace: monitoring

  selector:
    matchLabels:

CI/CD Pipelines

A robust CI/CD pipeline for Kubernetes typically involves: building a Docker image, pushing to a registry, updating the Kubernetes manifests, and letting GitOps tooling deploy the changes.

GitHub Actions Pipeline

yaml — .github/workflows/deploy.yml
name: Build and Deploy
on:
  push:
    branches: [main]

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4

    - name: Build Docker image
      run: |
        IMAGE="ghcr.io/${{ github.repository }}:${{ github.sha }}"
        docker build -t $IMAGE .
        echo ${{ secrets.GITHUB_TOKEN }} | docker login ghcr.io -u $ --password-stdin
        docker push $IMAGE

    - name: Update image in deployment
      run: |
        kubectl set image deployment/webapp \
          webapp=ghcr.io/${{ github.repository }}:${{ github.sha }}
      env:
        KUBECONFIG: ${{ secrets.KUBECONFIG }}

    - name: Verify rollout
      run: kubectl rollout status deployment/webapp --timeout=300s

GitOps with ArgoCD

GitOps is a pattern where the Git repository is the single source of truth for infrastructure. ArgoCD continuously watches your Git repo and syncs the cluster state to match what's in the repo.

bash — install ArgoCD
kubectl create namespace argocd
kubectl apply -n argocd -f \

kind: Application
metadata:
  name: webapp
  namespace: argocd

  project: default
  source:

    path: k8s/production

    server: "https://kubernetes.default.svc"

    automated:

Module 19

Security Hardening

Pod Security Standards

Kubernetes 1.25+ replaced PodSecurityPolicy with built-in Pod Security Standards (PSS) enforced via namespace labels. Three profiles exist: Privileged, Baseline, and Restricted.

bash — enforce restricted security for a namespace
kubectl label namespace production \
  pod-security.kubernetes.io/enforce=restricted \
  pod-security.kubernetes.io/enforce-version=latest

SecurityContext

Set security constraints at the Pod or container level to reduce the attack surface:

yaml — hardened security context
spec:
  securityContext:               # pod-level
    runAsNonRoot: true           # never run as root
    runAsUser: 1000
    fsGroup: 2000
    seccompProfile:
      type: RuntimeDefault       # default seccomp profile
  containers:
  - name: app
    securityContext:             # container-level
      allowPrivilegeEscalation: false
      readOnlyRootFilesystem: true  # immutable container FS
      capabilities:
        drop: ["ALL"]          # drop all Linux capabilities
        add: ["NET_BIND_SERVICE"] # only if port < 1024

Image Security Best Practices

  • Use specific image tags (never :latest in production)
  • Use minimal base images (distroless, Alpine)
  • Scan images with tools like Trivy, Snyk, or Grype
  • Use an admission controller (OPA Gatekeeper, Kyverno) to enforce policies
  • Enable image pull secrets and use private registries
bash — scan an image with Trivy
trivy image my-webapp:1.0.0
trivy k8s --report summary cluster

Admission Controllers: OPA Gatekeeper

OPA Gatekeeper lets you write custom policy rules (in Rego) enforced as admission webhooks. Common policies: require labels on all resources, prohibit privileged containers, enforce resource limits.

yaml — Gatekeeper constraint: require resource limits
apiVersion: constraints.gatekeeper.sh/v1beta1
kind: K8sRequiredLimits
metadata:
  name: require-resource-limits
spec:
  match:
    kinds:
    - apiGroups: ["*"]
      kinds: ["Pod"]
    namespaces: ["production"]
  parameters:
    cpu: "2"
    memory: "2Gi"

Multi-cluster & Service Mesh

Why a Service Mesh?

As microservices grow, cross-cutting concerns like mutual TLS, traffic management, retries, circuit breaking, and distributed tracing become complex to implement in each service. A service mesh handles these at the infrastructure layer via sidecar proxies — no code changes required.

Istio Architecture

Istio injects an Envoy sidecar proxy into every pod. All traffic flows through these proxies, giving you control plane visibility over every service-to-service call.

Service A Pod                      Service B Pod
┌──────────────────┐               ┌──────────────────┐
│  ┌────────────┐  │               │  ┌────────────┐  │
│  │ app-a      │  │               │  │ app-b      │  │
│  └──────┬─────┘  │               │  └──────┬─────┘  │
│         │        │               │         │        │
│  ┌──────┴─────┐  │  mTLS + RBAC  │  ┌──────┴─────┐  │
│  │envoy-proxy │──┼───────────────┼──│envoy-proxy │  │
│  └────────────┘  │               │  └────────────┘  │
└──────────────────┘               └──────────────────┘
         │                                  │
         └──────── Istiod (Control Plane) ──┘
                  (Pilot, Citadel, Galley)
yaml — Istio VirtualService: traffic splitting
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: webapp-vs
spec:
  hosts: ["webapp"]
  http:
  - route:
    - destination:
        host: webapp
        subset: v1
      weight: 90             # 90% to stable version
    - destination:
        host: webapp
        subset: v2
      weight: 10             # 10% canary traffic

Multi-cluster Considerations

  • Cluster Federation: Kubefed or ArgoCD's ApplicationSets for deploying across multiple clusters
  • Cross-cluster networking: Cilium Cluster Mesh, Submariner, or Istio multi-cluster
  • Global load balancing: Use cloud DNS + health checks to route to healthy clusters
  • Shared registries: Use a central container registry accessible from all clusters

Troubleshooting Guide

Debugging Kubernetes issues is a learnable skill. Following a systematic approach saves you hours. This is the module you'll return to most in production.

Pod Debugging Workflow

bash — systematic debugging
# Step 1: Check pod status and events
kubectl get pods -o wide
kubectl describe pod <pod-name>    # look at "Events" section!

# Step 2: Read the logs
kubectl logs <pod-name> --previous  # if pod keeps crashing

# Step 3: Shell into a running pod
kubectl exec -it <pod-name> -- /bin/sh

# Step 4: Deploy a debug container on a node
kubectl debug node/<node-name> -it --image=ubuntu

# Step 5: Check resource usage
kubectl top pods
kubectl top nodes

Common Pod Status Issues

StatusLikely CauseHow to Fix
PendingNo schedulable node (insufficient resources, taints, affinity)Check node capacity, describe pod events
CrashLoopBackOffContainer exits immediately after startingCheck logs with --previous flag
ImagePullBackOffCannot pull container imageCheck image name, tag, and imagePullSecrets
OOMKilledContainer exceeded memory limitIncrease memory limit or fix memory leak
ErrorContainer exited with non-zero codeRead logs to find application error
Terminating (stuck)Finalizers preventing deletionPatch to remove finalizers: kubectl patch pod ... -p '{"metadata":{"finalizers":[]}}'
