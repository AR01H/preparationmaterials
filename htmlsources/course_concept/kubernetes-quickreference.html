Kubernetes Setup
Windows & Linux

Install kubectl & minikube, write manifests, deploy Pods/Services/Deployments, and connect to EKS — complete end to end.

Windows Linux kubectl + minikube
1
Install kubectl
CLI tool to talk to any Kubernetes cluster
Windows — PowerShell
# Option 1: winget
winget install Kubernetes.kubectl

# Option 2: Chocolatey
choco install kubernetes-cli

# Option 3: Manual download
# https://kubernetes.io/releases/download/
# Download kubectl.exe → add to PATH

# Verify
kubectl version --client
# should show: Client Version: v1.x.x
Linux — curl install
# Download latest kubectl binary
curl -LO "https://dl.k8s.io/release/$(curl -L -s \
  https://dl.k8s.io/release/stable.txt)/bin/linux/amd64/kubectl"

# Install
sudo install -o root -g root -m 0755 kubectl /usr/local/bin/kubectl

# OR via apt (Ubuntu)
sudo apt install -y kubectl

# Verify
kubectl version --client
2
Install Minikube — Local Kubernetes Cluster
Run K8s on your laptop for testing — needs Docker installed first
Windows
# Install minikube
winget install Kubernetes.minikube
# OR:
choco install minikube

# Start cluster (uses Docker as driver)
minikube start --driver=docker

# Check cluster is running
minikube status
kubectl cluster-info
kubectl get nodes
# should show 1 node: minikube   Ready

# Open dashboard in browser
minikube dashboard
Linux
# Download and install minikube
curl -LO https://storage.googleapis.com/minikube/releases/latest/minikube-linux-amd64
sudo install minikube-linux-amd64 /usr/local/bin/minikube

# Start cluster
minikube start --driver=docker

# Check status
minikube status
kubectl get nodes
# should show: minikube   Ready

# Open dashboard
minikube dashboard

Core Concepts — What You Need to Know
The 4 main K8s objects you'll use every day
Pod
# Smallest unit
# Runs 1 or more containers
# Temporary — gets recreated
# when it dies

1 Pod = 1 or more
     containers running
     together
Deployment
# Manages multiple Pods
# Handles rolling updates
# Keeps desired replicas
# running always

1 Deployment = N Pods
     auto-heals if
     pods crash
Service
# Stable network access
# to a set of Pods
# Pods can die & change IP
# Service stays the same

Service = stable URL
     pointing to pods
     via label selector
ConfigMap
# Store config separately
# from container image
# Inject as env vars
# or as files

ConfigMap = env vars
     and config files
     for your app

YAML Manifests — Deploy Your App
Write these files then apply with kubectl — same on Windows & Linux
deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 3         ← run 3 pods
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      containers:
      - name: my-app
        image: my-app:latest
        ports:
        - containerPort: 3000
        env:
        - name: NODE_ENV
          value: "production"
        resources:
          requests:
            memory: "128Mi"
            cpu: "250m"
          limits:
            memory: "256Mi"
            cpu: "500m"
service.yaml
apiVersion: v1
kind: Service
metadata:
  name: my-app-service
spec:
  selector:
    app: my-app       ← must match Deployment label
  ports:
  - protocol: TCP
    port: 80          ← external port
    targetPort: 3000   ← container port
  type: LoadBalancer  ← use NodePort for local/minikube

---
# For minikube (local testing) use:
#   type: NodePort
# For AWS EKS use:
#   type: LoadBalancer
terminal — apply manifests
# Apply (create or update)
kubectl apply -f deployment.yaml
kubectl apply -f service.yaml

# Apply all yaml files in a folder
kubectl apply -f ./k8s/

# Check what was created
kubectl get deployments
kubectl get pods
kubectl get services

# For minikube — get URL to access service
minikube service my-app-service --url
5
kubectl Daily Commands
Everything you need — same on Windows & Linux
terminal — most used kubectl commands
# ── View resources ────────────────────────────────
kubectl get pods                ← list all pods
kubectl get pods -o wide        ← with node & IP info
kubectl get all                 ← pods + services + deployments
kubectl describe pod my-pod     ← detailed info + events

# ── Logs ─────────────────────────────────────────
kubectl logs my-pod             ← view logs
kubectl logs -f my-pod          ← follow logs live
kubectl logs my-pod -c my-app   ← specific container

# ── Shell into pod ───────────────────────────────
kubectl exec -it my-pod -- bash
kubectl exec -it my-pod -- sh   ← for alpine

# ── Scale up / down ──────────────────────────────
kubectl scale deployment my-app --replicas=5
kubectl scale deployment my-app --replicas=1

# ── Update image (rolling update) ────────────────
kubectl set image deployment/my-app my-app=my-app:v2
kubectl rollout status deployment/my-app   ← watch rollout
kubectl rollout undo deployment/my-app     ← rollback

# ── Delete resources ─────────────────────────────
kubectl delete pod my-pod
kubectl delete -f deployment.yaml
kubectl delete deployment my-app

# ── Port forward (access pod locally) ────────────
kubectl port-forward pod/my-pod 8080:3000
# now open http://localhost:8080

Namespaces — Separate Environments
Isolate dev / staging / prod in the same cluster
terminal — namespace commands
# Create namespaces
kubectl create namespace dev
kubectl create namespace staging
kubectl create namespace prod

# List namespaces
kubectl get namespaces

# Deploy to a specific namespace
kubectl apply -f deployment.yaml -n dev
kubectl apply -f deployment.yaml -n prod

# Get pods in a namespace
kubectl get pods -n dev
kubectl get pods -n prod

# Set default namespace for current session
kubectl config set-context --current --namespace=dev
# now all kubectl commands use dev namespace

Connect to AWS EKS (Production Cluster)
Point kubectl to your EKS cluster on AWS
AWS CLI must be configured (aws configure) before connecting to EKS. See the AWS SDK guide.
terminal — connect kubectl to EKS (same on Windows & Linux)
# Update kubeconfig to point to your EKS cluster
aws eks update-kubeconfig \
  --region us-east-1 \
  --name my-eks-cluster

# For secondary AWS profile
aws eks update-kubeconfig \
  --region us-east-1 \
  --name my-eks-cluster \
  --profile secondary

# Verify connection
kubectl cluster-info
kubectl get nodes
# should show your EKS worker nodes

# Switch between minikube and EKS
kubectl config get-contexts     ← list all clusters
kubectl config use-context minikube               ← local
kubectl config use-context arn:aws:eks:...        ← AWS EKS
