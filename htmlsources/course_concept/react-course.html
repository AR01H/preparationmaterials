Chapter 01 / Foundations

Introduction to React

Understanding what React is, why it exists, and how to get your environment ready.

What is React?

React is an open-source JavaScript library created by Facebook (Meta) in 2013 for building user interfaces. It is not a full framework — it is focused solely on the view layer of your application. React lets you build complex UIs by composing small, independent, reusable pieces called components.

React operates on a simple but powerful philosophy: describe what you want the UI to look like given some data, and React figures out how to update the browser efficiently.

Declarative

You describe the desired UI state, not the steps to get there. React handles the DOM updates.

Component-Based

Everything is a component. Build encapsulated pieces that manage their own state, then compose them.

Unidirectional Data Flow

Data flows one way: parent to child through props. This makes bugs easier to trace.

Virtual DOM

React maintains a lightweight copy of the DOM. Changes are diffed and batched before hitting the real DOM.

The Virtual DOM Explained

Directly manipulating the browser's DOM is slow. Every change can trigger reflow and repaint operations across the entire page. React solves this with the Virtual DOM — an in-memory JavaScript representation of the real DOM tree.

When your component's state changes, React:

  1. Creates a new Virtual DOM tree reflecting the new state
  2. Diffs it against the previous Virtual DOM (the "reconciliation" algorithm)
  3. Calculates the minimum set of changes needed
  4. Batches and applies only those changes to the real DOM
💡 Key Insight
This diffing process (called Fiber reconciliation since React 16) is what makes React fast — not that the Virtual DOM itself is faster than the real DOM, but that it enables intelligent, batched updates.

JSX & Rendering

Understanding JSX syntax, how it compiles, and how React renders to the DOM.

What is JSX?

JSX (JavaScript XML) is a syntax extension that lets you write HTML-like markup inside JavaScript. It is not HTML — it is syntactic sugar that compiles to React.createElement() calls.

// What you write (JSX)
const element = <h1 className="title">Hello World</h1>

// What Babel compiles it to
const element = React.createElement(
  'h1',
  { className: 'title' },
  'Hello World'
)

Since React 17+, you no longer need to import React in every file for JSX to work — the new JSX transform handles this automatically via the build tool.

JSX Rules

1. Return a Single Root Element

// ❌ Wrong — multiple root elements
return (
  <h1>Title</h1>
  <p>Paragraph</p>
)

// ✅ Wrap in a div
return (
  <div>
    <h1>Title</h1>
    <p>Paragraph</p>
  </div>
)

// ✅ Use a Fragment (no extra DOM node)
return (
  <>
    <h1>Title</h1>
    <p>Paragraph</p>
  </>
)

2. Use className, Not class

// HTML attributes use camelCase in JSX
<div className="container"         // class → className
     htmlFor="email"            // for → htmlFor (on labels)
     onClick={handleClick}     // onclick → onClick
     tabIndex={0}>             // tabindex → tabIndex
</div>

3. Close All Tags

// Self-closing tags must end with />
<img src="photo.jpg" alt="Photo" />
<input type="text" />
<br />

4. JavaScript Expressions with Curly Braces

const name = 'Shankar'
const score = 95

return (
  <div>
    <p>Hello, {name}!</p>                    // Variable
    <p>Score: {score * 2}</p>               // Expression
    <p>{score > 90 ? 'Pass' : 'Fail'}</p>   // Ternary
    <p>{[1,2,3].join(', ')}</p>            // Method call
  </div>
)
⚠️ What You Cannot Do in JSX
You cannot use if statements, for loops, or switch statements directly inside JSX curly braces. These are statements, not expressions. Use ternary operators, logical &&, or extract logic above the return statement.

Rendering Elements

React elements are the smallest building blocks of a React app. An element describes what you want to see on the screen.

// Elements are immutable plain objects
const element = <h1>Hello</h1>
// React.createElement returns: { type: 'h1', props: { children: 'Hello' } }

// Rendering into the DOM
const root = ReactDOM.createRoot(document.getElementById('root'))
root.render(element)

// React only updates what changed
function tick() {
  const element = (
    <p>Time: {new Date().toLocaleTimeString()}</p>
  )
  root.render(element) // Only the time text node updates
}
setInterval(tick, 1000)

Components

The core building blocks of every React application.

Function Components

Modern React uses function components exclusively. A component is simply a JavaScript function that accepts a props object and returns JSX. The function name must start with a capital letter to distinguish it from HTML elements.

// Simplest possible component
function Greeting() {
  return <h1>Hello, World!</h1>
}

// Arrow function syntax (also valid)
const Greeting = () => <h1>Hello, World!</h1>

// A more realistic component
function UserCard({ name, role, avatar }) {
  return (
    <div className="user-card">
      <img src={avatar} alt={name} />
      <h3>{name}</h3>
      <span>{role}</span>
    </div>
  )
}

// Using the component
function App() {
  return (
    <div>
      <UserCard 
        name="Shankar"
        role="Full-Stack Engineer"
        avatar="/avatar.jpg"
      />
    </div>
  )
}

Component Composition

Components can render other components. This nesting is how you build complex UIs from simple pieces. React's component tree is a hierarchy where the topmost component (usually App) renders children, which render their own children, and so on.

function Avatar({ src, name }) {
  return <img src={src} alt={name} className="avatar" />
}

function UserInfo({ user }) {
  return (
    <div className="user-info">
      <Avatar src={user.avatarUrl} name={user.name} />
      <p>{user.name}</p>
    </div>
  )
}

function Comment({ author, text, date }) {
  return (
    <div className="comment">
      <UserInfo user={author} />   // Composed!
      <p>{text}</p>
      <time>{date}</time>
    </div>
  )
}

Class Components (Legacy)

Before hooks (React 16.8, 2019), components that needed state or lifecycle methods had to be written as classes. You may encounter these in older codebases.

import { Component } from 'react'

class Counter extends Component {
  constructor(props) {
    super(props)
    this.state = { count: 0 }
  }

  render() {
    return (
      <div>
        <p>Count: {this.state.count}</p>
        <button onClick={() => this.setState({ count: this.state.count + 1 })}>
          Increment
        </button>
      </div>
    )
  }
}
⚠️ Class Components Are Legacy
The React team recommends writing all new code as function components with hooks. Class components will continue to work but won't receive new features.

Keeping Components Pure

React expects components to be pure functions with respect to their inputs. Given the same props and state, a component must always return the same JSX. It should not have side effects during rendering.

// ❌ Impure — mutates external state during render
let count = 0
function BadComponent() {
  count++  // Side effect during render!
  return <p>Count: {count}</p>
}

// ✅ Pure — same input always produces same output
function PureComponent({ count }) {
  return <p>Count: {count}</p>
}

// Side effects belong in event handlers or useEffect

Props & Data Flow

How data flows through the component tree.

Understanding Props

Props (short for properties) are how you pass data from a parent component to a child component. Props are read-only — a child must never modify its own props. This enforces the principle of unidirectional data flow.

// Defining and using props
function Button(props) {
  return (
    <button
      onClick={props.onClick}
      disabled={props.disabled}
      className={`btn btn-${props.variant}`}
    >
      {props.children}
    </button>
  )
}

// Destructuring props (preferred)
function Button({ onClick, disabled = false, variant = 'primary', children }) {
  return (
    <button
      onClick={onClick}
      disabled={disabled}
      className={`btn btn-${variant}`}
    >
      {children}
    </button>
  )
}

// Using the Button
<Button variant="danger" onClick={handleDelete}>
  Delete Account
</Button>

The children Prop

Everything between a component's opening and closing tags is passed as props.children. This is powerful for building layout and wrapper components.

function Card({ title, children }) {
  return (
    <div className="card">
      {title && <h2 className="card-title">{title}</h2>}
      <div className="card-body">
        {children}  // Anything passed between <Card> tags
      </div>
    </div>
  )
}

// Usage
<Card title="User Profile">
  <p>Name: Shankar</p>
  <p>Role: Full-Stack Engineer</p>
  <Button>Edit</Button>
</Card>

Prop Spreading & Forwarding

// Spread operator to pass all props
function Input({ label, id, ...rest }) {
  return (
    <div>
      <label htmlFor={id}>{label}</label>
      <input id={id} {...rest} />  // Forward type, value, onChange, etc.
    </div>
  )
}

Lifting State Up

When two sibling components need to share state, you "lift" that state up to their closest common parent. The parent holds the state and passes it down as props.

// Parent controls the shared state
function TemperatureConverter() {
  const [celsius, setCelsius] = useState(0)

  const fahrenheit = (celsius * 9/5) + 32

  return (
    <div>
      <TempInput
        unit="Celsius"
        value={celsius}
        onChange={setCelsius}   // Pass setter down
      />
      <TempInput
        unit="Fahrenheit"
        value={fahrenheit}
        onChange={val => setCelsius((val - 32) * 5/9)}
      />
    </div>
  )
}

function TempInput({ unit, value, onChange }) {
  return (
    <label>
      {unit}: 
      <input
        type="number"
        value={value}
        onChange={e => onChange(Number(e.target.value))}
      />
    </label>
  )
}

State & Events

Making components interactive with state and event handling.

What is State?

State is data that changes over time in a component. Unlike props (which come from the parent), state is internal and owned by the component. When state changes, React re-renders the component and its children.

💡 State vs Props vs Variables
Regular variables reset on every render. Props are external, read-only data. State is internal, mutable data that persists across renders and triggers re-renders when changed. 
import { useState } from 'react'

function Counter() {
  // useState returns [currentValue, setterFunction]
  const [count, setCount] = useState(0)  // 0 is the initial value

  return (
    <div>
      <p>Count: {count}</p>
      <button onClick={() => setCount(count + 1)}>+</button>
      <button onClick={() => setCount(count - 1)}>-</button>
      <button onClick={() => setCount(0)}>Reset</button>
    </div>
  )
}

State Updates are Asynchronous

React batches state updates for performance. This means calling setCount(count + 1) multiple times in a row will not increment three times — it reads the same stale count value each time. Use the functional updater form instead:

// ❌ Problem — reads stale state
function handleTripleIncrement() {
  setCount(count + 1)  // count is still 0
  setCount(count + 1)  // count is still 0
  setCount(count + 1)  // count is still 0 → result: 1
}

// ✅ Functional updater — receives the latest state
function handleTripleIncrement() {
  setCount(prev => prev + 1)  // 0 → 1
  setCount(prev => prev + 1)  // 1 → 2
  setCount(prev => prev + 1)  // 2 → 3 → result: 3
}

Object & Array State

State must be treated as immutable. Never mutate state directly — always create a new object or array:

const [user, setUser] = useState({ name: 'Shankar', age: 25 })
const [items, setItems] = useState(['React', 'Node.js'])

// ❌ Wrong — mutating state directly
user.name = 'New Name'       // React won't re-render!
items.push('Swift')          // React won't re-render!

// ✅ Correct — create new object/array
setUser({ ...user, name: 'New Name' })         // Spread + override
setItems([...items, 'Swift'])                   // Append
setItems(items.filter(i => i !== 'React'))     // Remove
setItems(items.map(i => i === 'Node.js' ? 'Bun' : i)) // Update

Event Handling

function Form() {
  const [value, setValue] = useState('')

  // Event handlers receive a SyntheticEvent object
  const handleChange = (e) => {
    setValue(e.target.value)     // e.target is the DOM element
  }

  const handleSubmit = (e) => {
    e.preventDefault()           // Prevent page reload
    console.log('Submitted:', value)
  }

  const handleKeyDown = (e) => {
    if (e.key === 'Enter') handleSubmit(e)
    if (e.key === 'Escape') setValue('')
  }

  return (
    <form onSubmit={handleSubmit}>
      <input
        value={value}
        onChange={handleChange}
        onKeyDown={handleKeyDown}
      />
      <button type="submit">Submit</button>
    </form>
  )
}

useState & useReducer

Deep dive into React's two fundamental state management hooks.

useState in Depth

useState is the primary hook for local component state. Here are patterns you'll encounter in real applications:

// Lazy initialization — expensive computation runs once
const [state, setState] = useState(() => {
  const saved = localStorage.getItem('key')
  return saved ? JSON.parse(saved) : defaultValue
})

// Multiple state values — prefer separate for unrelated state
const [name, setName] = useState('')
const [email, setEmail] = useState('')
const [loading, setLoading] = useState(false)

// Related state can be grouped as object
const [form, setForm] = useState({
  name: '', email: '', password: ''
})

const updateField = (field, value) =>
  setForm(prev => ({ ...prev, [field]: value }))

useReducer

useReducer is preferred over useState when state logic is complex, when the next state depends on the previous state in non-trivial ways, or when you have multiple related state updates that should happen together.

import { useReducer } from 'react'

// Define the shape of all possible state changes
const cartReducer = (state, action) => {
  switch (action.type) {
    case 'ADD_ITEM':
      const existing = state.items.find(i => i.id === action.item.id)
      if (existing) {
        return {
          ...state,
          items: state.items.map(i =>
            i.id === action.item.id
              ? { ...i, qty: i.qty + 1 }
              : i
          )
        }
      }
      return { ...state, items: [...state.items, { ...action.item, qty: 1 }] }

    case 'REMOVE_ITEM':
      return {
        ...state,
        items: state.items.filter(i => i.id !== action.id)
      }

    case 'CLEAR_CART':
      return { ...state, items: [] }

    default:
      return state
  }
}

function Cart() {
  const [state, dispatch] = useReducer(cartReducer, { items: [] })

  return (
    <div>
      {state.items.map(item => (
        <div key={item.id}>
          {item.name} × {item.qty}
          <button onClick={() => dispatch({ type: 'REMOVE_ITEM', id: item.id })}>
            Remove
          </button>
        </div>
      ))}
      <button onClick={() => dispatch({ type: 'CLEAR_CART' })}>Clear</button>
    </div>
  )
}
✅ When to Use useReducer vs useState
Use useReducer when: (1) state has 3+ sub-values, (2) state transitions are complex or interdependent, (3) you want to extract and test state logic separately, or (4) you want clear action names for debugging.

useEffect & Lifecycle

Synchronizing components with external systems: APIs, timers, subscriptions.

Understanding Effects

useEffect lets you run side effects — code that reaches outside of React (API calls, DOM manipulation, subscriptions, timers). Effects run after the render is committed to the screen.

import { useState, useEffect } from 'react'

function UserProfile({ userId }) {
  const [user, setUser] = useState(null)
  const [loading, setLoading] = useState(true)

  useEffect(() => {
    // Effect runs after every render where userId changed
    let cancelled = false   // Prevent stale state update

    setLoading(true)
    fetch(`/api/users/${userId}`)
      .then(r => r.json())
      .then(data => {
        if (!cancelled) {
          setUser(data)
          setLoading(false)
        }
      })

    // Cleanup function — runs before next effect or on unmount
    return () => { cancelled = true }
  }, [userId])   // Dependency array — only re-run when userId changes

  if (loading) return <p>Loading...</p>
  return <h1>{user?.name}</h1>
}

The Dependency Array

Dependency ArrayWhen Effect Runs
Not providedAfter every single render
[] (empty)Once, after the first render (mount)
[a, b]After mount, and whenever a or b changes
 
// Run once on mount (like componentDidMount)
useEffect(() => {
  document.title = 'My App'
  fetchInitialData()
}, [])

// Run on every render (rare, usually a mistake)
useEffect(() => {
  console.log('Rendered')
})

// Update document title when count changes
useEffect(() => {
  document.title = `Count: ${count}`
}, [count])

// Cleanup — e.g. event listeners, timers, subscriptions
useEffect(() => {
  const id = setInterval(() => setTime(new Date()), 1000)
  return () => clearInterval(id)   // ← cleanup
}, [])
🚫 Common Mistake: Missing Dependencies
If you use a variable inside useEffect without listing it in the dependency array, you'll get stale closures. Always include everything your effect uses. Use the eslint-plugin-react-hooks rule exhaustive-deps to catch this automatically.

Fetching Data with AbortController

useEffect(() => {
  const controller = new AbortController()

useRef, useMemo & useCallback

Refs for DOM access and memoization hooks for performance.

useRef

useRef returns a mutable ref object whose .current property persists across renders without causing re-renders. It has two main use cases:

1. Accessing DOM Elements

import { useRef, useEffect } from 'react'

function SearchInput() {
  const inputRef = useRef(null)

  useEffect(() => {
    inputRef.current.focus()   // Auto-focus on mount
  }, [])

  const handleClear = () => {
    inputRef.current.value = ''
    inputRef.current.focus()
  }

  return (
    <div>
      <input ref={inputRef} type="search" />
      <button onClick={handleClear}>Clear</button>
    </div>
  )
}

2. Storing Mutable Values Without Re-renders

function Stopwatch() {
  const [time, setTime] = useState(0)
  const intervalRef = useRef(null)   // Persists, doesn't cause re-render

  const start = () => {
    intervalRef.current = setInterval(
      () => setTime(t => t + 1), 1000
    )
  }

  const stop = () => clearInterval(intervalRef.current)

  return (
    <div>
      <p>{time}s</p>
      <button onClick={start}>Start</button>
      <button onClick={stop}>Stop</button>
    </div>
  )
}

useMemo

useMemo memoizes a computed value. The computation only re-runs when its dependencies change. Use it for expensive calculations that don't need to run on every render.

import { useMemo } from 'react'

function DataTable({ data, filterText, sortField }) {
  // Only recomputes when data, filterText, or sortField change
  const processed = useMemo(() => {
    return data
      .filter(item => item.name.toLowerCase().includes(filterText))
      .sort((a, b) => a[sortField] > b[sortField] ? 1 : -1)
  }, [data, filterText, sortField])  // dependencies

  return <Table rows={processed} />
}

useCallback

useCallback memoizes a function — it returns the same function reference across renders unless its dependencies change. Useful when passing callbacks to memoized child components.

import { useCallback, memo } from 'react'

const ExpensiveList = memo(({ items, onDelete }) => {
  // Only re-renders when items or onDelete reference changes
  return (
    <ul>
      {items.map(item => (
        <li key={item.id}>
          {item.text}
          <button onClick={() => onDelete(item.id)}>×</button>
        </li>
      ))}
    </ul>
  )
})

function Parent() {
  const [items, setItems] = useState(initialItems)

  // Stable reference — won't cause ExpensiveList to re-render
  const handleDelete = useCallback((id) => {
    setItems(prev => prev.filter(item => item.id !== id))
  }, [])  // No deps needed — setItems is stable

  return <ExpensiveList items={items} onDelete={handleDelete} />
}
⚠️ Don't Over-Optimize
useMemo and useCallback have their own overhead. Only use them when you have a measured performance problem. Premature optimization is a common React anti-pattern.

useContext & Global State

Sharing state across the component tree without prop drilling.

The Prop Drilling Problem

When deeply nested components need data from a high-level ancestor, passing props through every intermediate component is called prop drilling. It makes code verbose and hard to maintain. Context solves this.

// ❌ Prop drilling — every level passes theme down
<App theme={theme} />
  <Layout theme={theme} />
    <Sidebar theme={theme} />
      <ThemeButton theme={theme} />  // Finally used here

Creating and Using Context

import { createContext, useContext, useState } from 'react'

// 1. Create the context
const ThemeContext = createContext({
  theme: 'light',
  toggleTheme: () => {}
})

// 2. Create a Provider component
export function ThemeProvider({ children }) {
  const [theme, setTheme] = useState('light')

  const toggleTheme = () =>
    setTheme(t => t === 'light' ? 'dark' : 'light')

  return (
    <ThemeContext.Provider value={{ theme, toggleTheme }}>
      {children}
    </ThemeContext.Provider>
  )
}

// 3. Custom hook for consuming (best practice)
export function useTheme() {
  const context = useContext(ThemeContext)
  if (!context) throw new Error('useTheme must be inside ThemeProvider')
  return context
}

// 4. Wrap your app with the Provider
function App() {
  return (
    <ThemeProvider>
      <Layout />   // No theme prop needed!
    </ThemeProvider>
  )
}

// 5. Consume anywhere in the tree
function ThemeButton() {
  const { theme, toggleTheme } = useTheme()
  return (
    <button onClick={toggleTheme}>
      Switch to {theme === 'light' ? 'Dark' : 'Light'}
    </button>
  )
}
⚠️ Context Performance
Every component that consumes a context will re-render when the context value changes. For high-frequency updates, split your contexts (e.g., separate AuthContext for user data from UIContext for theme), or use a dedicated state management library like Zustand.

Custom Hooks

Extracting and reusing stateful logic across components.

Creating Custom Hooks

A custom hook is a JavaScript function whose name starts with use and that calls other hooks. They let you extract component logic into reusable functions.

// useLocalStorage — persists state to localStorage
function useLocalStorage(key, initialValue) {
  const [value, setValue] = useState(() => {
    try {
      const item = localStorage.getItem(key)
      return item ? JSON.parse(item) : initialValue
    } catch { return initialValue }
  })

  const setStoredValue = (newValue) => {
    const valueToStore = newValue instanceof Function
      ? newValue(value)
      : newValue
    setValue(valueToStore)
    localStorage.setItem(key, JSON.stringify(valueToStore))
  }

  return [value, setStoredValue]
}

Why Keys Matter

Keys help React identify which items in a list have changed, been added, or removed. Without keys, React re-renders the entire list. With good keys, only changed items re-render.

Error Boundaries

Gracefully handling runtime errors in the component tree.

What are Error Boundaries?

An Error Boundary is a React component that catches JavaScript errors in its child component tree, logs those errors, and displays a fallback UI. They are still written as class components (no hook equivalent yet).

class ErrorBoundary extends Component {
  constructor(props) {
    super(props)
    this.state = { hasError: false, error: null }
  }

  // Called when a descendant throws
  static getDerivedStateFromError(error) {
    return { hasError: true, error }
  }

  // For logging errors to a service
  componentDidCatch(error, info) {
    logErrorToService(error, info.componentStack)
  }

  render() {
    if (this.state.hasError) {
      return this.props.fallback || (
        <div>
          <h2>Something went wrong</h2>
          <button onClick={() => this.setState({ hasError: false })}>
            Try again
          </button>
        </div>
      )
    }
    return this.props.children
  }
}

// Usage — wrap sections independently
<ErrorBoundary fallback={<p>Chart failed to load</p>}>
  <ComplexChart />
</ErrorBoundary>
⚠️ What Error Boundaries Don't Catch
Event handlers (use try/catch inside them), async code (setTimeout, fetch), server-side rendering errors, and errors thrown in the error boundary itself.

Key Best Practices

1. One Component Per File

Each file exports one primary component. This improves discoverability and tree-shaking.

2. Colocate Related Code

Keep styles, tests, and types close to the component they belong to. A ProductCard folder might contain ProductCard.jsx, ProductCard.test.jsx, and ProductCard.module.css.

3. Avoid Anonymous Default Exports

  return <div>...</div>

4. Never Put Logic in JSX

// ✅ Compute above, render below
const total = data.items.filter(i => i.active).reduce((sum, i) => sum + i.price, 0)
return <p>{total}</p>

5. Common Anti-Patterns to Avoid

  • Derived state in useState — compute it from existing state instead
  • useEffect for derived state — calculate during render, not in an effect
  • Overusing Context — use for truly global data; Zustand for shared state
  • Premature optimization — profile before adding useMemo/useCallback
  • Mutating state directly — always return a new object/array
  • Index as key in dynamic lists — use stable, unique IDs
  • Nested component definitions — define components at the top level, not inside other components
// ❌ Component defined inside render — recreated every render
function Parent() {
  function Child() { return <p>...</p> }  // Bad!
  return <Child />
}

// ✅ Defined at module level
function Child() { return <p>...</p> }
function Parent() { return <Child /> }

Thinking in React — A Mental Model

When building a React feature, follow this mental process:

  1. Break UI into a component hierarchy — draw boxes around each UI piece
  2. Build a static version first — no state, just props and hardcoded data
  3. Identify the minimal state — what's the least state to make it interactive?
  4. Decide where state lives — what's the common ancestor of all components that need it?
  5. Add inverse data flow — pass setters down so children can update ancestor state
🎯 The Golden Rule
If you can compute it from existing state or props — don't store it in state. Derived values should be computed during render, not synchronized with effects.

The Core Philosophy

There is no single "correct" folder structure, but there is a wrong one: one you can't navigate six months later. The best structures follow two rules — colocation (keep related things together) and feature isolation (features don't reach into each other's guts).

💡 Golden Rule
Things that change together should live together. Don't split a feature across 6 folders just because they have different file types.

✅ The Index Barrel Rule
Each feature folder has an index.js that acts as its public API. Other features import from 'features/auth', never from 'features/auth/store/authSlice'. This lets you refactor internals without breaking importers.

import { useAuth } from '../../features/auth/hooks/useAuth'  // Messy ❌

Vite In-Depth

Everything Vite can do — from zero config to advanced customisation.

Why Vite?

Vite (French for "fast") replaces webpack/CRA with two key innovations: during development it serves files as native ES Modules — no bundling, instant server start. For production it uses Rollup to produce optimised bundles. The result: dev server starts in ~300ms regardless of project size.

Native ESM Dev Server

Browser fetches modules on demand. Only the file you changed is re-processed — not the whole bundle.

Pre-bundled Dependencies

node_modules are pre-bundled with esbuild once. 10–100× faster than webpack's JS-based processing.

True HMR

Module-level hot replacement. React Fast Refresh is built in — state survives file edits.

Rollup Production Build

Mature, tree-shaking optimised production bundles with automatic code splitting.

vite.config.ts — Complete Reference

import { defineConfig } from 'vite'
import react from '@vitejs/plugin-react'
import path from 'path'

export default defineConfig({
  plugins: [
    react(),    // React Fast Refresh + JSX transform
  ],

  // ── Path aliases ────────────────────────────
  resolve: {
    alias: {
      '@': path.resolve(__dirname, './src'),
      '@components': path.resolve(__dirname, './src/components'),
      '@features': path.resolve(__dirname, './src/features'),
      '@hooks': path.resolve(__dirname, './src/hooks'),
      '@utils': path.resolve(__dirname, './src/utils'),
    }
  },

  // ── Dev server ──────────────────────────────
  server: {
    port: 3000,
    open: true,         // Auto-open browser
    proxy: {
      // Forward /api requests to backend (avoid CORS in dev)
      '/api': {
        target: 'http://localhost:8080',
        changeOrigin: true,
        rewrite: path => path.replace(/^\/api/, '')
      }
    }
  },

  // ── Build options ───────────────────────────
  build: {
    outDir: 'dist',
    sourcemap: true,           // Enable for production debugging
    chunkSizeWarningLimit: 1000, // Warn if chunk > 1MB
    rollupOptions: {
      output: {
        // Manual chunk splitting — keeps vendor libs separate
        manualChunks(id) {
          if (id.includes('node_modules')) {
            if (id.includes('react')) return 'react-vendor'
            if (id.includes('@tanstack')) return 'query-vendor'
            return 'vendor'
          }
        }
      }
    }
  },

  // ── CSS ─────────────────────────────────────
  css: {
    modules: {
      localsConvention: 'camelCase'  // .my-class → styles.myClass
    }
  },

  // ── Environment variables ────────────────────
  // Vite exposes VITE_* vars to client via import.meta.env
  envPrefix: 'VITE_',

  // ── Preview server (production preview) ─────
  preview: {
    port: 4173
  }
})

Why Monorepo?

A monorepo stores multiple projects (apps + shared libraries) in one git repository. You share code, run one lint/test command across everything, and ship atomic changes that touch both frontend and backend in a single PR.

One packages/ui library used by web, mobile, and docs — no publish/versioning needed.

Atomic Changes

A breaking API change + all consuming apps fixed in one commit. No inter-repo coordination.

Unified Tooling

pnpm Efficiency

pnpm uses a content-addressable store — each package installed once globally, symlinked into projects.

    "test": { "outputs": ["coverage/**"] }

Aliases & Path Setup

Clean imports with no ../../ hell — and making TypeScript & Vite agree.

The Problem: Relative Path Hell

// ❌ Fragile — breaks when you move a file
import { Button } from '../../../../components/Button'
import { useAuth } from '../../../hooks/useAuth'
import { formatDate } from '../../utils/formatDate'

// ✅ Clean, refactor-proof
import { Button } from '@/components/Button'
import { useAuth } from '@/hooks/useAuth'
import { formatDate } from '@/utils/formatDate'

Step 1: Configure Vite

// vite.config.ts
import { defineConfig } from 'vite'
import react from '@vitejs/plugin-react'
import path from 'path'

export default defineConfig({
  plugins: [react()],
  resolve: {
    alias: {
      '@': path.resolve(__dirname, 'src'),
    }
  }
})

Step 2: Configure TypeScript

Vite handles runtime resolution but TypeScript needs its own config to understand the alias for type-checking and IDE autocomplete.

// tsconfig.json
{
  "compilerOptions": {
    "baseUrl": ".",
    "paths": {
      "@/*": ["src/*"],
      "@components/*": ["src/components/*"],
      "@features/*": ["src/features/*"],
      "@hooks/*": ["src/hooks/*"],
      "@utils/*": ["src/utils/*"],
      "@types/*": ["src/types/*"]
    }
  }
}
