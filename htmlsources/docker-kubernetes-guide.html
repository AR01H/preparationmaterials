🐳 DOCKER
D-01

Core Concepts & Definitions

📦

Image

A read-only template with instructions for creating a container. Built from a Dockerfile, stored in a registry.

immutable blueprint
🟢

Container

A running instance of an image. Isolated process with its own filesystem, network, and process space.

image + runtime
📜

Dockerfile

A text file with sequential instructions to build a Docker image layer by layer. Each instruction = a layer.

build recipe
💾

Volume

Persistent storage that survives container restarts. Managed by Docker, mounted into container paths.

persistent data
🌐

Network

Virtual network connecting containers. Types: bridge (default), host, none, overlay (swarm).

container comms
🏪

Registry

Storage and distribution service for images. Docker Hub is the default public registry.

image storage

Dockerfile — Instructions & Best Practices

InstructionDefinitionUse case
FROMSets the base image for the buildFROM node:20-alpine
WORKDIRSets working directory inside containerWORKDIR /app
COPYCopies files from host → containerCOPY package*.json ./
ADDLike COPY but also handles URLs and tarballsADD archive.tar.gz /opt/
RUNExecutes command and creates a new layerRUN npm install
CMDDefault command when container starts (overridable)CMD ["node", "index.js"]
ENTRYPOINTMain command — not overridden by argsENTRYPOINT ["python"]
ENVSet environment variablesENV PORT=3000
ARGBuild-time variable (not in final image)ARG VERSION=1.0
EXPOSEDocuments which port the container usesEXPOSE 3000
VOLUMECreates a mount point for persistent dataVOLUME ["/data"]
USERSets the user to run subsequent commandsUSER node
HEALTHCHECKCommand to check if container is healthyHEALTHCHECK CMD curl -f localhost/health
Dockerfile — Production Node.js (multi-stage) DOCKERFILE 
# ── Stage 1: Build ───────────────────────────────────────────────
FROM node:20-alpine AS builder

WORKDIR /app

# Copy dependency files first (better layer caching)
COPY package*.json ./
RUN npm ci --only=production

COPY . .
RUN npm run build

# ── Stage 2: Runtime (lean final image) ──────────────────────────
FROM node:20-alpine AS runtime

# Security: don't run as root
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
USER appuser

WORKDIR /app

# Copy ONLY what's needed from builder
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules

ENV NODE_ENV=production
ENV PORT=3000

EXPOSE 3000

HEALTHCHECK --interval=30s --timeout=3s \
  CMD wget -qO- http://localhost:3000/health || exit 1

CMD ["node", "dist/index.js"]
✅ BEST PRACTICES
Use multi-stage builds to keep final images small. Order layers from least-changed to most-changed (deps before source code) for faster caching. Pin base image versions (node:20.11-alpine3.19, not node:latest).

Essential Docker Commands

images SHELL 
# ── IMAGE COMMANDS ────────────────────────────────────────────────
docker build -t myapp:1.0 .             # build image from Dockerfile
docker build -t myapp:1.0 --no-cache .   # force rebuild all layers
docker build --build-arg VERSION=2.0 .   # pass ARG at build time
docker images                              # list local images
docker image ls                            # same as above
docker pull nginx:alpine                  # pull from registry
docker push username/myapp:1.0           # push to Docker Hub
docker tag myapp:1.0 username/myapp:1.0   # rename / tag image
docker rmi myapp:1.0                      # delete image
docker image prune -a                     # delete all unused images
docker history myapp:1.0                  # view layer history
docker inspect myapp:1.0                  # full image metadata
docker save myapp:1.0 -o myapp.tar       # export to tar file
docker load -i myapp.tar                  # import from tar file
containers SHELL 
# ── CONTAINER LIFECYCLE ───────────────────────────────────────────
docker run nginx                           # run container (foreground)
docker run -d nginx                        # run detached (background)
docker run -d -p 8080:80 --name web nginx # with port mapping and name
docker run -it ubuntu bash                # interactive terminal
docker run --rm alpine echo "hello"       # auto-remove when done
docker run -e ENV=prod -v /host:/cont myapp # env + volume mount
docker run --memory 512m --cpus 1 myapp  # resource limits
docker run --network my-net myapp         # join custom network

docker ps                                  # list running containers
docker ps -a                               # all containers incl. stopped
docker stop web                            # graceful stop (SIGTERM)
docker kill web                            # force stop (SIGKILL)
docker start web                           # start stopped container
docker restart web                         # stop then start
docker rm web                              # delete stopped container
docker rm -f web                           # force remove running container

# ── DEBUGGING ────────────────────────────────────────────────────
docker logs web                            # view stdout logs
docker logs -f web                         # follow/tail logs
docker logs --tail 100 web                 # last 100 lines
docker exec -it web bash                  # shell into running container
docker exec web cat /etc/hosts            # run single command
docker inspect web                         # full container metadata
docker stats                               # live CPU/memory usage
docker top web                             # running processes inside
docker cp web:/app/log.txt ./log.txt       # copy file from container
volumes & networks SHELL 
# ── VOLUMES ──────────────────────────────────────────────────────
docker volume create mydata               # create named volume
docker volume ls                           # list volumes
docker volume inspect mydata              # volume details
docker volume rm mydata                    # delete volume
docker volume prune                        # delete all unused volumes

# Bind mount (host dir): -v /host/path:/container/path
docker run -v $(pwd)/data:/app/data myapp
# Named volume: -v volumename:/container/path
docker run -v mydata:/app/data myapp

# ── NETWORKS ─────────────────────────────────────────────────────
docker network create my-net              # create bridge network
docker network ls                          # list networks
docker network inspect my-net             # network details
docker network connect my-net web         # add container to network
docker network disconnect my-net web      # remove from network
docker network rm my-net                   # delete network

# ── REGISTRY ─────────────────────────────────────────────────────

docker logout                              # logout
docker search nginx                        # search Docker Hub
D-04

Docker Compose

Docker Compose — tool to define and run multi-container applications from a single YAML file.

docker-compose.yml — full stack example COMPOSE 
services:

  app:
    build:
      context: .
      dockerfile: Dockerfile
      target: runtime           # multi-stage target
    image: myapp:latest
    container_name: api
    ports:
      - "3000:3000"
    environment:
      - NODE_ENV=production
      - DB_URL=postgres://user:pass@db:5432/mydb
    env_file: .env                # load from .env file
    volumes:
      - ./logs:/app/logs         # bind mount for logs
    depends_on:
      db:
        condition: service_healthy  # wait for DB ready
    networks:
      - backend
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:3000/health"]
      interval: 30s
      timeout: 10s
      retries: 3

  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_USER: user
      POSTGRES_PASSWORD: pass
      POSTGRES_DB: mydb
    volumes:
      - pgdata:/var/lib/postgresql/data  # named volume → persistent
    networks:
      - backend
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U user"]
      interval: 10s
      timeout: 5s
      retries: 5

  redis:
    image: redis:7-alpine
    command: redis-server --appendonly yes
    volumes:
      - redisdata:/data
    networks:
      - backend

  nginx:
    image: nginx:alpine
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./nginx.conf:/etc/nginx/nginx.conf:ro
      - ./certs:/etc/nginx/certs:ro
    depends_on: [app]
    networks:
      - backend
      - frontend

volumes:
  pgdata:
  redisdata:

networks:
  backend:
    driver: bridge
  frontend:
    driver: bridge

Real-World Docker Use Cases

🧪
Local Dev Environment
USE CASE 1

Run databases and services locally without installing them. Team gets identical environments.

# Spin up postgres + redis for local dev in seconds
docker compose -f docker-compose.dev.yml up -d
# Dev workflow: code on host, services in containers
🚀
CI/CD Pipeline — Build & Push
USE CASE 2

Build images in CI, push to registry, pull in production. Ensures what you test is what you deploy.

# In GitHub Actions / Jenkins:
docker build -t myapp:${{ github.sha }} .
docker push myapp:${{ github.sha }}
docker tag myapp:${{ github.sha }} myapp:latest
docker push myapp:latest
🔒
Isolated Test Runs
USE CASE 3

Run tests in a clean container that gets destroyed after. No state pollution between test runs.

docker run --rm -v $(pwd):/app -w /app node:20 npm test
# Container auto-removed after tests finish (--rm)

K-01

Core Concepts & Definitions

🏗️

Cluster

A set of machines (nodes) running Kubernetes. Has one control plane and one or more worker nodes.

top-level unit
🖥️

Node

A single worker machine (VM or physical). Runs Pods via the kubelet process and container runtime.

worker machine
🫛

Pod

Smallest deployable unit. Wraps one or more containers that share network and storage on a node.

1+ containers
📋

Deployment

Manages a ReplicaSet to ensure N replicas of your Pod are always running. Handles rolling updates.

desired state
🌐

Service

Stable network endpoint (ClusterIP/NodePort/LoadBalancer) that routes traffic to a set of Pods.

pod routing
📥

Ingress

HTTP/HTTPS routing rules. Exposes services externally via host/path rules, handles TLS termination.

L7 routing
🗺️

ConfigMap

Stores non-sensitive key-value config data. Injected into Pods as env vars or mounted as files.

app config
🔐

Secret

Like ConfigMap but base64-encoded for sensitive data (passwords, tokens, certs). Not encrypted by default.

sensitive data
💾

PersistentVolume

Cluster-level storage resource. PVC (claim) requests storage from a PV on behalf of a Pod.

PV + PVC
📦

Namespace

Virtual cluster within a cluster. Isolates resources between teams/environments (dev, staging, prod).

isolation scope
🔄

StatefulSet

Like Deployment but for stateful apps (DBs). Gives stable identities and ordered pod creation.

stateful pods
⚙️

DaemonSet

Ensures a copy of a Pod runs on every node. Used for log collectors, monitoring agents.

per-node pods

Core Object YAML Manifests

deployment.yaml DEPLOYMENT 
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-deployment
  namespace: production
  labels:
    app: api
    version: v1
spec:
  replicas: 3                   # run 3 pods always
  selector:
    matchLabels:
      app: api
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1               # 1 extra pod during update
      maxUnavailable: 0          # zero downtime
  template:
    metadata:
      labels:
        app: api
    spec:
      containers:
        - name: api
          image: username/myapp:1.0
          ports:
            - containerPort: 3000
          env:
            - name: NODE_ENV
              value: production
            - name: DB_PASSWORD
              valueFrom:
                secretKeyRef:             # from Secret
                  name: db-secret
                  key: password
            - name: APP_CONFIG
              valueFrom:
                configMapKeyRef:          # from ConfigMap
                  name: app-config
                  key: log_level
          resources:
            requests:
              memory: 128Mi
              cpu: 100m           # 0.1 CPU core
            limits:
              memory: 512Mi
              cpu: 500m
          livenessProbe:
            httpGet:
              path: /health
              port: 3000
            initialDelaySeconds: 15
            periodSeconds: 20
          readinessProbe:
            httpGet:
              path: /ready
              port: 3000
            initialDelaySeconds: 5
            periodSeconds: 10
service.yaml + ingress.yaml SERVICE / INGRESS 
# ── Service: exposes Pods internally ─────────────────────────────
apiVersion: v1
kind: Service
metadata:
  name: api-service
  namespace: production
spec:
  selector:
    app: api               # routes to pods with this label
  ports:
    - protocol: TCP
      port: 80            # service port
      targetPort: 3000   # pod/container port
  type: ClusterIP        # ClusterIP | NodePort | LoadBalancer

---

# ── Ingress: exposes to internet via nginx ingress controller ──────
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: api-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
    cert-manager.io/cluster-issuer: letsencrypt
spec:
  tls:
    - hosts: [api.example.com]
      secretName: api-tls
  rules:
    - host: api.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: api-service
                port:
                  number: 80

kind: Secret
metadata:
  name: db-secret
type: Opaque
data:
  password: cGFzc3dvcmQxMjM=   # base64 of "password123"

---

# ── HorizontalPodAutoscaler ───────────────────────────────────────
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: api-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: api-deployment
  minReplicas: 2
  maxReplicas: 10
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70   # scale when CPU > 70%

Additional YAML Manifests

statefulset.yaml — PostgreSQL STATEFULSET 
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: postgres
spec:
  serviceName: postgres        # stable DNS: postgres-0.postgres
  replicas: 1
  selector:
    matchLabels: { app: postgres }
  template:
    metadata:
      labels: { app: postgres }
    spec:
      containers:
        - name: postgres
          image: postgres:16
          env:
            - name: POSTGRES_PASSWORD
              valueFrom:
                secretKeyRef: { name: db-secret, key: password }
          volumeMounts:
            - name: pgdata
              mountPath: /var/lib/postgresql/data
  volumeClaimTemplates:             # auto PVC per pod
    - metadata: { name: pgdata }
      spec:
        accessModes: [ReadWriteOnce]
        resources:
          requests: { storage: 10Gi }
cronjob.yaml CRONJOB 
apiVersion: batch/v1
kind: CronJob
metadata:
  name: db-backup
spec:
  schedule: "0 2 * * *"           # every day 2 AM UTC
  concurrencyPolicy: Forbid       # don't overlap runs
  jobTemplate:
    spec:
      template:
        spec:
          restartPolicy: OnFailure
          containers:
            - name: backup
              image: postgres:16
              command: ["/bin/sh", "-c"]
              args:
                - pg_dump $DATABASE_URL | gzip > /backup/db-$(date +%Y%m%d).gz
              env:
                - name: DATABASE_URL
                  valueFrom:
                    secretKeyRef: { name: db-secret, key: url }

Auto-Scaling on Load
USE CASE 2

HPA automatically adds/removes pods based on CPU/memory. Min 2 pods, max 10, scale at 70% CPU.

# Apply HPA (or use YAML from above)
kubectl autoscale deployment api --min=2 --max=10 --cpu-percent=70
kubectl get hpa                          # watch replicas change
🔍
Debugging a Crashing Pod
USE CASE 3

Standard debugging workflow when a pod shows CrashLoopBackOff or Error status.

kubectl get pods                         # identify crashed pod name
kubectl describe pod api-xyz             # check Events section
kubectl logs api-xyz                     # current logs
kubectl logs --previous api-xyz          # logs BEFORE crash
# If no shell available, run debug pod alongside:
kubectl debug -it api-xyz --image=busybox --share-processes
🌍
Multi-Environment (Namespaces)
USE CASE 4

Isolate dev / staging / prod in the same cluster using namespaces with different configs.

kubectl create namespace staging
kubectl apply -f manifests/ -n staging
# Each namespace gets its own Secrets, ConfigMaps, Services
kubectl get all -n staging
kubectl get all -n production
