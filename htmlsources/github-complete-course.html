Git & GitHub Basics
Version control concepts, Git internals, and getting started with GitHub

What is Git?

Git is a distributed version control system (DVCS) created by Linus Torvalds in 2005. It tracks changes in source code during software development, enabling multiple developers to collaborate on the same project without overwriting each other's work.

рҹ’Ў
Key Insight
Git stores snapshots, not diffs. Each commit is a complete picture of your project at a point in time, making operations like checkout and branching extremely fast.

Never Commit Secrets
Always add .env files and any file containing API keys, passwords, or tokens to .gitignore before your first commit. Once pushed to GitHub, secrets are permanent in history even if deleted later.

Merging vs Rebasing

merge vs rebase bash 
# в”Җв”Җ MERGE в”Җв”Җ (preserves history, creates merge commit)
git checkout main
git merge feature/login
git merge --no-ff feature/login     # always create merge commit
git merge --squash feature/login    # squash all commits into one

# в”Җв”Җ REBASE в”Җв”Җ (linear history, rewrites commits)
git checkout feature/login
git rebase main                     # replay commits on top of main
git rebase -i HEAD~3                # interactive: squash, reorder, edit

# Interactive rebase options:
# pick   = keep commit
# reword = keep, edit message
# squash = combine with previous
# drop   = remove commit

# Cherry pick a single commit
git cherry-pick abc1234
вҡ пёҸ
Golden Rule of Rebasing
Never rebase commits that have been pushed to a shared branch. Rebasing rewrites commit hashes, which will cause conflicts for other developers who have the old commits.

Pull Request Best Practices

  • Small PRs вҖ” under 400 lines of change when possible
  • One concern per PR вҖ” don't mix features and refactors
  • Write a good description вҖ” what, why, and how to test
  • Link related Issues вҖ” use Closes #123 in description to auto-close
  • Request specific reviewers вҖ” don't just ping the whole team
  • Respond to all comments вҖ” resolve or discuss every review comment

What is GitHub Actions?

GitHub Actions is a CI/CD (Continuous Integration/Continuous Deployment) platform built directly into GitHub. It lets you automate build, test, and deployment pipelines triggered by GitHub events.

GitHub Event вҶ’ Trigger вҶ’ Workflow вҶ’ Job(s) вҶ’ Step(s) вҶ’ Action/Command push to main в”Ӯ в–ј в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ .github/workflows/ci.yml в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ в”Ӯ в”Ӯ в”Ӯ Job: build Job: test Job: deploy в”Ӯ в”Ӯ в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ в”Ӯ в”Ӯ в”Ӯ Step: checkout в”Ӯ в”Ӯ Step: checkout в”Ӯ в”Ӯ needs: в”Ӯ в”Ӯ в”Ӯ в”Ӯ Step: setup в”Ӯ в”Җв”Җв”Җв–ә в”Ӯ Step: run testsв”Ӯв”Җв”Җв”Җв–әв”Ӯ [build, в”Ӯ в”Ӯ в”Ӯ в”Ӯ Step: build в”Ӯ в”Ӯ Step: report в”Ӯ в”Ӯ test] в”Ӯ в”Ӯ в”Ӯ в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ в”Ӯ в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ

Key Concepts

Event/TriggerGitHub event that starts a workflow (push, PR, schedule, etc.)
JobA set of steps that run on the same runner (runs in parallel by default)

Your First Workflow

.github/workflows/hello.yml yaml 
# Workflow name (appears in GitHub UI)
name: Hello World

# When to run this workflow
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

# Jobs to run
jobs:
  greet:                             # job ID
    name: Greet the World            # display name
    runs-on: ubuntu-latest           # runner OS

    steps:
      # Step 1: Check out source code
      - name: Checkout repository
        uses: actions/checkout@v4

      # Step 2: Print message
      - name: Say Hello
        run: |
          echo "Hello from GitHub Actions!"
          echo "Current branch: $GITHUB_REF"
          echo "Repository: $GITHUB_REPOSITORY"

      # Step 3: Use Node.js
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'

      - name: Run script
        run: node -e "console.log('Node.js works!')"

Complete Node.js CI Example

.github/workflows/ci.yml yaml 
name: CI Pipeline

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  build-and-test:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js 20
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'           # cache node_modules

      - name: Install dependencies
        run: npm ci               # clean install from lock file

      - name: Run linter
        run: npm run lint

      - name: Run tests
        run: npm test -- --coverage

      - name: Build project
        run: npm run build

      - name: Upload build artifact
        uses: actions/upload-artifact@v4
        with:
          name: dist
          path: dist/
          retention-days: 7

Jobs, Steps & Runners
Parallel vs sequential jobs, dependencies, environment setup, and runner types

Job Dependencies (needs)

multi-job pipeline yaml 
jobs:
  lint:                              # runs immediately
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm run lint

  test:                              # runs immediately (parallel to lint)
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm test

  build:
    runs-on: ubuntu-latest
    needs: [lint, test]              # waits for BOTH to succeed
    steps:
      - uses: actions/checkout@v4
      - run: npm run build

  deploy:
    runs-on: ubuntu-latest
    needs: build                    # waits for build
    if: github.ref == 'refs/heads/main'
    steps:
      - run: echo "Deploying to production..."

  # Access outputs from another job
  release:
    needs: build
    runs-on: ubuntu-latest
    steps:
      - run: echo "Version: ${{ needs.build.outputs.version }}"

What is OIDC?

OpenID Connect (OIDC) allows GitHub Actions workflows to authenticate with cloud providers (AWS, GCP, Azure) without storing long-lived credentials as secrets. Instead, GitHub issues a short-lived JWT token that the cloud provider validates.

GitHub Actions GitHub OIDC Provider AWS IAM / GCP / Azure в”Ӯ в”Ӯ в”Ӯ в”Ӯ Request JWT token в”Ӯ в”Ӯ в”Ӯв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–әв”Ӯ в”Ӯ в”Ӯ в”Ӯ в”Ӯ в”Ӯ Returns JWT (1 hour) в”Ӯ в”Ӯ в”Ӯв—„в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Ӯ в”Ӯ в”Ӯ в”Ӯ в”Ӯ Exchange JWT for short-lived cloud credentials в”Ӯ в”Ӯв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–әв”Ӯ в”Ӯ в”Ӯ в”Ӯ Returns temp access token (15 min вҖ“ 1 hour) в”Ӯ в”Ӯв—„в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Ӯ в”Ӯ в”Ӯ Use temp token to access cloud resources в”Ӯв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–ә

          { title: 'Commit', value: sha.substring(0, 7), short: true }
        ]
      }]
    };

    // Send to Slack
    const response = await fetch(webhookUrl, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify(payload)
    });

    if (!response.ok) throw new Error(`Slack error: ${response.status}`);

    // Set output
    core.setOutput('message-id', new Date().getTime().toString());
    core.info('Slack notification sent!');

  } catch (error) {
    core.setFailed(error.message);  // marks step as failed
  }
}

Workflow Security Checklist

1. Pin Action Versions to Full SHA

pinning actions yaml 
# вқҢ BAD вҖ” tag can be moved to a malicious commit

- uses: actions/checkout@v4

# вң… GOOD вҖ” pinned to immutable commit SHA

# Use a comment to track the version
- uses: actions/setup-node@39370e3970a6d050c480ffad4ff0ed4d3fdee5af  # v4.1.0

least privilege yaml 
# Set restrictive defaults at workflow level
permissions:
  contents: read    # default: no write unless needed

jobs:
  deploy:
    permissions:
      contents: write  # only this job gets write access

3. Never Trust User Input

injection prevention yaml

    PR_TITLE: ${{ github.event.pull_request.title }}

Auto-Release Workflow

auto-release.yml yaml 
name: Auto Release

on:
  push:
    tags: ['v*.*.*']

permissions:
  contents: write

jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0        # full history for changelog

      - name: Generate changelog
        id: changelog
        run: |
          PREV_TAG=$(git describe --tags --abbrev=0 HEAD~1 2>/dev/null || echo "")
          if [ -z "$PREV_TAG" ]; then
            CHANGES=$(git log --pretty=format:"- %s" HEAD)
          else
            CHANGES=$(git log --pretty=format:"- %s" $PREV_TAG..HEAD)
          fi
          echo "changes<> $GITHUB_OUTPUT
          echo "$CHANGES" >> $GITHUB_OUTPUT
          echo "EOF" >> $GITHUB_OUTPUT

      - name: Create GitHub Release
        uses: softprops/action-gh-release@v2
        with:
          name: Release ${{ github.ref_name }}
          body: |
            ## What's Changed
            ${{ steps.changelog.outputs.changes }}

            **Full Changelog**: https://github.com/${{ github.repository }}/compare/.../${{ github.ref_name }}
          generate_release_notes: true
          draft: false
          prerelease: ${{ contains(github.ref, 'alpha') || contains(github.ref, 'beta') }}
