01
Phase 1
Discovery & Ideation
Clarify the problem you're actually solving, who it's for, and whether it's worth building. Most failures happen because this phase was skipped.
  • 01
    Define the Problem Statement
    Write one clear sentence: "Users struggle to [X] because [Y], which causes [Z]." If you can't write this, you don't know what you're building yet.
  • 02
    Identify Your Target Users
    Create 1–3 user personas. Define their demographics, goals, pain points, and tech comfort level. Be specific — "everyone" is not a user.
  • 03
    Market & Competitor Research
    List existing solutions. Identify their gaps. Answer: why would a user choose your product over them? Your differentiation must be real and defensible.
  • 04
    Validate the Idea (Before Building)
    Talk to 5–10 potential users. Use surveys, landing pages, or fake-door tests. Confirm there is real demand before investing in development.
  • 05
    Define Success Metrics
    How will you know if this product succeeded? Choose 2–3 KPIs upfront: activation rate, DAU, NPS, revenue, churn. Make them measurable.
  • 06
    Decide on Business Model
    Free / freemium / subscription / one-time / marketplace / B2B / B2C? Nail this early — it affects your entire product strategy and architecture.
Deliverables
  • Problem Statement Doc
  • User Personas
  • Competitive Analysis
  • Validation Report
  • KPI Dashboard Plan
  • Business Model Canvas
02
Phase 2
Requirements & Scope
Translate the problem into concrete features, user stories, and a defined MVP. Scope creep kills projects — be ruthlessly specific here.
  • 07
    Write User Stories
    Format: "As a [user type], I want to [action] so that [benefit]." Write stories for every feature idea. These become your development tickets.
  • 08
    Prioritize with MoSCoW
    Sort features into: Must Have, Should Have, Could Have, Won't Have (this time). Fight to keep Must Have small. Your MVP must be truly minimal.
  • 09
    Define Functional Requirements
    What the system must DO: user registration, search, payment processing, file upload, notifications, etc. List every function needed for the MVP.
  • 10
    Define Non-Functional Requirements
    How the system must PERFORM: response time (<200ms), uptime (99.9%), concurrent users (10k), data security (SOC2/GDPR), accessibility (WCAG AA).
  • 11
    Create a Product Roadmap
    Break into phases: MVP → V1 → V2. Each phase should deliver standalone value. Set rough timelines. This is a living document — revisit it monthly.
  • 12
    Set Up Project Management
    Choose a tool (Jira, Linear, Notion, Trello). Create a backlog. Define sprints (1–2 weeks). Assign ownership. Define definition of done for every ticket.
Deliverables
  • User Stories Backlog
  • MoSCoW Prioritization
  • Functional Spec (FRD)
  • Non-Functional Spec
  • Product Roadmap
  • Sprint Board Setup
03
Phase 3
System Design & Architecture
Design the technical skeleton of your product. Bad architecture is the most expensive mistake you can make — it compounds forever.
  • 13
    Choose Architecture Pattern
    Monolith (start here for most MVPs), Microservices (when teams and scale demand it), Serverless (for event-driven low-traffic apps). Don't over-engineer early.
  • 14
    Select the Tech Stack
    Frontend (React / Vue / Next.js), Backend (Node / Python / Go / Java), Database (PostgreSQL / MongoDB / Redis), Cloud (AWS / GCP / Azure). Choose based on team expertise and use-case fit.
  • 15
    Design the Data Model
    Draw all entities (tables/collections), their fields, and relationships (one-to-many, many-to-many). Normalize for integrity, denormalize for performance where needed. Write migration strategy.
  • 16
    Define the API Contract
    Design REST or GraphQL endpoints. Write an OpenAPI / Swagger spec for every endpoint: method, path, request body, response schema, error codes. This is the contract between frontend and backend.
  • 17
    Plan Authentication & Authorization
    Choose auth strategy: JWT, OAuth2, session-based, SSO. Design RBAC (Role-Based Access Control). Define what each role can read, write, delete. Never skip this step.
  • 18
    Plan Infrastructure & DevOps
    Define environments: Local → Dev → Staging → Production. Choose containerization (Docker), orchestration (Kubernetes/ECS), CI/CD pipeline (GitHub Actions / GitLab CI). Plan from day one.
  • 19
    Plan Security Architecture
    HTTPS everywhere. Input validation and sanitization. Rate limiting. Secrets management (env vars, Vault). OWASP Top 10 checklist. Data encryption at rest and in transit.
  • 20
    Plan Scalability & Performance
    Caching strategy (Redis / CDN). Database indexing plan. Horizontal vs vertical scaling decisions. Message queues for async jobs (RabbitMQ / SQS). Load balancing approach.
Deliverables
  • Architecture Diagram (C4)
  • Tech Stack Decision Log
  • Entity Relationship Diagram
  • OpenAPI Spec
  • Auth & RBAC Design
  • Infrastructure Diagram
  • Security Checklist
04
Phase 4
Design & User Experience
Build the user interface — from information architecture to high-fidelity mockups and a design system. Users judge your product in milliseconds.
  • 21
    Create Information Architecture
    Map all screens and their hierarchy. Draw a site map / screen flow. Identify the critical path (the key journey every user must complete successfully).
  • 22
    Build Low-Fidelity Wireframes
    Sketch or use Figma/Balsamiq for rough wireframes of every screen. Focus on layout and functionality — not colors or fonts. Validate flow with stakeholders early.
  • 23
    Create Design System / UI Kit
    Define: color palette, typography scale, spacing system (4px grid), button states, form elements, icons, component library. This ensures consistency across the entire product.
  • 24
    Build High-Fidelity Mockups
    Apply the design system to all screens. Design for all states: loading, empty, error, success, mobile, desktop. Use real content — never lorem ipsum in hi-fi mocks.
  • 25
    Create Interactive Prototype
    Link mockups into a clickable prototype (Figma, InVision). Test the full user journey. Conduct user testing with 5 real people — watch where they get confused.
  • 26
    Accessibility & Responsiveness Plan
    Design for mobile-first. WCAG AA compliance: color contrast 4.5:1, keyboard navigation, ARIA labels, focus indicators. Accessibility is not optional.
Deliverables
  • Site Map
  • Wireframes (all screens)
  • Design System
  • Hi-Fi Mockups
  • Clickable Prototype
  • User Test Report
05
Phase 5
Development
Build the actual product. Work iteratively in short sprints. Ship working features, not perfect features. Done beats perfect for an MVP.
  • 27
    Set Up Dev Environment & Repo
    Initialize Git repo. Define branching strategy (GitFlow or trunk-based). Set up linting, formatting (ESLint, Prettier), pre-commit hooks, README with setup instructions.
  • 28
    Build the Database Layer
    Implement the data model with migrations. Write seeds for development data. Set up connection pooling. Test all CRUD operations. Document every table and column.
  • 29
    Build Backend / API
    Implement endpoints per your API spec. Add middleware: auth, rate limiting, logging, error handling. Write business logic. Return consistent error responses with proper HTTP codes.
  • 30
    Implement Authentication
    Build signup, login, logout, password reset, email verification. Use a battle-tested library (Passport.js, Auth.js, Supabase Auth). Never roll your own crypto.
  • 31
    Build Frontend / UI
    Implement components from the design system. Connect to the API. Handle all states: loading, error, empty, success. Write accessible markup with semantic HTML.
  • 32
    Integrate Third-Party Services
    Payments (Stripe), Email (SendGrid/Resend), Storage (S3), Analytics (PostHog/Amplitude), Monitoring (Sentry), Search (Algolia). Use official SDKs. Handle API failures gracefully.
  • 33
    Set Up CI/CD Pipeline
    Automate: lint → test → build → deploy. Every PR must pass all checks before merge. Deploy to staging automatically on merge to main. Never manually deploy to production.
  • 34
    Code Review Process
    Require at least one approving review per PR. Write a PR template. Check for: correctness, security, test coverage, performance, and readability. Automate what you can.
Deliverables
  • Git Repo + Branching Rules
  • Database Migrations
  • REST/GraphQL API
  • Auth System
  • Frontend App
  • CI/CD Pipeline
  • Integration Tests
06
Phase 6
Testing & Quality Assurance
Catch bugs before your users do. Testing is not optional — untested code is broken code you haven't found yet.
  • 35
    Unit Testing
    Test every pure function and business logic unit in isolation. Target 70–80% code coverage on critical paths. Use Jest, Vitest, PyTest, JUnit depending on stack.
  • 36
    Integration Testing
    Test that services work together: API + database, auth flow, payment flow. Use a test database. Test happy paths and all error branches.
  • 37
    End-to-End Testing
    Automate critical user journeys in a real browser: signup → onboarding → core action → payment. Use Playwright or Cypress. These are your safety net before every deploy.
  • 38
    Performance Testing
    Load test with realistic traffic (k6, Locust). Measure p50, p95, p99 response times. Find your breaking point before your users do. Optimize database queries with EXPLAIN ANALYZE.
  • 39
    Security Testing
    Run OWASP ZAP scan. Check: SQL injection, XSS, CSRF, auth bypass, insecure direct object references, broken access control. Rotate all secrets before production.
  • 40
    User Acceptance Testing (UAT)
    Have real users test on staging. Provide test scenarios matching user stories. Log all feedback. Fix critical issues before launch — this is your last chance.
Deliverables
  • Unit Test Suite (70%+ coverage)
  • Integration Test Suite
  • E2E Test Suite
  • Load Test Report
  • Security Audit Report
  • UAT Sign-Off
07
Phase 7
Deployment & Launch
Get the product in front of real users. Plan for things to go wrong. A good deployment is boring — that's the goal.
  • 41
    Set Up Production Infrastructure
    Provision servers / cloud resources. Configure DNS, SSL certificates (Let's Encrypt), CDN. Set up load balancer. Configure environment variables securely — never commit secrets.
  • 42
    Set Up Monitoring & Alerting
    Application monitoring (Sentry for errors), infrastructure monitoring (Datadog / CloudWatch), uptime monitoring (Better Uptime). Set alerts for error spikes, high latency, and downtime.
  • 43
    Set Up Logging
    Structured JSON logs for every request, error, and important business event. Use a centralized log platform (Datadog, Logtail, AWS CloudWatch Logs). Log enough to debug any incident.
  • 44
    Create a Deployment Runbook
    Document the exact steps to deploy and to rollback. Include: pre-deploy checklist, deploy commands, post-deploy verification tests, rollback procedure. Practice the rollback before you need it.
  • 45
    Database Backup Strategy
    Automated daily backups to a separate region. Point-in-time recovery enabled. Test your restore procedure — a backup you haven't tested is not a backup.
  • 46
    Soft Launch / Beta
    Release to a small group first (friends, waitlist, internal users). Collect feedback. Fix critical issues. Validate that everything works at real traffic. Then announce publicly.
  • 47
    Public Launch
    Announce across channels: Product Hunt, social media, email list, communities. Have a war room for the first 24 hours. Monitor dashboards in real time. Have team on standby.
Deliverables
  • Production Environment
  • Monitoring Dashboard
  • Alerting Rules
  • Deployment Runbook
  • Backup & Recovery Plan
  • Launch Checklist
08
Phase 8
Post-Launch & Iteration
Launch is not the end — it's the beginning. Now you learn what users actually want and build toward it. This phase never ends.
  • 48
    Measure Your KPIs
    Are your success metrics moving? Check weekly: activation, retention, conversion, NPS. Use product analytics (PostHog, Mixpanel, Amplitude) with funnel tracking set up from day one.
  • 49
    Collect User Feedback
    In-app surveys (Typeform, Hotjar), user interviews, support tickets, NPS surveys, session recordings. The most valuable feedback is often in support tickets — read every one.
  • 50
    Fix Bugs & Stability
    P0 bugs (product broken) → fix within hours. P1 (major feature broken) → fix within 24h. P2 (minor issues) → fix within sprint. Stability earns user trust more than features do.
  • 51
    Prioritize Next Features
    Use the ICE framework: Impact × Confidence ÷ Effort. Build features with the highest score first. Resist the temptation to build everything users ask for — find the underlying need.
  • 52
    Performance Optimization
    Audit Core Web Vitals (LCP, CLS, FID). Optimize slow database queries. Add caching where needed. Reduce bundle size. Performance is a feature — users notice slowness.
  • 53
    Technical Debt Management
    Reserve 20% of each sprint for refactoring and debt reduction. Document known debt in a tech debt register. Unmanaged debt compounds — eventually it stops all feature development.
  • 54
    Plan for Scale
    At 10x current users, what breaks first? Identify bottlenecks proactively. Consider database read replicas, horizontal scaling, CDN expansion, async job queues before you need them.
Deliverables
  • KPI Weekly Report
  • User Feedback Log
  • Bug Priority Queue
  • ICE-Scored Feature Backlog
  • Performance Audit
  • Tech Debt Register
Complete System Design
The full technical blueprint you need to think through before writing code.
🏗️
Frontend Layer
Single Page App or Server-Side Rendered web app. Mobile app if required.
  • React / Next.js / Vue
  • Component library (Shadcn, MUI)
  • State management (Zustand, Redux)
  • API client (TanStack Query, SWR)
  • Deployed via CDN / Vercel / Netlify
⚙️
API Gateway / BFF
Entry point for all client requests. Handles routing, auth, rate limiting.
  • REST or GraphQL API
  • JWT / OAuth2 token validation
  • Rate limiting per IP / user
  • Request logging & tracing
  • API versioning strategy
🧠
Application / Business Logic
Where your product's actual value lives. Stateless, horizontally scalable.
  • Node.js / Python / Go / Java
  • Service layer (business rules)
  • Repository pattern (data access)
  • Event emitters for side effects
  • Background workers (Bull, Celery)
🗄️
Data Layer
Persistent storage for all application data. The most important design decision you'll make.
  • Primary DB: PostgreSQL / MySQL
  • Cache: Redis (sessions, hot data)
  • Search: Elasticsearch / Algolia
  • File storage: S3 / GCS
  • Read replicas for scale
📨
Async / Event System
Decouple services. Handle long-running tasks without blocking the request.
  • Message queue: SQS / RabbitMQ
  • Email: SendGrid / Resend
  • Push notifications: FCM / APNs
  • Webhooks for external systems
  • Scheduled jobs (cron)
🔐
Auth & Security
Every entry point must be authenticated and authorized.
  • Auth provider (Auth0, Supabase)
  • JWT with short expiry + refresh tokens
  • RBAC with row-level security
  • Secrets in Vault / AWS Secrets Manager
  • HTTPS + HSTS everywhere
☁️
Infrastructure
Where everything runs. Infrastructure as Code from day one.
  • Cloud: AWS / GCP / Azure
  • Containers: Docker + Kubernetes/ECS
  • IaC: Terraform / Pulumi
  • Multi-AZ for high availability
  • CDN: CloudFront / Cloudflare
📊
Observability
You can't fix what you can't see. Full-stack visibility from the start.
  • Error tracking: Sentry
  • Metrics: Datadog / Grafana
  • Logs: Logtail / CloudWatch
  • Traces: OpenTelemetry
  • Uptime: Better Uptime / Pingdom
🚀
CI/CD Pipeline
Automated, repeatable delivery. Every commit should be deployable.
  • Source: GitHub / GitLab
  • CI: GitHub Actions / GitLab CI
  • Lint → Test → Build → Push image
  • Auto-deploy to staging on merge
  • Manual gate for production
