Requirements Gathering

A task manager is complex — far more than CRUD. Clarify scope with the interviewer: real-time collaboration? Offline support? Mobile? Here we design a cloud-first web platform at Jira-scale.

● Functional — Core Task Management

Create/update/delete projects, epics, sprints, issues, subtasks
Issue types: Story, Bug, Task, Epic, Sub-task
Issue fields: title, description (rich text), assignee, reporter, priority, status, labels, due date, story points
Kanban board view (drag-and-drop columns) and Backlog view
Sprint planning: create sprint, move issues, start/close sprint
Comments (threaded), @mentions, reactions, edit history
File attachments (images, PDFs, docs) up to 25MB per file
Issue linking (blocks, is-blocked-by, relates-to, duplicates)
Activity feed / audit trail per issue and per project

◆ Functional — Collaboration & Notifications

Real-time updates: when an issue changes, all viewers see it instantly
@mention triggers immediate in-app + email notification
Watch/unwatch issues for change notifications
Notification channels: In-app, Email, Slack, Mobile Push
Notification preferences per user (mute, digest mode, channel selection)
Daily/weekly email digest of watched issues
Full-text search across issue titles, descriptions, comments
Role-based permissions: Admin, Project Lead, Developer, Viewer

Lexorank for ordering: Jira uses the Lexorank algorithm for drag-and-drop ordering. Instead of integer positions (which require re-numbering all rows on reorder), Lexorank uses strings that can always be inserted between any two values. Example: between "aaa" and "bbb" insert "aaU" — no other rows change. This turns reorders into single-row updates.
Sharding Strategy: Shard PostgreSQL by workspace_id. All data for a workspace (projects, issues, comments, members) lives on the same shard — no cross-shard JOINs needed. Large workspaces (enterprise) get dedicated shards. Use Citus or application-level sharding via consistent hashing.

System Architecture

Microservice-oriented architecture with domain-separated services. Services communicate via REST (sync) and Kafka (async). WebSocket Gateway handles all real-time connections centrally.

Tier 1 — Client Layer
🌐 Web App (React SPA)

Board, backlog, issue detail views. WebSocket client. Optimistic UI updates.

📱 Mobile (React Native)

Consumes same REST API. Push notifications via FCM/APNs.

🤖 Integrations

Slack bot, GitHub Actions, Zapier webhooks consuming API.

↓ HTTPS / WSS via CDN + WAF + API Gateway
Tier 2 — Edge & Gateway
☁️ CDN (CloudFront)

Serves static assets (JS/CSS). Caches public GET responses. Edge termination.

🛡️ WAF + Rate Limiter

OWASP rules. Per-user API rate limits via Redis token bucket. DDoS protection.

🚪 API Gateway (Kong)

JWT auth validation. Routes requests to services. Request logging. Plugin ecosystem.

↓ Internal routing to microservices
Tier 3 — Application Services (Stateless, K8s Pods)
📋 Issue Service

CRUD for issues, subtasks, comments, attachments. Publishes events to Kafka on every mutation.

🗂️ Project Service

Projects, sprints, board config, member management, RBAC enforcement.

🔌 WebSocket Gateway

Manages 50K+ persistent WS connections. Subscribes to Redis Pub/Sub for event fan-out to clients.

🔔 Notification Service

Consumes Kafka events. Resolves watchers. Routes to channels (email, push, Slack, in-app).

🔍 Search Service

Wraps Elasticsearch. Handles query parsing, filtering, highlighting. Sync from Kafka consumer.

👤 Auth Service

JWT issue/refresh, OAuth2 (Google, GitHub SSO), session management, invite flows.

↓ Async via Kafka / Sync via service calls
Tier 4 — Messaging & Event Bus
📨 Apache Kafka

Topics: issue.events, notification.events, search.sync, audit.log. 50+ partitions each. 7-day retention.

📡 Redis Pub/Sub

WebSocket event routing. WS Gateway subscribes to project channels. Real-time presence (who's online).

⏰ Job Scheduler

Digest emails, sprint reminders, due-date alerts. BullMQ (Redis) + cron workers.

↓ Persistence layer
Tier 5 — Data Storage
🐘 PostgreSQL (Sharded)

Primary store. Issues, comments, projects, users. Sharded by workspace_id via Citus.

⚡ Redis Cluster

Board state cache, sessions, rate limiting, presence tracking, WS pub/sub channels.

🔍 Elasticsearch

Full-text search index. Synced from Kafka. 3-node cluster, 1 replica per shard.

🗃️ S3 (Object Store)

File attachments, avatar images. Signed URL upload/download. CDN fronted for reads.

📊 ClickHouse

Analytics: sprint velocity, issue throughput, cycle time reports. Async ingest from Kafka.

Request Flows

Three critical flows: updating an issue (most common write), loading a board (most common read), and creating a comment with @mention (triggers notification pipeline).

🔄 Issue Status Update

1
User drags card on board. Client sends PATCH /issues/:id with new status + optimistic UI update shown immediately.
2
API Gateway validates JWT. Routes to Issue Service. Checks RBAC permission (can user edit in this project?).
3
Issue Service runs optimistic lock check: WHERE version = request.version. If version mismatch → 409 Conflict (someone else edited).
4
Write to PostgreSQL, increment version. Write activity log entry. Invalidate Redis board cache for this project.
5
Publish issue.updated event to Kafka. Return 200 to client immediately — rest is async.
6
Kafka consumers: (a) Notification Service fans out to watchers, (b) Search Service updates Elasticsearch index, (c) WebSocket Gateway pushes real-time event to all board viewers.

📋 Board Load

1
User opens project board. Client sends GET /projects/:pid/board + opens WebSocket connection wss://.../projects/:pid.
2
Project Service checks Redis for cached board state: Redis GET board:{project_id}. Cache TTL = 60s, invalidated on any change.
3
Cache HIT (most cases): Return serialized board JSON instantly. <20ms response.
4
Cache MISS: Query PostgreSQL for all active sprint issues, group by status, include assignee names/avatars. ~50–150ms.
5
Write board state to Redis cache. Return to client. Client registers WebSocket listeners for live updates.
6
All subsequent updates arrive via WebSocket — client patches local state without re-fetching the entire board.

💬 Comment with @mention → Notification Pipeline

1
User submits comment: "@Alice can you review this bug?" POST /issues/:id/comments
2
Issue Service parses @mentions from comment body. Resolves display names to user_ids.
3
Saves comment to PostgreSQL. Publishes comment.created event to Kafka with mention list.
4
WebSocket Gateway receives from Redis Pub/Sub, pushes comment to all board viewers instantly (<100ms).
5
Notification Service consumes event. Loads watcher list + mentioned users. Deduplicates. Checks user preferences.
6
Creates in-app notification records in PostgreSQL for each recipient. Pushes to user's WS channel for real-time bell badge.
7
Routes to Email / Push / Slack channels based on user preferences. Email via SES, Push via FCM, Slack via webhooks.
8
Failed deliveries queued for retry with exponential backoff. Dead-letter queue for permanent failures. Delivery receipt tracked.

Real-time Updates

Real-time collaboration is a core differentiator. We use WebSockets (not polling) for push-based updates. The challenge: routing events from any API server to the correct WebSocket connection on any WS server.

websocket_gateway.py
# WebSocket Gateway — manages all client connections # Problem: API servers and WS servers are different pods. # Solution: Redis Pub/Sub as the event bus between them. class WebSocketGateway: def __init__(self): self.connections: dict[str, set[WebSocket]] = {} # project_id → connections self.user_connections: dict[str, set[WebSocket]] = {} # user_id → connections self.redis_sub = redis.pubsub() async def connect(self, ws: WebSocket, project_id: str, user_id: str): await ws.accept() self.connections.setdefault(project_id, set()).add(ws) self.user_connections.setdefault(user_id, set()).add(ws) # Subscribe to project channel in Redis Pub/Sub await self.redis_sub.subscribe(f"project:{project_id}") # Broadcast user presence to project members await self.publish_presence(project_id, user_id, "online") async def broadcast_to_project(self, project_id: str, event: dict): # API server publishes to Redis; ALL WS gateway pods receive it await redis.publish(f"project:{project_id}", json.dumps(event)) async def on_redis_message(self, message): channel = message['channel'] # "project:uuid" project_id = channel.split(':')[1] event = json.loads(message['data']) # Fan-out to all WS connections watching this project (on THIS pod) dead = set() for ws in self.connections.get(project_id, set()): try: await ws.send_json(event) except ConnectionClosed: dead.add(ws) self.connections[project_id] -= dead # cleanup dead connections # ── Event types pushed to clients ───────────────────────────────────── EVENT_TYPES = { "issue.updated": "issue_id, changed_fields, updated_by, version", "issue.created": "issue_id, issue_data, created_by", "comment.created": "comment_id, issue_id, body, author", "member.online": "user_id, display_name, avatar_url", "member.offline": "user_id", "notification.new": "notif_id, type, payload, unread_count", }
Horizontal Scaling of WebSocket Servers: Each WS Gateway pod independently manages its connections. Redis Pub/Sub is the glue — when Issue Service publishes project:uuid, every WS pod subscribed to that channel receives the message and fans out to its local connections. This scales linearly: add WS pods as needed. Redis Cluster handles the pub/sub load.

Notification System

The notification system is the most complex subsystem. It must handle high fan-out (one event → many users), multiple delivery channels, user preferences, deduplication, retry logic, and digest batching.

🔔

In-App

Stored in PostgreSQL. Real-time delivery via WebSocket push to active clients. Unread badge count updated live. Persistent inbox.

📧

Email

Sent via AWS SES / SendGrid. Immediate for @mentions, digest (daily/weekly) for low-priority events. Unsubscribe link in footer. GDPR compliant.

📱

Mobile Push

FCM (Android) + APNs (iOS). Token stored per device. Supports notification grouping. Respects quiet hours in user preferences.

💬

Slack

OAuth integration. Workspace-level bot posts to DM or channel. Rich formatting with issue title, status, assignee. User can /connect-slack to link account.

notification_service.py
# ── Notification Service: Consumer + Fan-out ─────────────────────── NOTIFICATION_TRIGGERS = { "issue.assigned": {"recipients": ["assignee"], "priority": "high"}, "issue.commented": {"recipients": ["watchers", "reporter"], "priority": "medium"}, "issue.mentioned": {"recipients": ["mentioned_users"], "priority": "high"}, "issue.status_changed":{"recipients": ["watchers"], "priority": "low"}, "sprint.started": {"recipients": ["project_members"], "priority": "medium"}, "due_date.approaching":{"recipients": ["assignee", "reporter"], "priority": "high"}, } async def process_event(event: dict): trigger = NOTIFICATION_TRIGGERS.get(event["type"]) if not trigger: return # 1. Resolve recipient list recipients = await resolve_recipients(event, trigger["recipients"]) recipients = [r for r in recipients if r != event["actor_id"]] # don't notify self for user_id in recipients: prefs = await get_user_prefs(user_id) # 2. Deduplication: skip if same notif delivered <5min ago dedup_key = f"notif:{user_id}:{event['type']}:{event['entity_id']}" if await redis.exists(dedup_key): continue await redis.setex(dedup_key, 300, 1) # 3. Save in-app notification (always) notif = await db.create_notification(user_id, event) # 4. Push via WebSocket if user is online await redis.publish(f"user:{user_id}:notifs", notif.to_json()) # 5. Route to external channels based on preferences + priority if prefs.email_enabled and trigger["priority"] == "high": await email_queue.enqueue(user_id, notif) # SES elif prefs.digest_mode: await digest_queue.add(user_id, notif) # batch for daily email if prefs.push_enabled and not is_quiet_hours(user_id): await push_queue.enqueue(user_id, notif) # FCM/APNs if prefs.slack_token: await slack_queue.enqueue(prefs.slack_token, notif) # Slack API # ── Retry Logic ─────────────────────────────────────────────────── # Each channel queue uses exponential backoff: 1s → 5s → 30s → 5min → give up # Failed deliveries after 5 retries → Dead Letter Queue (DLQ) for investigation # Email bounces: update user's email_deliverable=false, alert admin

● Fan-out Strategy

Push model (fan-out on write): When an event occurs, immediately write notification to each recipient's inbox. Fast reads, higher write cost. Works well for moderate watcher counts (<1000).
Pull model (fan-out on read): Store one event, each user fetches their feed. Low write cost, but expensive reads and harder to personalize. Use for large project broadcasts (sprint start).
Hybrid: Push for <500 watchers (most issues), pull for >500 (large projects / org-wide announcements).

◆ Digest Batching

Users who prefer digest mode: accumulate notifications in Redis sorted set (score = timestamp).
Cron job runs daily at user's preferred time (respects timezone from user profile).
Aggregates: "5 issues were updated in Project X, 2 @mentions waiting for you."
Renders HTML email from template, sends via SES. Clears digest queue after send.
Due date alerts run 24h before due: cron scans issues WHERE due_date = tomorrow AND status != done.

Full-Text Search

PostgreSQL LIKE queries don't scale to 500M issues. Elasticsearch provides sub-200ms full-text search with relevance ranking, facets, and typo-tolerance. The challenge is keeping it in sync with PostgreSQL.

search_index_mapping.json
// Elasticsearch index mapping for issues { "mappings": { "properties": { "issue_id": { "type": "keyword" }, "workspace_id": { "type": "keyword" }, // for tenant isolation "project_id": { "type": "keyword" }, "issue_key": { "type": "keyword" }, // PROJ-123 "title": { "type": "text", "analyzer": "english", "fields": { "keyword": { "type": "keyword" } } // for exact match + sort }, "description_text": { "type": "text", "analyzer": "english" }, "comment_bodies": { "type": "text", "analyzer": "english" }, // denormalized "status": { "type": "keyword" }, "priority": { "type": "keyword" }, "assignee_id": { "type": "keyword" }, "labels": { "type": "keyword" }, "updated_at": { "type": "date" } } } } // Sync strategy: Kafka consumer listens to issue.events topic // On issue.created / issue.updated: upsert Elasticsearch doc // On issue.deleted: delete from index // Lag: ~5-30 seconds (eventual consistency — acceptable for search) // Initial index build: pg_logical replication slot → bulk index
Search Security (Tenant Isolation): Every Elasticsearch query MUST include a workspace_id filter term. The Search Service enforces this — it injects the workspace from the JWT token, so users can never search another tenant's data even if they construct malicious queries. Use Elasticsearch aliases per workspace for strict isolation at scale.

# Permission check middleware — runs on every authenticated request PERMISSION_MAP = { ("PATCH", "/issues/:id"): ["developer", "lead", "admin"], ("DELETE", "/issues/:id"): ["lead", "admin"], ("POST", "/sprints/:id/start"): ["lead", "admin"], ("DELETE", "/comments/:id"): ["own_comment", "lead", "admin"], } async def check_permission(user_id: str, project_id: str, action: str) -> bool: # Cache key: permission:{user_id}:{project_id} → role (TTL 5min) cache_key = f"perm:{user_id}:{project_id}" role = await redis.get(cache_key) if not role: # Fetch from DB: workspace_members or project_members table role = await db.get_user_role(user_id, project_id) await redis.setex(cache_key, 300, role) # cache 5 minutes allowed_roles = PERMISSION_MAP.get(action, []) return role in allowed_roles # Invalidate cache when role changes (e.g., user promoted to lead) async def on_role_change(user_id: str, project_id: str): await redis.delete(f"perm:{user_id}:{project_id}")

Caching Strategy

Unlike URL shorteners, task managers have more complex cache invalidation — issues change frequently and board views are composite objects. We cache strategically, not aggressively.

● What We Cache (Redis)

Board state: board:{project_id} — full Kanban view, TTL 60s, invalidated on any issue change. Saves complex JOIN queries on every page load.
User sessions: session:{token} — JWT session data. TTL = token expiry. Avoids DB lookup on every request.
User permissions: perm:{user_id}:{project_id} — role lookup. TTL 5min. Invalidated on role change.
Unread notification count: unread:{user_id} — integer counter. Increment on new notif, reset on read-all.
Issue detail: issue:{issue_id} — denormalized view with assignee info. TTL 30s. Invalidated on update.

◆ What We DON'T Cache

Comments: Always fetched fresh — comments must be real-time accurate to avoid confusion in discussions.
Activity logs: Append-only, no caching needed — each view fetches the latest tail.
Search results: Elasticsearch handles its own caching. We don't double-cache in Redis.
User profile data: Changes rarely — rely on CDN caching of avatar images; profile data fetched once per session.
Notification inbox: Too personalized and frequently updated — fetch from PostgreSQL with proper pagination index.

Scaling & High Availability

Every component designed for horizontal scale and zero-downtime deployments. Multi-region active-passive with automated failover.

🐘

PostgreSQL HA

Primary + 2 read replicas. Reads distributed across replicas. Failover via Patroni (auto-promotes replica in <30s). Point-in-time recovery (PITR) via WAL archiving to S3. Backup every 6 hours.

⚡

Redis HA

Redis Cluster (3 primaries + 3 replicas). Automatic failover via Redis Sentinel. Data persistence: RDB snapshots + AOF. Separate Redis instances for cache vs Pub/Sub vs sessions.

📦

Stateless Services

All services (Issue, Project, Notification, Search) are stateless. K8s Deployment with HPA — auto-scale on CPU/RPS. Rolling updates with zero downtime. Health checks every 10s.

🔌

WebSocket Scaling

WebSocket pods scaled by active connection count. Redis Pub/Sub bridges pods. Sticky sessions NOT needed (any pod can serve any connection via Pub/Sub). 50K concurrent connections on 5 pods.

📨

Kafka HA

3-broker Kafka cluster. Replication factor 3. Min ISR = 2 (guarantees no data loss on 1 broker failure). Consumer groups allow independent scaling of Notification, Search, and Analytics consumers.

🔍

Elasticsearch HA

3-node cluster. 1 replica per shard. Master elections via zen discovery. Index aliases for zero-downtime re-indexing. Curator for index lifecycle management (archive old data).

🗃️

S3 Durability

11 nines durability (AWS S3). Cross-region replication enabled. Lifecycle policies: move attachments older than 1 year to Glacier. Pre-signed URLs expire in 1 hour for security.

🌍

Multi-Region

Primary region (US-East), DR region (EU-West). PostgreSQL WAL streaming replication to DR. RTO <15min, RPO <1min. Traffic failover via Route53 health checks + weighted routing.

📊

Observability

Prometheus + Grafana (metrics). Jaeger (distributed tracing — trace every request across services). ELK stack (centralized logs). PagerDuty alerts for P1/P2 incidents. SLO dashboards.

Bottlenecks & Mitigations

Anticipating failure modes and having clear mitigations demonstrates production thinking. These are the highest-risk bottlenecks for this system at scale.

BottleneckSeverityRoot Cause & SymptomMitigation
Notification fan-out spike HIGH Sprint start or @everyone mention in large project (10K watchers) triggers 10K simultaneous DB writes and 10K Slack API calls. Notification Service gets overwhelmed. Fan-out via Kafka with consumer concurrency throttling. Slack calls via rate-limited BullMQ queue. Max 100 fan-out writes/sec per event — queue the rest. Use push/pull hybrid for projects >500 members.
PostgreSQL write hotspot HIGH All teams in a large workspace write to the same DB shard during sprint ceremonies (10am standup spike). Single shard becomes the bottleneck. Shard by workspace_id. Large enterprise customers get dedicated shards. Use PgBouncer connection pooling to cap DB connections. Read replicas absorb board load queries.
Board cache stampede MED Board cache expires and 50 users reload simultaneously — all miss Redis and hit PostgreSQL with complex JOIN queries at the same moment. Use probabilistic early expiration (refresh cache at 90% of TTL with jitter). Redis lock-based population: first miss acquires lock and populates; others wait briefly then read.
WebSocket connection limit MED Each WS connection is a persistent TCP connection. OS limits (~65K FDs per process), memory (~50MB per pod for 5K connections). Sudden spike in users opening boards. Each WS pod handles ~5K connections. HPA auto-scales pods on connection count metric. Implement connection graceful degradation: fall back to SSE (Server-Sent Events) or 30s polling if WS unavailable.
Elasticsearch index lag MED Kafka consumer falls behind during heavy write bursts. Search results are stale by 5+ minutes. Users see recently created issues missing from search. Increase Elasticsearch consumer parallelism (more partitions, more consumers). Monitor consumer lag (Kafka offsets). Fallback search: if Elasticsearch lag >2min, route simple queries to PostgreSQL LIKE query with GIN index.
Email delivery backlog LOW SES rate limits or Slack API rate limits cause notification queue to back up. Users receive email notifications hours late. SES sending limits are soft quotas — request increase proactively. Per-channel queues are isolated (Slack backlog doesn't affect email). Exponential backoff + dead letter queue. Show in-app notification immediately regardless.
Rich-text description size LOW Users paste large documents or images into issue descriptions (JSONB). Single issue row becomes 5MB+, bloating cache and slowing serialization. Cap description JSONB at 500KB server-side. Images in descriptions → uploaded to S3, stored as reference URLs. Description stored separately in its own table with lazy loading (not included in board list queries).

Architecture Summary

Every design decision maps back to core constraints: collaborative writes, real-time updates, complex notification routing, and multi-tenant data isolation at 10M-user scale.

🐘

Primary DB

PostgreSQL sharded by workspace_id. ACID for issue state machine and sprint ops.

⚡

Cache

Redis for board state, sessions, permissions. Strategic caching, not aggressive.

🔌

Real-time

WebSocket Gateway + Redis Pub/Sub. Any pod can serve any user's connection.

📨

Events

Kafka decouples writes from notifications, search sync, analytics, and audit.

🔔

Notifications

Multi-channel fan-out with user preferences, dedup, retry, and digest batching.

🔍

Search

Elasticsearch for full-text. Async sync via Kafka. Tenant isolation enforced in query.

🛡️

RBAC

Workspace + project roles. Permission check cached 5min in Redis. Invalidated on change.

🌍

Multi-region

Primary + DR region. WAL streaming. Route53 health-check failover. RTO <15min.
