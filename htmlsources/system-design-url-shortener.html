Requirements Gathering

Always start by clarifying requirements before jumping into design. Ask the interviewer about scale, features, and constraints. Here we define both functional and non-functional requirements.

● Functional Requirements

Given a long URL, generate a unique short URL (alias)
When user visits short URL, redirect to original long URL
Users can optionally specify a custom short alias
Short URLs expire after a configurable TTL (default: 5 years)
Users can delete their own short URLs
Track click analytics (count, geography, referrer, device)

◆ Non-Functional Requirements

Redirect latency must be < 10ms (P99)
System must be highly available — 99.99% uptime
Eventual consistency acceptable for analytics
Short URLs must be unpredictable (not enumerable)
Support 100M URL creations and 10B redirects per day
Horizontal scalability with no single point of failure

◈ Capacity Assumptions

Read-to-write ratio: 100:1 (reads dominate heavily)
Average long URL size: 200 bytes
Short URL key: 7 characters = 62⁷ ≈ 3.5 trillion combinations
URLs stored for 5 years: ~180 billion total records
Data storage per URL entry: ~500 bytes
Total storage: ~90 TB over 5 years

◉ Out of Scope (v1)

User accounts and authentication (simplified to API key)
A/B testing and link rotation
QR code generation (can be added later)
URL preview / screenshot generation
Bulk URL creation API
Dashboard UI (assumed external)

API Design

RESTful API endpoints with clear contracts. Authentication via API key in header. Rate limiting applied per API key to prevent abuse.

POST
/api/v1/urls
Create a short URL. Body: long_url, optional custom_alias, optional expire_at. Returns short_url, short_key, expires_at.
201 Created
400 Bad Request
409 Conflict
GET
/:short_key
Resolve and redirect. Returns HTTP 301 (permanent, cached by browser) or 302 (temporary, bypasses browser cache — use 302 for analytics accuracy). Decision: use 302 to ensure every request hits our servers for tracking.
302 Redirect
404 Not Found
410 Gone (expired)
GET
/api/v1/urls/:short_key
Get metadata for a URL — original URL, creation time, expiry, click count. Requires ownership via API key.
200 OK
403 Forbidden
404 Not Found
DEL
/api/v1/urls/:short_key
Delete a short URL. Requires ownership. Soft-deletes the record and invalidates the cache entry. Future requests return 410 Gone.
204 No Content
403 Forbidden
404 Not Found
GET
/api/v1/urls/:short_key/analytics
Click analytics: total clicks, breakdown by date, country, referrer, browser, OS. Data is eventually consistent (up to 30s lag from async pipeline).
200 OK
403 Forbidden
404 Not Found
301 vs 302 Redirect: Use 302 Found even though 301 is more "semantically correct" for permanent redirects. Why? With 301, browsers cache the redirect and never hit our servers again — we lose analytics data. With 302, every redirect hits our servers, enabling accurate tracking.

Short URL Algorithm

The core problem: generate a unique 7-character key for each URL. The choice of algorithm significantly impacts uniqueness guarantees, collision risk, and system complexity.

Base62 Encoding

★ Recommended

Use a distributed counter (or Snowflake ID) and encode the integer in Base62 [a-z A-Z 0-9]. Counter increments atomically; encode gives a short, predictable-length key.

Pros
  • No collisions guaranteed
  • Predictable key length
  • Fast encoding O(1)
    • Cons
  • Sequential = guessable
  • Needs distributed counter
  • Counter = SPOF if naive

    • MD5 / SHA Hash

    Acceptable

    Hash the long URL, take first 7 chars of hex. Simple but collision risk: birthday paradox kicks in at scale. Duplicate long URLs produce the same hash — good for dedup, bad for tracking.

    Pros
  • No counter needed
  • Same URL → same key
  • Simple to implement
    • Cons
  • Collision risk at scale
  • Requires collision check
  • Can't deduplicate owners

    • Pre-generated Key Pool

    ★ Recommended

    A Key Generation Service (KGS) pre-generates millions of unique keys offline and stores them in a DB (key_pool). API servers claim keys as needed. Eliminates real-time generation latency.

    Pros
  • Fastest at claim time
  • Zero collision risk
  • Decouples generation
    • Cons
  • KGS needs HA setup
  • Wasted keys if unused
  • Extra DB table needed

    • Snowflake-style ID

    Acceptable

    64-bit ID composed of timestamp + datacenter ID + worker ID + sequence. Globally unique without coordination. Encode in Base62 to get a short string. Twitter/Discord use this pattern.

    Pros
  • No central counter
  • Time-sortable keys
  • Works across DCs
    • Cons
  • Clock skew issues
  • Key reveals timestamp
  • Slightly longer key
    • Recommended Approach: Key Generation Service (KGS)
    key_generation_service.py
    # ─── Key Generation Service ─────────────────────────────────────── # Runs as a background job. Generates keys in batches, marks as unused. # API servers pull from the pool atomically (row-level lock or Redis SPOP). import random, string CHARS = string.ascii_letters + string.digits # Base62: 62 chars KEY_LENGTH = 7 # 62^7 = 3.52 trillion combos BATCH_SIZE = 1_000_000 # Generate 1M keys per batch def generate_unique_key() -> str: return ''.join(random.choices(CHARS, k=KEY_LENGTH)) def populate_key_pool(db): keys = {generate_unique_key() for _ in range(BATCH_SIZE)} db.bulk_insert("INSERT IGNORE INTO key_pool (short_key) VALUES (?)", keys) # ─── API Server: Claim a key ────────────────────────────────────── def claim_key(redis_client) -> str: # Redis SPOP is O(1) and atomic - perfect for distributed key claim key = redis_client.spop("available_keys") if not key: # Fallback: claim from DB with row lock key = db.query_one(""" SELECT short_key FROM key_pool WHERE is_used = FALSE LIMIT 1 FOR UPDATE SKIP LOCKED """) db.execute("UPDATE key_pool SET is_used=TRUE WHERE short_key=?", key) return key

    System Architecture

    A layered architecture with CDN, load balancers, stateless API servers, Redis caching, and separated read/write paths. Every component is horizontally scalable and has no single point of failure.

    High-Level Architecture
    TIER 1 — CLIENT LAYER
    🌐 Browser / App
    📱 Mobile Client
    🤖 API Consumer
    HTTPS requests routed via DNS
    TIER 2 — EDGE & LOAD BALANCING
    ☁️ CDN (CloudFront / Cloudflare)
    Caches popular redirects at edge PoPs globally. Reduces latency to <5ms for cached hot links. Only misses reach origin.
    ⚖️ Load Balancer (L7, e.g. NGINX / ALB)
    Routes /:short_key to Read Service, /api/v1/* to Write Service. Round-robin with health checks. No state stored here.
    🛡️ WAF + Rate Limiter
    Blocks abusive IPs, enforces per-API-key rate limits (Redis token bucket). Protects against DDoS and enumeration.
    Routed to appropriate microservice
    TIER 3 — APPLICATION SERVICES (Stateless, Auto-scaled)
    🔍 Read Service (URL Resolver)
    Handles /:short_key lookups. Checks Redis → DB. Fires async click event. Issues 302 redirect. Scales to 50+ instances.
    ✏️ Write Service (URL Creator)
    Handles URL creation/deletion. Claims key from KGS pool. Writes to DB. Populates cache. 10-20 instances suffice.
    🔑 Key Generation Service
    Background service pre-generating keys. Runs as cron or daemon. Keeps key_pool stocked above 100M keys. 2 instances (primary + standby).
    Cache-aside pattern: check cache first, then DB
    TIER 4 — DATA STORAGE
    ⚡ Redis Cluster (Cache)
    Stores short_key → long_url mappings. LRU eviction. 170GB RAM covers top 20% of URLs which handle 80% of traffic. Sub-ms reads.
    🗄️ Cassandra / DynamoDB (URL Store)
    Primary URL store. Sharded by short_key. Handles 100M+ writes/day. Linear horizontal scale. Multi-region replication.
    📨 Kafka (Event Queue)
    Click events published here async. Decouples read path from analytics writes. Consumers process & aggregate to ClickHouse.
    📊 ClickHouse (Analytics)
    OLAP columnar store. Receives from Kafka consumers. Powers real-time dashboards and aggregated reports. Petabyte-scale.

    Request Flows

    Walking through the exact sequence of operations for the two most critical paths: creating a short URL and resolving a redirect.

    ✏️ Write Flow — Create Short URL

    1
    Client sends POST /api/v1/urls with API key + long URL. WAF validates; rate limiter checks quota (Redis token bucket).
    2
    Write Service validates the URL (syntax, malware check via Google Safe Browsing API). Reject if invalid.
    3
    Check if custom alias provided. If yes, verify uniqueness via DB lookup. Reject with 409 if taken.
    4
    If no custom alias, claim a pre-generated key via Redis SPOP("available_keys") — O(1), atomic, no lock contention.
    5
    Write record to Cassandra (short_key → long_url, metadata). Async replication to other regions.
    6
    Populate Redis cache with the new mapping. Return short URL to client. Total latency: ~50–100ms.

    🔍 Read Flow — Redirect Resolution

    1
    User visits https://short.ly/abc1234. DNS resolves to CDN edge. CDN checks its cache for this key.
    2
    CDN HIT: Return 302 redirect immediately from edge. Latency <5ms. ~80% of popular URLs served here.
    3
    CDN MISS: Request forwarded to Load Balancer → Read Service. Check Redis cluster for short_key mapping.
    4
    Redis HIT: Return 302 redirect. Publish click event to Kafka asynchronously (non-blocking). Latency <10ms.
    5
    Redis MISS: Query Cassandra by short_key. Populate Redis cache (LRU). Return 302 redirect. Latency ~50ms.
    6
    Not found / expired: Return 404 or 410. Kafka consumer processes click event, enriches with GeoIP, writes to ClickHouse.

    Caching Strategy

    Given the 100:1 read-to-write ratio and Pareto distribution of URL popularity, caching is the single most important optimization. We use a multi-layer cache hierarchy.

    Cache Hierarchy & Decision Logic
    cache_strategy.py
    async def resolve_url(short_key: str) -> str: # ── Layer 1: CDN Edge Cache ──────────────────────────────────────── # CDN handles this before request reaches our servers (invisible here) # Cache-Control: max-age=3600 for popular links # ── Layer 2: In-process LRU Cache (per server) ──────────────────── # Each API server keeps a 10K-entry LRU cache for ultra-hot links if long_url := local_lru.get(short_key): metrics.inc('cache_hit_local') return long_url # ── Layer 3: Redis Cluster ───────────────────────────────────────── # 170GB Redis across 6 shards. LRU eviction. TTL = URL expiry time. if long_url := await redis.get(f"url:{short_key}"): local_lru.set(short_key, long_url) # backfill L2 metrics.inc('cache_hit_redis') return long_url # ── Layer 4: Database (Cassandra) ───────────────────────────────── # Last resort. Expected to handle <1% of redirect traffic. record = await cassandra.get("SELECT long_url,expires_at FROM urls WHERE short_key=?", short_key) if not record: raise NotFoundException() if record.expires_at < now(): raise ExpiredException() ttl = (record.expires_at - now()).seconds await redis.setex(f"url:{short_key}", ttl, record.long_url) # populate Redis local_lru.set(short_key, record.long_url) # populate L2 metrics.inc('cache_miss_db') return record.long_url # ── Cache Invalidation on Delete ────────────────────────────────── async def delete_url(short_key: str): await cassandra.soft_delete(short_key) # mark is_deleted=True await redis.delete(f"url:{short_key}") # invalidate Redis await cdn.purge(short_key) # CDN purge API call # Local LRU cache: TTL-based expiry (max 60s stale) — acceptable
    Cache Eviction Policy — LRU with TTL: Use LRU (Least Recently Used) eviction for Redis. Set TTL equal to URL expiry time so expired URLs auto-evict. 20% of URLs typically drive 80% of traffic (Pareto principle), so 170GB of Redis can serve ~80% of all redirect traffic without touching Cassandra.

    Scaling & High Availability

    Every layer is designed for horizontal scale and fault tolerance. No component is a single point of failure. We target 99.99% availability (52 min downtime/year).

    📦

    Stateless API Servers

    Read and Write services store no local state. Any server can handle any request. Auto-scale based on CPU/RPS metrics. Kubernetes HPA scales from 5 to 200 pods within 60 seconds.

    ⚡

    Redis Cluster Sharding

    6 Redis nodes in cluster mode. Keys sharded by hash slot (16384 slots). Each node handles ~30GB. Add nodes to scale. Replicas for each primary ensure cache survives node failures.

    🗄️

    Cassandra Horizontal Scale

    URLs partitioned by short_key using consistent hashing. Replication factor 3. Add nodes seamlessly — data rebalances automatically. Multi-datacenter replication for geo-redundancy.

    🌍

    Multi-Region Deployment

    Active-active in 3 regions (US-East, EU-West, AP-Southeast). Cassandra replicates across regions. Users routed to nearest region via GeoDNS. One region can fail without service interruption.

    ⚖️

    Load Balancer HA

    Two NGINX instances in active-passive with keepalived. Virtual IP floats to secondary on failure. L7 load balancer does health checks every 2s. Routes around failed API servers instantly.

    🔄

    Circuit Breaker Pattern

    If DB fails, circuit breaker opens and serves from cache only. Graceful degradation: redirect works even if write service is down. Hystrix / Resilience4j patterns applied per service call.

    📨

    Kafka Topic Partitioning

    click_events topic with 100 partitions. Each partition consumed by one analytics worker — ensures ordering per short_key. Retention: 7 days. Consumers can replay if ClickHouse goes down.

    🧹

    URL Expiry Cleanup

    Batch job runs nightly: scans for expired URLs, marks soft-deleted, removes from Redis. Alternatively, set Cassandra TTL = URL expiry time for auto-deletion. Frees storage progressively.

    📊

    Monitoring & Alerting

    Prometheus + Grafana for metrics. Alert if cache hit rate drops below 95%, redirect P99 exceeds 50ms, or Kafka consumer lag exceeds 100K. PagerDuty on-call rotation.

    Bottlenecks & Mitigations

    Identifying bottlenecks before they happen — and their mitigations — demonstrates systems thinking depth. Here are the most critical failure points.

    Bottleneck Severity Symptom Mitigation
    Cache stampede HIGH A popular URL expires from Redis. Thousands of requests simultaneously miss cache and hammer the DB, causing overload. Use "lock-based" cache population: first request acquires a Redis lock and fetches from DB; others wait. Or probabilistic early expiration (refresh cache before it expires).
    Key collision MED Two API servers claim the same key from the pool (race condition). Use Redis SPOP (atomic set pop) — only one server can claim a key. DB fallback uses FOR UPDATE SKIP LOCKED. Pre-generated key pool eliminates real-time generation conflicts.
    Hot key in Redis HIGH A viral URL generates 100K+ req/s to a single Redis shard, overwhelming it. Duplicate hot keys across multiple Redis nodes. Read from a random copy. Or use local in-process LRU cache (L2) to absorb per-server hot traffic before Redis.
    Write amplification MED Every redirect fires a Kafka event. At 116K/s, Kafka producers could become a bottleneck. Batch events before publishing (micro-batch every 100ms). Use async non-blocking producers. Kafka is designed for millions of messages/sec — proper batching keeps this safe.
    Cassandra hotspot MED All recent URLs hash to same token range, overloading one node. Short keys are randomly generated (not sequential), so hash distribution is naturally uniform. Murmur3 partitioner ensures even spread across nodes.
    Long URL storage bloat LOW URLs with query params can be 2,000+ bytes, bloating DB and cache. Cap long_url at 2,048 characters (HTTP spec limit). For cache efficiency, store full URL in DB but only cache the first 500 chars as a pointer if needed.
    DNS propagation delay LOW On region failover, GeoDNS change can take 30–60s to propagate, causing brief outage. Use low TTL (30s) for GeoDNS records. Pre-warm secondary region. Use Anycast IP routing for instant failover without DNS changes.

    Security Considerations

    URL shorteners are frequently abused for phishing, malware distribution, and spam. Security is not optional — it's a core product feature.

    ● Malicious URL Detection

    Validate URL format (RFC 3986) before accepting
    Check against Google Safe Browsing API on creation
    Periodic re-scanning of stored URLs (async job)
    If flagged: return 451 Unavailable For Legal Reasons
    Human review queue for borderline cases

    ◆ Abuse Prevention

    Rate limiting: 100 URL creations/hour per API key
    CAPTCHA for unauthenticated creation attempts
    IP-based rate limiting as secondary defense
    Key space is random — enumeration is impractical
    WAF rules block known bad actor IP ranges

    ◈ Data Privacy

    IP addresses stored as SHA-256 hashes (GDPR)
    User data deletion fully removes all URLs and events
    Analytics data is aggregated, not per-user
    HTTPS enforced everywhere (HSTS headers)
    API keys rotatable without data loss

    ◉ Infrastructure Security

    Cassandra and Redis on private VPC — not public-facing
    API servers in private subnet; only LB is internet-facing
    mTLS between internal microservices
    Secrets managed via Vault / AWS Secrets Manager
    Audit logs of all admin actions stored immutably
