Windows & Linux

1
Install Terraform
One-time install — pick your OS
Windows — PowerShell / winget
# Option 1: winget (recommended)
winget install HashiCorp.Terraform

# Option 2: Chocolatey
choco install terraform

# Option 3: Manual download
# Go to: https://developer.hashicorp.com/terraform/downloads
# Download Windows AMD64 zip → extract → add folder to PATH

# Add to PATH manually if needed (PowerShell):
$env:PATH += ";C:\terraform"

# Verify (open new terminal)
terraform version
# should show: Terraform v1.x.x on windows_amd64
Linux — apt / manual
# Option 1: via HashiCorp apt repo (Ubuntu/Debian)
wget -O- https://apt.releases.hashicorp.com/gpg | \
  sudo gpg --dearmor -o /usr/share/keyrings/hashicorp.gpg

echo "deb [signed-by=/usr/share/keyrings/hashicorp.gpg] \
  https://apt.releases.hashicorp.com $(lsb_release -cs) main" | \
  sudo tee /etc/apt/sources.list.d/hashicorp.list

sudo apt update && sudo apt install terraform

# Option 2: Manual download (any Linux)
wget https://releases.hashicorp.com/terraform/1.7.5/terraform_1.7.5_linux_amd64.zip
unzip terraform_1.7.5_linux_amd64.zip
sudo mv terraform /usr/local/bin/

# Verify
terraform version
# should show: Terraform v1.x.x on linux_amd64
2
Project Folder Structure
Recommended layout for a Terraform project
recommended structure
my-terraform-project/
├── main.tf          ← main resources go here
├── provider.tf      ← AWS provider config
├── variables.tf     ← input variables
├── outputs.tf       ← output values
├── terraform.tfvars ← variable values (don't commit!)
└── .terraform/      ← auto-created on init (don't touch)
3
Configure AWS Provider
provider.tf — default + secondary (from your notes)
provider.tf
terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

# Default provider — uses [default] from ~/.aws/credentials
provider "aws" {
  region = "us-east-1"
  # profile = "default"  ← optional, default is used automatically
}

# Secondary provider — uses [secondary] from ~/.aws/credentials
provider "aws" {
  alias   = "secondary"
  region  = "us-east-2"
  profile = "secondary"
}
main.tf — use both providers in resources
# S3 bucket in default account (us-east-1)
resource "aws_s3_bucket" "main" {
  bucket = "my-main-bucket-12345"
}

# S3 bucket in secondary account (us-east-2)
resource "aws_s3_bucket" "backup" {
  provider = aws.secondary
  bucket   = "my-backup-bucket-12345"
}

# EC2 instance in default account
resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.micro"
  tags = {
    Name = "web-server"
  }
}

Variables & tfvars
Keep sensitive values out of code
variables.tf — define variables
variable "region" {
  description = "AWS Region"
  type        = string
  default     = "us-east-1"
}

variable "environment" {
  description = "Environment name"
  type        = string
}

variable "bucket_name" {
  description = "S3 bucket name"
  type        = string
}
terraform.tfvars — actual values
# ⚠️ Add terraform.tfvars to .gitignore!

environment = "production"
bucket_name = "my-app-bucket-prod"
region      = "ap-south-1"

5
Core Terraform Commands
Same commands on Windows & Linux — run inside project folder
terminal — daily workflow commands
# ── Step 1: Initialize project (run once, or after adding providers)
terraform init
# downloads provider plugins into .terraform/

# ── Step 2: Format code (auto-fix indentation)
terraform fmt

# ── Step 3: Validate syntax
terraform validate

# ── Step 4: Preview what will be created/changed/destroyed
terraform plan
# shows: + create, ~ update, - destroy

# ── Step 5: Apply changes (creates real infrastructure)
terraform apply
# type "yes" when prompted

# Apply without confirmation prompt
terraform apply -auto-approve

# ── Step 6: See current state
terraform show
terraform state list

# ── Step 7: Destroy everything
terraform destroy
# type "yes" when prompted
terraform apply creates REAL AWS resources that cost money. Always run terraform plan first to review what will be created. Use terraform destroy when done testing.
6
State File & Remote Backend
Store state in S3 so team can share it
backend.tf — remote state in S3
terraform {
  backend "s3" {
    bucket         = "my-terraform-state-bucket"
    key            = "prod/terraform.tfstate"
    region         = "us-east-1"
    profile        = "default"
    encrypt        = true
    dynamodb_table = "terraform-state-lock"  ← optional, prevents concurrent apply
  }
}

# After adding backend, re-run:
terraform init
# it will ask to migrate local state to S3 — type "yes"
Never commit terraform.tfstate to git. Use S3 backend for team projects. Add *.tfstate and *.tfvars to your .gitignore.
7
Workspaces — Dev / Staging / Prod
Manage multiple environments with one codebase
terminal — workspace commands
# Create workspaces
terraform workspace new dev
terraform workspace new staging
terraform workspace new prod

# List workspaces
terraform workspace list
# shows: * default  dev  staging  prod

# Switch workspace
terraform workspace select dev
terraform apply   ← deploys to dev

terraform workspace select prod
terraform apply   ← deploys to prod

# Use workspace name in resources
# resource "aws_s3_bucket" "x" {
#   bucket = "my-app-${terraform.workspace}"  ← dev / prod
# }
